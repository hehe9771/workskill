# 学生管理系统 AI 原生交互设计：从"加个聊天框"到意图驱动的生成式 UI

## 一、为什么交互是 AI 原生最要命的问题

把 AI 塞进一个既有学生管理系统，最常见也最致命的做法是**在表单系统旁挂一个聊天侧栏**——这就是 bolt-on（贴片式）改造。它的本质问题不是"AI 不够强"，而是**交互范式没有变**：学生管理系统仍然是导航驱动（左侧多级菜单树）、表单驱动（逐项填写提交）、表格驱动（分页固定表格）、报表驱动（固定模板导出 PDF/Excel），AI 只是在右侧开了一个聊天窗回答问题。结果是：

- **chatbox trap（聊天框陷阱）**：表单系统 + 聊天侧栏 = 两套割裂的交互。家长越忙越迷茫、教师越讲越疲惫，AI 反而加剧消息轰炸。
- **空白页焦虑与表达屏障**：超过半数用户写不出有效 prompt（GoodData），空聊天框让人僵住。
- **黑箱不可验证**：AI 自信地错，信任不可逆摧毁。
- **线性聊天五大失败**：洞察埋藏、无空间记忆、上下文失忆、单线程、关浏览器即毁（Steven Cen）——把"看王同学全貌"塞进对话流，相关分析按时间顺序而非空间邻近存在，跨维度关联（成绩下滑+考勤异常+评教差评同时出现才说明问题）根本看不出来。
- **确定性事务打字比点击慢**：录 40 个学生成绩、批量审批 50 条退费，打字比点表格慢一个量级。

现有 web/app 的导航/表单/表格范式与 AI 原生之间存在**根本张力**：web 的强项是结构化的信息密度、视觉扫描、批量操作、精确编辑；AI 的强项是模糊意图解析、综合推理、个性化叙事、主动触达。简单叠加会两败俱伤——既丢了 web 的结构效率，又让 AI 沦为聊天框。AI 原生交互的核心命题，是**让 AI 重新定义"界面如何生成"，而不是给旧界面加个聊天窗**。这份设计给出的答案是：**外壳作锚、内容区 AI 原生生成的混合正解**，并通过 tool-call→组件渲染把 AI 的输出锁死在受控组件库里，从机制层杜绝退化为聊天框。

## 二、AI 原生交互的 7 条原则

| # | 原则 | 在学生管理系统的具体落地 |
|---|---|---|
| 1 | **意图驱动而非导航驱动** | 用户在命令面板（Cmd+K / 移动端长按）表达"看王同学数学掌握度""三班有掉队风险的学生"，系统按意图+上下文（学期/角色/当前关注点）生成视图，而非点 4-5 层菜单筛选。意图路由器是守门人，决定是否退化。 |
| 2 | **生成式 UI（UI-on-demand）** | LLM 输出 tool-call，前端从 11 个受控组件库渲染对应组件+填 props（`<StudentSnapshotCard {...part.output} />`）。机制是 Vercel AI SDK 的 UIMessage parts + Zod inputSchema 作 props 契约，**不是任意 HTML/SVG**，防 prompt injection 注入恶意 UI。 |
| 3 | **情境化内嵌而非独立聊天面** | 选中表格行/图表元素 → inline AI 操作"长在选区"（浮现"标记风险/生成评语/发家长"菜单），选中成绩 → 就地展开根因分析，不另开聊天侧栏、不跳页。 |
| 4 | **可纠错（correctability）** | 每个 AI 输出都是可编辑草稿：学情报告可改字删段加批注、版本可回退；批量录入草稿表行级可编辑+撤销栈；家长沟通草稿必经编辑+确认门才发送。 |
| 5 | **不确定性表达** | 置信度三态徽章（绿>90%自动展开 / 黄70-90%附注 / 红<70%暂停问人）+ 可折叠"Why"推理披露（离散因子列表）+ 每条事实带角标来源（Q3成绩单 p.12 / 考勤系统 2026-06-15）。不伪装成确定事实。 |
| 6 | **copilot/autopilot/out-of-loop 映射到 UI** | copilot（人主导+确认门）：录入草稿、生成报告；autopilot（AI自主+人监督 HOTL）：风险预警三级升级；out-of-loop（仅通知不阻断）：考勤自动采集、成绩汇总。高风险/不可逆（发家长、改成绩、删档案）必有 durable 确认门；可逆/低风险仅通知。 |
| 7 | **混合正解** | 数据密集表格/批量操作/精确编辑/浏览对比保留 web 强项（AI 生成+增强，不塞进对话）；模糊意图/综合推理才走对话，且对话产物落回快照组件而非聊天流。不是"form + chat"叠加。 |

## 三、核心交互范式与机制

### 3.1 外壳与内容分离（关键妥协）

web/app 外壳作"锚"保留 web 结构效率，内容区是 AI 原生生成的——既不丢导航/身份/品牌，又不沦为聊天框。

- **外壳（Shell，固定锚）**：
  - 顶部栏 = 身份 + 角色切换（教师/家长/管理员/学生）+ 全局命令面板入口（Cmd+K 桌面 / 长按移动端）+ agent 监控脉动点
  - 左侧上下文锚 = 当前班级 / 学期 / 学科（可点改）
  - 底部状态栏 = agent 运行态
- **内容区（Content，AI 按意图生成）**：用户表达意图 → 意图路由器判定类型 → 调对应工具 → 流式渲染组件。

### 3.2 tool-call → 组件渲染：生成式 UI 的技术机制

采用 **Vercel AI SDK UIMessage parts 模型**（生产推荐，非 experimental RSC）：

1. 每个 AI 输出 = UIMessage，含 `parts` 数组，工具调用作为 typed part（`tool-${toolName}`），携带 `state` 字段驱动渐进式渲染：
   - `input-available` → 显示 loading 骨架卡
   - `output-available` → 展开真实组件（execute 返回的结构化数据直接作 props）
   - `output-error` → 显示错误卡
2. 工具用 `createTool({ description, inputSchema: z.object(...), execute })` 定义，`inputSchema` 用 Zod 强约束，`.describe()` 给模型语义提示（如 `city → studentId`、`threshold:.describe('挂科率阈值0-1')`）。这构成 LLM 提取参数与组件 props 间的**类型安全桥梁**，不是任意 HTML。
3. 服务端 `streamText` + `toUIMessageStreamResponse()` 产出 UI 消息流，客户端 `useChat`/`sendMessage` 发送；文本 part 与工具 part 混排，token 级流式 + 工具结果"弹出"为富组件。

**受控组件库（11 个）**：StudentSnapshotCard、MasteryRadar、RiskTimeline、EditableDraftForm、PlanApprovalPanel、AugmentedDataTable、ConfidenceBadge、ReasoningDisclosure、CommunicationDraft、NotificationCard、ApprovalCard。LLM 只能输出 schema 合法的 props。

**三级渐进式披露**（Anthropic Agent Skills）：Level1 组件元数据（name/description，约100token/组件）始终加载进 system prompt；Level2 触发时加载组件定义入上下文；Level3 数据按需取（脚本输出回灌上下文，代码本身不入 context，零额外 token 成本）。

### 3.3 混合正解：何时用对话、何时用结构化 UI

| 意图类型 | 走对话 | 走结构化 UI | 理由 |
|---|:---:|:---:|---|
| 查询单学生单维度（"王同学数学掌握度"） | ✗ | ✓ StudentSnapshotCard | 确定性查询，结构化卡比对话流更清晰 |
| 浏览对比多学生多维度（"三班风险学生"） | ✗ | ✓ AugmentedDataTable | 数据密集，视觉扫描是 web 强项 |
| 批量录入（40 人成绩/考勤） | ✗ | ✓ EditableDraftTable | 精确编辑，打字比点击慢 |
| 精确 CRUD / 数据修正 | ✗ | ✓ 直连表格 inline edit | 确定性事务，AI 只做异常检测 |
| 报表生成 | ✗ | ✓ 可编辑文档（Artifacts） | 结构化产物可下载可追溯 |
| 批量审批（50 条退费） | ✗ | ✓ BatchApprovalTable + 风险排序 | 数据密集，AI 做风险标注 |
| 模糊综合推理（"哪些学生需关注"） | ✓ | 产物落回 AugmentedDataTable | 开放式跨表关联，原则7允许 |
| 开放式建议（"孩子抗拒数学怎么办"） | ✓ | 每轮产微型 artifact | 综合推理通道，有硬上限 |

**护栏**：凡有确定可枚举维度的筛选必须用 chips/dropdown 组件（构建期 lint 约束，禁止落自由文本）；"卡片是第一公民、对话是入口不是容器"；导出/批量必走计划审批面板而非对话确认。

### 3.4 流式渐进式 UI 与 inline AI

- **流式渐进式**：RSC `streamUI` 的 async generator——`yield` loading 骨架立即推客户端 → `await` 异步取数 → `return` 最终组件替换。置信度三态驱动渐进披露（高自动展开 / 中附注 / 低暂停问人）。
- **inline AI**：表格行选中 → 浮动 inline AI 菜单（**非侧栏**），操作长在选区；环境 agent 建议态渐进式披露（脉动圆点 → 灯泡 + 两按钮 → 完整告警）。

## 四、模块级/交互面级交互重设计

### 4.1 核心交互范式与入口路由

**传统交互**：左侧多级折叠菜单树（学生档案/班级管理/成绩录入/考勤管理/通知发送/报表中心），顶部面包屑，主区固定页面。找一个学生要"学生管理→按班级筛选→翻页→点姓名→切成绩 Tab"翻 4-5 层。录入成绩是长表单逐项提交，报表固定模板导出 PDF/Excel，家校沟通靠微信群、通知群发，与系统数据割裂。

**痛点**：
1. 导航驱动认知负荷高：跨表查一个学生要翻 4-5 层菜单+多张表人工拼接，约 30% 时间花在重新找已发现信息；
2. 表单录入慢且重复易错（40+ 学生逐项录成绩）；
3. 报表固定无法按需生成（要"数学成绩下滑+考勤异常的学生"得人工跨表筛选）；
4. 数据分散无空间关系，相关分析按菜单分割无法并排对比；
5. 家校沟通消息轰炸+信息碎片化；
6. 综合推理缺位（"哪些学生需关注"要人工跨表）；
7. 若简单旁挂聊天侧栏 = chatbox trap：空白页焦虑、表达屏障、黑箱无法验证、确定性事务打字比点击慢。

**AI 原生交互设计**：应用全部 7 原则。外壳作锚（导航/身份/角色/品牌/上下文），内容区 AI 按意图生成。**意图路由器是关键守门人**——用户在命令面板输入"王同学 数学"→ LLM 补全为结构化意图"查看王某某数学掌握度"→ 路由到对应渲染路径而非直接聊天：
- 查询型（单学生单维度）→ `get_student_mastery` 工具 → 流式生成 StudentSnapshotCard
- 浏览对比型（多学生多维度）→ 生成 AugmentedDataTable（保留结构化表格强项，AI 填风险/趋势列）
- 综合推理型（跨表关联"哪些学生需关注"）→ 对话+内嵌组件（唯一走对话的场景）
- 录入型 → 智能表单（AI 预填+草稿确认门）
- 报表型 → 生成可编辑报表草稿（Artifacts 模式）

**生成组件**：StudentSnapshotCard、MasteryRadar、RiskTimeline、EditableDraftForm、PlanApprovalPanel、AugmentedDataTable、ConfidenceBadge、ReasoningDisclosure、CommunicationDraft、NotificationCard、ApprovalCard（11 个）。

**用户流程**（教师查王同学数学掌握度，端到端）：
1. 教师按 Cmd+K → 命令面板，输入"王同学 数学"→ 面板自动补全意图"查看王某某数学掌握度"（建议提示词填补空白页）；
2. 意图路由器判定为"查询型：单学生单学科"，调用 `get_student_mastery`（Zod inputSchema: `{studentId, subject}`）；
3. 流式渲染：先输出 loading 骨架卡（`part.state='input-available'`）→ 工具执行取数 → `output-available` 弹出 StudentSnapshotCard；
4. 快照卡显示：基本信息 + MasteryRadar（数学各知识点）+ RiskTimeline（最近3次下降事件，ConfidenceBadge 78% 黄色）+ "Why"折叠（连续3次下降12分、作业完成率95%→70%、考勤2次异常）；
5. 教师点风险时间线某事件 → 展开来源角标（Q3成绩单 p.12 / 考勤系统 2026-06-15）；
6. 教师选中快照卡 → inline AI 菜单浮现"生成家长沟通"→ 生成 CommunicationDraft 草稿；
7. 教师编辑草稿两处 → 确认门（durable ApprovalCard，可暂停跨天）→ 提交发送；
8. 同时 autopilot 后台已检测风险：之前给过轻量建议（灯泡）教师 dismiss 过 → 现升级完整告警 NotificationCard → 教师可选生成辅导计划 PlanApprovalPanel（分步计划全可编辑→预览→确认门）。

**协同分级**：
- **copilot**（人主导+确认门）：批量录入成绩（AI提取→EditableDraftForm草稿→确认门提交）、生成学情报告（PlanApprovalPanel分步计划全可编辑→预览→确认门）、家长沟通草稿（生成→编辑→发送确认门）。
- **autopilot**（AI自主+人监督 HOTL）：风险预警（环境 agent 三级升级零打扰→建议→告警，HOTL 仪表盘+Pause Agent/Undo Last Action 终极控制）、考勤异常自动标记（低风险自动+通知归档）。
- **out-of-loop**（仅通知不阻断）：日常通知归档、报表定时生成草稿+通知。
- **确认门阈值**：高风险/不可逆（发家长、改成绩、删档案）必有 durable 确认门（可暂停跨天+schedule 超时升级 4h 提醒/24h 升级主管+不可变审计表记录 workflowId/决策/approver/理由/时间戳）；可逆/低风险（标记、草稿、内部备注）仅通知。置信度<70% 暂停问人（HITL），>90% 自动执行，70-90% 执行但附注。MultiApproval 敏感操作（改成绩需教师+教务双签达阈才触发）。

**聊天框陷阱检查**：总体**不会**退化为聊天框——日常约 80-90% 操作（查/录/批/报）被意图路由器强制导向生成式 UI/增强表格/智能表单/可编辑文档，对话仅留给真正模糊的跨表推理。但**三处真实退化风险**：
1. 命令面板被标为"主入口"，与聊天框共享同一文本输入面，区分全靠路由器+结构化兜底；若兜底被削弱或路由器误判率高，瞬间退化为空白页焦虑+表达屏障的聊天框；
2. "综合推理型→对话+内嵌组件"是设计里唯一的聊天面，用户一旦进了对话倾向继续打字而非跳出看表；
3. "生成家长沟通草稿"若工程偷懒成"AI 写好直接发"会失去可纠错，沦为黑箱群发。

**对抗核验结论**：`truly_ai_native_interaction`，但须落实修正。

**修正**：
1. **降权命令面板、提权表格**：把增强数据表格/智能表单/可编辑文档设为各自场景的一等入口（点击驱动默认落地），命令面板仅作"模糊起点"快捷方式而非 THE 入口；上下文锚（班级/学期/学科）选择即默认浏览视图，提供 always-available 的点击逃逸舱。
2. **收紧"综合推理→对话"边界**：跨表推理也优先渲染为 AugmentedDataTable（AI 生成"需关注学生"列）+ ReasoningDisclosure，自由对话作最后回落；对话内输出一律组件化锚定，禁止纯文本表格数据。
3. **前置回退阈值**：去掉"连续两轮"滞后判定，首次返回 >N 行或结构化字段即主动提示/切换表格视图。
4. **角色自适应渲染**、**提取置信逐行化**、**建议节流**、**MultiApproval SLA**、**补学生角色流程**。

**技术机制**：见第三章 3.2 / 3.4。durable 审批门（Cloudflare `waitForApproval({timeout:'7 days'})` + `schedule` 超时升级 + 不可变 SQLite 审计表 + `pendingApprovals` 经 `setState()` 持久化跨连接中断 + React `useAgent()` 渲染 ApprovalCard）。

### 4.2 "学生快照"统一视图

**传统交互**：事务型 CRUD 集合，按模块切页——左侧导航"学生档案/成绩管理/考勤管理/评教管理/家校沟通/统计报表"，每模块=列表页→筛选→详情页。看一个学生全貌要跳 4-5 个页面、每页重新筛选、上下文丢失。

**痛点**：
1. 多页跳转、上下文丢失，无法并排看出跨维度关联；
2. 信息碎片化、洞察埋藏，期末才出学情报告，问题发现时干预黄金期已过（AI 比阈值法可提前约 3 周识别风险，干预成功率从 80% 跌到 20%）；
3. 角色失明：同一套数据三种角色需求不同却给同一界面；
4. 消息轰炸，已读率塌陷；
5. 黑箱无行动性：阈值红黄灯只告警不归因；
6. 设计缺口：学生无独立主动入口，管理员角色缺位。

**AI 原生交互设计**：四角色共享"意图→快照"生成管线，变体在组件选择与信息密度。

- **教师变体（效率+批量+风险发现）**：顶部 StudentSnapshotCard（一句话状态摘要）+ MasteryRadar（点击某科轴展开知识点明细）+ RiskTimeline（红点=风险/绿点=进步，点击红点→inline AI 展开根因+置信度+来源）+ 批量视图（一句话"三班有掉队风险的学生"→ 风险学生列表 + BatchActionBar）+ 可编辑草稿。
- **家长变体（信息+可执行指导）**：简化快照卡（不展示风险分原始值与排名）+ 趋势图（进步标绿）+ 可执行建议卡（"本周可和孩子一起做 X 来提升 Y"，附置信度+来源）+ 沟通入口（自动引用快照上下文）。
- **学生变体（自主查看+进步导向）**：我的进步卡（正面导向，不展示风险评分）+ 改进建议卡（苏格拉底式"给提示而非给答案"，Khanmigo 式）+ 无风险预警/无排名。
- **管理员变体（补缺位）**：权限管理/数据质量看板/跨班全局风险排名/系统健康，权限变更走 autopilot 审批门。

**生成组件**：CommandPalette、StudentSnapshotCard、MasteryRadar、RiskTimeline、InlineExplanationPanel、GradeTrendChart、AttendanceHeatmap、FeedbackCard、EditableDraftForm、PlanApprovalPanel、ConfidenceBadge、SourceCitation、BatchActionBar、NotificationToast。

**用户流程**（教师查王同学数学掌握度）：
1. 命令面板输入或语音"王同学数学掌握度"；
2. LLM 解析意图+识别教师身份+取当前学期 → `get_student_snapshot(studentId, subject, context)`；
3. 流式渲染：先出 StudentSnapshotCard（loading 态）→ 数据到位弹出"数学掌握度 72%，近 2 周下滑"→ MasteryRadar → RiskTimeline；
4. 教师点击时间线红色风险点 → 触发 `explain` tool → InlineExplanationPanel 流式展开："这周下滑与 2 次缺席+未交作业相关，置信度 78%"，可折叠 Why 显示 SHAP top-3 因子 + sources；
5. 教师点"生成家长沟通草稿" → AI 起草 EditableDraftForm → 教师编辑措辞；
6. 教师点发送 → 触发 autopilot PlanApprovalPanel 确认门 → 确认；
7. 发送后仅 NotificationToast 通知（out-of-loop，不打扰）。全程无跳页、无侧栏聊天、草稿可撤销。

**协同分级**：查看/分析/起草=copilot 无门；发消息/批量标记/生成报告/触发工单=autopilot 走 PlanApprovalPanel；考勤自动采集/成绩汇总/掌握度向量更新=out-of-loop 仅 NotificationToast。风险评分置信度阈值驱动升级（>90% 自动标记+通知；70-90% 建议复核；<70% 不标记）。心理危机类主动沟通强制辅导员/临床复核，精度优先宁漏报勿误报，提供 opt-out。

**聊天框陷阱检查**：核心**不会**退化——主路径是"一次性意图→生成快照画布→组件内交互"，交付物是结构化卡片而非消息气泡。但存在 5 条"设计意图有、架构强制缺"的退化裂缝：(1) 最致命——命令面板若做成持续文本流而非真·一次性面板，"再看语文""那考勤呢"会退化为线性聊天；(2) 家长语音路径天然偏会话；(3) InlineExplanationPanel 是否接受自由文本追问未定义；(4) 模糊意图→对话→回落快照接缝无硬切换机制；(5) 学生苏格拉底提示最接近辅导聊天。

**对抗核验结论**：`truly_ai_native_interaction`，但 5 条是 enforcement gap，必须落实修正后才稳态。

**修正**：
1. **命令面板硬性一次性**：实现为 Linear/Raycast 式 true command palette——输入→生成快照→面板关闭清空；输入框拒收"再看 X/那 Y 呢"续聊语义，或接受但强制重新生成整张快照；追问仅通过快照内组件交互。
2. **家长语音回落锚定**：语音回答强制为"前置一两句摘要+推送快照卡片/链接"，禁止纯语音多轮；追问自动生成新快照。
3. **InlineExplanationPanel 单次不可续聊**：面板内不设自由文本输入框；深挖用结构化按钮（"展开 Why""看更多来源""跳转相关事件"）触发新 explain。
4. **模糊意图→对话→回落强制管线**：对话仅作意图澄清（1-2 轮收敛），收敛后强制调 tool 生成快照组件，对话内容不保留为交付物；超阈值轮数未收敛给候选快照入口。
5. **草稿精炼走字段非对话**：EditableDraftForm 修订仅支持字段级编辑+"重新生成整段"按钮+风格预设按钮，禁止自由对话式修订。
6. **补管理员变体**；**置信度对家长/学生降维**（自然语言三态+一句话来源）；**精确 CRUD 直连组件 inline edit 不走 AI**；**复合意图解析兜底**（失败时回落结构化筛选器组件）；**移动端组件降级策略**（雷达→条形列表、热力图→简表、时间线→卡片流）。

**技术机制**：tool-call→component（Vercel AI SDK parts 模型、Zod inputSchema 作 props 契约、预定义组件库、不生成任意 HTML 防注入），流式渐进渲染。角色路由层按角色身份+意图选不同 tool 子集与信息层级。数据模型倒置：作答事件流+知识状态向量是唯一持久化核心，分数/排名是按需投影（移除 AI 无法生成任何学业输出）。

### 4.3 数据录入交互（成绩录入/考勤/缴费录入）

**传统交互**：导航驱动+固定表单+表格。成绩录入：点"成绩管理"→选考试→选班级→进入名单表格→逐行点单元格敲分数→保存；或走导入：下载 Excel 模板→严格按表头填写→上传→预览→确认，任一表头不符即整批失败，报错笼统。考勤：点名表逐行单选。缴费：搜学生→填金额/日期/方式→保存，或导入银行流水 CSV 按列名硬匹配。

**痛点**：导航成本高；纯手工敲键慢且易错（错行、错人、漏录，分数敲成 1000、给已毕业学生录考勤等无即时校验）；模板导入脆弱（错一列整批失败）；纸质数据无法直用须人工二次誊录；移动端难用；无草稿无纠错；上下文断裂（录入时看不到业务规则）。

**AI 原生交互设计**：核心是录入入口="多模态采集器"（一个 action launcher，**非聊天输入框**），采集后 AI 输出 tool-call → 渲染"可编辑草稿表格"。

- **入口**：任意页面浮动"录入"按钮（相机/文件图标，视觉上非文本框）+ Cmd+K。弹出多模态采集器：拍照/语音/导入文件/选区补录。无空白聊天输入框。
- **采集→流式**：上传 → loading 骨架表 → AI 识别意图 → 意图确认卡（缩略图+结构化字段：班级/考试/课程为**系统数据下拉，非自由文本**）→[确认][改(结构化下拉)][重拍]。
- **草稿表**：行=学生（自动花名册匹配），列=姓名/学号/成绩/置信度/来源。置信度三态边框（绿>90%/黄70-90%/红<70%或校验失败）+始终可见图例。默认视图=仅看需复核（已批准绿行折叠于"显示40条已批准"后）。
- **来源锚定**：hover 格 → 高亮原照片对应区域（provenance）；可折叠"为何是这个值"=仅平实来源、只读。
- **内联纠错**：点红/黄格 → 内联纠错面板（照片裁切+候选下拉+手输）→ 格变绿。结构性错配 → "重匹配参数调整"（按学号列 vs 姓名列）或[重拍]。潦草扎堆（红格>30%）主动提示重拍而非逼逐格手输。
- **即时校验**：AI 边录边查业务规则（满分上限/考勤节假日/重复缴费/学生状态），异常格红+平实"为何异常"。
- **批量信任栏**：[批准全部高置信][仅看需复核]。

**生成组件**：MultimodalCaptureBar、IntentConfirmCard、EditableDraftTable、ConfidenceBadge、SourceAnchor、InlineCorrectionPanel、ValidationBanner、DiffView、BulkActionBar、ConfirmGatePanel、VoiceTranscriptBubble、RosterMatchDropdown。

**用户流程**（教师手机拍纸质期中数学成绩单）：
1. 课堂巡视中点浮动"录入"→多模态采集器→点"拍照"→拍纸质成绩单；
2. 上传后流式：骨架表→AI 识别→意图确认卡（高三2班 期中数学 成绩 45人，缩略图）→教师点[确认]；
3. 渲染草稿表：45 行学生自动匹配花名册。大部分格绿色，3 个同名匹配黄色，2 个手写潦草红色"无法识别"，1 个"150"被校验条标红"超满分100"并高亮原照片该格；
4. 教师点红格"150"→内联纠错面板显示照片裁切→看图手输"95"→格变绿；点另一红格选候选"85"；3 个黄格用下拉选定正确学生；
5. 点"批准全部高置信"→40 行锁定；剩 5 条复核完；
6. 确认门面板："将提交45条：42自动+3修正+1异常已改"→[提交]；
7. 成功通知"已录入，3条已修正"+24h 撤销链接。全程未打字聊天、未点多层菜单。

**协同分级**：
- **Autopilot（>90% 绿）**：AI 自动填入并预勾选，一键"批准全部"；信任模式常规导入可自动提交+仅通知，异常才弹 UI；
- **Copilot（70-90% 黄）**：生成草稿但必须人工逐格复核，高亮提示，提供内联纠错面板，不自动提交；
- **Escalate（<70% 或校验失败 红）**：AI 不填值，明确升级人工，必须手输或重拍，**禁止静默提交**；
- **确认门**：所有提交前 ConfirmGatePanel 分层汇总；提交后 24h 撤销（durable）。异常率阈值覆盖信任模式（红/黄率超 X% 自动禁用信任模式强制复核）。敏感操作（退费/大额缴费）走多审批人阈值。

**聊天框陷阱检查**：整体**不会**退化为聊天框——主面是"可编辑草稿表格"（数据密集+精确编辑=web UI 强项，原则7）；多模态采集器是 action launcher（命令面板式，非空白聊天输入框）；AI 输出永远是结构化组件；模糊意图仅用"意图确认卡"（按钮+下拉）解决；纠错走内联结构化操作，不用"用自然语言告诉 AI 哪里错"。端到端 user_flow 全程零聊天回合。

**对抗核验结论**：`truly_ai_native_interaction`，风险低-中，但残余退化点必须硬化。

**修正（边界加固）**：
1. **强制结构化意图纠正**：[改课程/班级/考试]必须为系统数据下拉，禁止自由文本意图描述；无匹配记录走结构化"新建记录"子流——这是抵御聊天框退化的最重要边界；
2. **明文编码"禁止多轮语音编辑"**：语音/命令仅一次性捕获意图→生成草稿表，后续所有编辑均为表格直接操作；
3. **推理面板只读+去术语化**：显示平实来源（"手写95，与花名册王伟对齐"）而非模型内部，纠错永远走输入框/候选下拉；
4. 始终可见置信度图例 + 默认视图改为"仅看需复核"；
5. 异常率阈值强制覆盖信任模式；
6. **置信度呈现分层**：录入教师看三态；最终查看者（家长/学生）看简化"已核验/待复核"态或隐藏原始置信度，避免侵蚀信任；
7. **架构级禁令：全系统不得出现任何"问 AI"按钮**，异常永远升级为结构化人工输入；
8. 潦草扎堆场景（红格>30%）主动提示重拍而非逼逐格手输（通过 Marc Friedman 底线测试）；
9. 撤销链接显著化（通知中心持续可见，非一次性 toast）。

**技术机制**：定义工具 `extractGradeSheet(image)`/`matchRoster(names)`/`validateEntry(rule)`/`commitEntries(data)`，execute 返回结构化数据直接作 `<EditableDraftTable {...part.output} />` props。Zod inputSchema 强约束班级/学号/成绩字段类型。置信度三态可视化（绿实线/黄虚线/红点线边框）。来源锚定每格绑定原照片区域坐标，hover 高亮。durable 确认门+24h 撤销+不可变审计表。提议-确认两步（Sensible Agent，NASA-TLX 心理需求 65→21）。模态情境选择（YAMNet 噪声分类，嘈杂课堂静默视觉卡片、双手忙时语音）。

### 4.4 浏览与探索交互（选课/报表/分析）

**传统交互**：左侧目录树（学院→专业→班级→学生）；报表构建器（拖拽维度/度量、勾选字段，需懂 OLAP）；多级导航菜单；组合筛选区（先配后看）；分页固定表格；固定仪表盘（3-5 个预设图表，答不了临时问题）；导出死文件。

**痛点**：目录树太深；固定仪表盘答不了临时问题（"这学期挂科率最高的课""A班 vs B班数学均分"要么手工导两张表比对要么等 IT）；筛选"先配后看"看不到中间结果；下钻路径写死不能跨维度跳；对比难要并排两表手动对；空白页焦虑；导出是死文件无追溯。

**AI 原生交互设计**：主界面=**空间画布**（非聊天流）。顶部持久上下文条（学期/学院/身份 chips，可点改）+ 左侧意图历史作"锚"，中央画布是 AI 原生生成的卡片群。

- **意图入口（三选一，非唯一）**：(a) Cmd+K 命令面板（带建议提示词/热门查询/历史按钮）；(b) 画布常驻意图栏，可文本可语音；(c) 任意卡片元素选中 → 内联操作栏。
- **生成与下钻**：教师输入"这学期挂科率最高的课"→ tool-call(`query_high_fail_rate_courses{term,threshold}`)→ 画布"弹出"课程挂科率排行卡（条形图 top10）+ 置信度徽章 + 数据来源脚注 + 可折叠"为何这样算"。卡片底部一排"建议下钻 chips"（按学院拆分/对比上学期/看具体学生/导出）。点"按学院拆分"→同卡就地变分组条形（一次点击、不重新提问）。选中某条 → 浮出内联操作栏（下钻到学生/对比/加入对比区/导出）。
- **对比**：点"对比上学期"→ 画布右侧新增第二张卡，两卡连线链接，AI 自动生成第三张"对比摘要卡"。
- **可纠错**：拖拽重排卡片成空间布局（解决"无空间记忆"）；每卡右上"编辑"→展开筛选 chips，改一个 chip→该卡局部重渲染；"撤销"按钮回退。每卡 props 草稿栈+撤销栈服务端持久化，跨会话/设备可继续。
- **导出（切 autopilot）**：生成计划（导出选课明细+挂科学生名单，Excel，含统计页，是否脱敏学号）→ 计划审批面板 → Approve/Reject/Edit → 执行 → 画布落"导出产物卡"（可下载+**双向 provenance**：点击导出行可反向重开源卡）。

**生成组件**：课程挂科率排行卡、分组对比卡、学生名单表卡、掌握度雷达卡、趋势时间线卡、对比摘要卡、可编辑筛选草稿 chips、置信度三态徽章、数据来源脚注、建议下钻 chips 行、计划审批面板、导出产物卡、意图历史侧栏、画布连线/分组容器。

**用户流程**（教师查王同学数学掌握度，画布版）：
1. 教师打开分析工作台，画布顶部上下文条已带 2026春/计算机学院；
2. Cmd+K 输"王同学 数学掌握度"→ tool-call(`get_student_mastery`)→ 画布弹出掌握度雷达卡（整体黄徽章，因部分知识点缺近期测验数据→脚注"2个知识点样本<5，置信度降"）；
3. 卡片底部建议 chips：看各次测验明细/对比班级均值/看薄弱知识点练习建议/导出；
4. 点"对比班级均值"→ 画布右侧新增"班级均值雷达卡"+自动"对比摘要卡"（"王同学在函数章节低于班级均值15%"，带来源脚注）；
5. 选中雷达"函数"点 → 内联操作栏 → 点"看具体学生"→ 第三张测验明细表卡；
6. 点"导出"→ autopilot 计划审批面板 → Approve → 执行 → 画布落"导出产物卡"可下载+provenance 链。全程无聊天流，卡片空间共存可回看，可逐卡撤销编辑。

**协同分级**：浏览/查询/生成视图/下钻/对比/筛选编辑/拖拽编排=copilot 人主导无门（纯读无副作用）；导出/批量落库=autopilot 计划审批面板（durable 超时升级+不可变审计表）；缓存预生成报告（每周一自动更新学院挂科率排行）=out-of-loop 纯通知（画布落"已更新"卡+轻量角标，不弹窗不打断）。读多写少是这个面的特征。

**聊天框陷阱检查**：**不会**退化——主交互面是空间画布而非聊天流；数据密集视图是真实 web 组件（非对话流里的文本/图片）；下钻是一键内联不重新提问；确定性筛选用可编辑筛选 chips 不塞进对话；导出走计划审批面板。残留风险：若 Cmd+K 被做成全屏聊天流或简单筛选被引导走文本输入会回退——护栏是"凡有确定可枚举维度的筛选必须用 chips/dropdown 组件"。

**对抗核验结论**：`truly_ai_native_interaction`，退化风险低-中且已被设计主动设防。

**修正**：
1. **分角色双主面**：家长/学生默认落地"预生成卡面"（本周学习情况/我的成绩单，out-of-loop 纯通知模式延伸），教师/管理员默认"探索画布"，一键可切"答案模式"（单聚焦卡+建议下一步）与"探索模式"（全画布拖拽连线）；
2. **filter-chip 护栏构建期硬化**：对 inputSchema 可枚举字段做构建期 lint，凡可枚举维度必须映射 chip/dropdown，禁止落自由文本（不可绕过的类型约束）；
3. **建议覆盖率 SLA+自动提升**：度量建议 chips 覆盖率（>70%），低于阈值自动把高频自由文本查询提升为建议 chip；
4. **置信度可行动化**：红徽章<70% 不仅"暂停"，须提供内联修复 chips（补数据/换数据源/缩小范围重算/降低阈值重算）；
5. **可纠错会话持久化**：每卡 props 草稿栈+撤销栈服务端持久化；
6. **导出双向 provenance**；**语音意图草稿态**（语音→意图草稿可编辑预览→确认再生卡）；**画布卫生自动化**（自动分组/对齐/折叠+清理画布/快照视图+首次进入引导式示例查询）。

**技术机制**：tool-call→typed tool part（state 三态），Zod inputSchema 强约束 props 契约；工具注册表 `query_high_fail_rate_courses`/`drill_down_students`/`compare_periods`/`get_student_mastery`/`generate_summary`/`export_data`/`generate_report`。画布布局用 react-flow/类似库做空间编排+连线。内联 AI 选中元素 → 触发 `augment_card` tool-call 返回新 props 局部重渲染（非整页）。durable 审批门（Cloudflare waitForApproval 模式）。

### 4.5 代理与审批交互（异动办理/学业干预/缴费审批）

**传统交互**：导航驱动+表单+表格+固定审批链。异动办理走多步向导表单（几十字段手工填，学号→姓名/院系/欠费本可联动却要重复录）；审批链固定线性"申请人→辅导员→院系→教务处/财务处"，低风险请假也走完 3 级；待审批列表是扁平表格无优先级无风险标识；审批依据靠人工翻校规；只有操作日志无决策推理追溯；休学同时涉及学籍+课程+学费+宿舍却分散在 4 个子系统分别填 4 张表。

**痛点**：导航迷宫；表单割裂；审批黑箱（退费怎么算的？休学后 GPA 影响？有无欠费？是否符合校规第几条？全靠人工核对，幻觉风险高）；审批队列无智能；审批链僵化；不可纠错；无追溯；批量操作痛苦；审批被误解的个人化挫败（高风险学籍场景尤甚）。

**AI 原生交互设计**：**plan-first 委派**（仿 GitHub Copilot Workspace）——agent 产出结构化计划草稿（步骤+影响+风险+依据），经 tool-call→component 渲染为可编辑面板。

- **入口**：命令面板（"王同学 休学 身体原因"）/ 学生快照选中"办理异动"/ 移动端语音仅捕获意图。意图模板建议（办休学/办干预/批退费）+ 近期历史避免空白页焦虑。批量意图："数学挂科≥3门的学生生成学业干预计划"。**补全申请人侧**：学生/家长发起入口，看到简化版计划卡。
- **计划草稿生成**：agent 基于学生上下文 streamText 渐进式产出：学生快照卡（自动拉取学籍/成绩/欠费/课程/历史异动）+ PlanDraftPanel（可编辑步骤列表，每步带依据）+ ImpactPreviewPanel（执行后变化可视化，数值可点开看计算）+ RiskAssessmentCard（置信度三态+可展开推理因子+来源引用角标）。全部为可编辑草稿。
- **审批门**：ApprovalGateCard 显示金额/描述/请求人/置信度/影响摘要 + Approve/Reject/**Modify** + 必填理由（why 非 what，用于模型反馈学习）。durable 可暂停数天，24h 未批→升级院系，跨连接中断不丢状态。高风险多审批人阈值。
- **执行与可回溯**：ExecutionTraceTimeline 逐步状态流式更新，provenance 透明；UndoLastActionButton 回滚最近一次 agent 动作；不可变审计表记录 workflowId/决策/approver/时间戳/理由/置信度；执行后生成可分享 ApprovalArtifact（独立面板，非聊天文本）。

**生成组件**：StudentSnapshotCard、PlanDraftPanel、ImpactPreviewPanel、RiskAssessmentCard、ApprovalGateCard、BatchApprovalTable、ExecutionTraceTimeline、UndoLastActionButton、CommandPalette、ApprovalArtifact、SupervisionDashboard。

**用户流程**（异动办理）：
1. 教师在学生快照页选中王同学点"办理异动"；
2. streamText 渐进式生成：先弹学生快照卡（loading skeleton→展开），再流式产出休学计划草稿步骤；
3. PlanDraftPanel：[提交申请→辅导员核实身体证明→院系审批→学籍休学→课程保留4门退选2门→退费¥3200→宿舍退订]，每步附依据（"退费¥3200=年学费¥8000×剩余学期40%，依据校规[学费管理第12条]"）；
4. ImpactPreviewPanel：GPA冻结、退费¥3200、保留4门退选2门、宿舍已退订——教师点退费数值看计算明细；
5. RiskAssessmentCard：置信度88%（黄边附注），因子=[身体证明齐备✓/欠费¥0✓/校规允许✓/挂科2门需干预提示⚠]，来源=[学籍管理第23条/既往休学案例3例]；
6. 教师改步骤（加"保留1门选修"）、改退费备注，草稿迭代至对路径有信心；
7. 确认门：中风险→copilot，教师点 Approve + 填理由"身体原因合理，证明齐备"；
8. durable 审批门持久化，辅导员下班→次日审批；24h 未批→升级院系；
9. 执行：流式时间线更新每步状态；
10. 半年后审计：查不可变审计表，回溯"当时置信度88%、依据校规23条、审批人=教师X、理由=身体原因合理"。

**协同分级**（按 风险×置信度 矩阵路由）：
- **Autopilot（自动+通知，零门）**：低风险+高置信——请假≤3天、小额退费≤¥500、自动预填表单、自动关联政策依据；
- **Copilot（确认门，HITL）**：中风险+中置信——休学/复学/学业干预/中等退费，ApprovalGateCard，durable 可暂停+24h 超时升级，单审批人；
- **Out-of-loop（强制多审批人，HOTL→HITL）**：高风险+低置信——退学/开除/大额退费>¥5000/学籍开除，多审批人阈值（辅导员+财务+教务3人达阈）+ Emergency Stop + 不可变审计表；
- **HOTL 监督仪表盘**（管理员）：actions/hour、成功率、异常告警、活动日志流、Pause Agent/Adjust Boundaries/Emergency Stop/Undo Last Action。

**聊天框陷阱检查**：整体**不会**退化——plan-first 委派产出结构化计划草稿面板（非文本流），审批是门动作非聊天回复，批量/明细/对比保留结构化表格，命令面板是"直达导航+自然语言兜底"混合，streamText 期间显示 loading skeleton 而非逐字冒泡聊天气泡。但**4 条具体退化路径需封堵**：(1) **Modify 未定义=最大风险**——若 Modify 弹出"请说明要改什么"的纠正对话框、用户用自然语言描述修改、agent 重新生成，就变成"我问你答"纠错对话循环；(2) 批量生成→卡片洪流（N 张 PlanDraftPanel 堆叠滚动 overwhelm）；(3) 模糊判断错误→用户被迫重新用自然语言解释背景→"我没听懂"循环；(4) 计划迭代无界（反复改一步→agent 重新流式生成整张计划）。

**对抗核验结论**：`truly_ai_native_interaction`，方向正确但 Modify 流和批量聚合两处规约不足，是真实退化缺口。

**修正**：
1. **封堵路径1——Modify 为就地内联编辑**：点击 Modify 后 PlanDraftPanel 步骤列表直接进入可编辑态（字段变 input），用户直接改数值/增删步骤，而非打开纠正对话框；只有"添加依据"用一行 inline 文本，禁止多轮自然语言纠错对话；
2. **封堵路径2——批量先生成聚合仪表盘再下钻**：批量意图先产出 BatchSummaryDashboard（按风险/院系/金额聚类汇总），教师先在汇总层勾选/过滤，点击单条才下钻到该生 PlanDraftPanel，禁止一次性铺开 N 张计划卡；
3. **封堵路径3——模糊判断错误时编辑因子而非重述**：用户直接编辑 RiskAssessmentCard 的因子勾选/假设，系统重算置信度，而非让用户用自然语言重新解释背景；
4. **封堵路径4——迭代有界+引导直接编辑**：同一计划 regenerate 不超过 3 次，第 3 次后提示"建议直接编辑具体步骤而非重新生成"并高亮可编辑字段；
5. **置信度翻译为动作语义**（绿"可自动执行"/黄"需你确认后再执行"/红"建议多人会签或人工核验"，百分比作次要 detail 折叠；高风险学籍附"此决策不可逆"警示）；
6. **路由可解释**：每次路由到 copilot/out-of-loop 时审批卡顶部显示"本请求因[退费>¥5000/低置信度62%]需3人会签"；
7. **补全申请人侧流程**、**移动端分层降密度**（摘要卡→展开）、**Undo 边界明确化**（可逆/半可逆/不可逆分级，不可逆禁用 Undo 并提示线下冲正，禁止虚假回滚承诺）、**批量个体防淹没**（多因子叠加高风险个体置顶强制单独审批）。

**技术机制**：生成式 UI（Vercel AI SDK parts 模型，tool-call→component）。工具 `renderStudentSnapshot`/`renderPlanDraft`/`renderImpactPreview`/`renderRiskAssessment`/`renderApprovalGate`/`renderBatchApprovalTable`/`renderExecutionTrace`，execute 返回结构化数据作 props。durable 审批门后端（Cloudflare `waitForApproval(step,{timeout:'7 days'})` + `schedule()` 超时升级 + `approveWorkflow`/`rejectWorkflow` + 不可变 SQLite 审计表 + `pendingApprovals` 经 `setState()` 持久化）。MultiApprovalAgent 跟踪 requiredApprovals/currentApprovals 数组去重达阈才触发。内嵌 AI：学生快照选中"办理异动"→就地 inline 生成计划面板。

### 4.6 主动与 ambient 交互（通知/预警触达/家校沟通）

**传统交互**：导航驱动入口（点 3-4 层看某科某次成绩详情）；群发模板通知（全班级统一文案"作业未提交"，不分学生不分学科不分原因）；静态表单填报；周期性报表（月度 PDF 滞后）；微信群沟通（碎片化、时间顺序混乱、关浏览器即毁）；阈值预警（连续3次缺勤硬阈值发固定短信，无可执行建议）。

**痛点**：消息轰炸+信息碎片化（家长同时收作业平台/考勤/微信群/通知 app 多源推送，越忙越迷茫）；群发模板零个性；通知时机差（不考虑家长通勤/开会/做饭，统一即时推送打断心流）；无可行动 affordance（通知止于"告知"，家长不知该辅导什么怎么辅导找谁）；协同靠经验无数据支撑；模态单一；线性聊天陷阱。

**AI 原生交互设计**：外壳作锚（app 消息流容器/身份/品牌保留），内容区是 AI 原生生成的叙事卡+行动 affordance。ambient 三级升级+提议-确认两步+时机感知轻推。

- **A. 后台 ambient 监控（autopilot 零打扰）**：学情监控 agent 持续运行，app 折叠态仅显示脉动圆点+"Agent monitoring"标签，正常运作完全不通知。检测到机会/风险时按三级升级。
- **B. 时机感知推送（非即时轰炸）**：系统结合家长/教师日历与活动状态择时推送。通勤结束默认语音简短版+详情链接；晚饭后默认视觉卡片完整版。19:30 教师备课时段选"建议级"轻推而非全量告警。**上下文兜底**：模态切换=主动日历+被动 YAMNet 双信号，冲突时降级视觉卡片（最稳态）；连续3次语音无响应自动转文字版。
- **C. 主动推送叙事卡（非群发模板）**：以"王同学数学预警"为例，教师端收到动态生成的学生快照卡——叙事正文 3-5 句针对该生具体学情（"王同学数学函数章节掌握度3周内从85%降至62%，主因集中在二次函数图像类题，疑似前置知识坐标系变换未巩固"）。置信度徽章 78% 黄虚线。来源角标"12次作业+2次小测"可点击跳转原始数据。"Why this recommendation?"可折叠推理披露（离散因子列表）。**可行动 affordance 按钮组**4选项+dismiss：①生成针对性练习包（copilot草稿）②起草给家长的沟通建议（copilot草稿）③安排课后辅导（确认门，需教师+家长双确认）④dismiss"暂不处理"（无摩擦拒绝）。**建议节流**：per-user 建议预算/冷却+"不再提示此项"学习偏好；dismiss 后风险仍在才升级，不立即重提。
- **D. 可编辑草稿（可纠错）**：教师选②→AI 生成话术草稿（内嵌快照缩略+具体可执行辅导建议"建议本周陪孩子复习P45-50坐标系变换，每天15分钟，附件3道诊断题"）。草稿全段可编辑、可撤销、可版本回退。
- **E. 家长端多模态接收**：晚饭后收到视觉卡片完整版；通勤途中收到语音简短版（前置最重要信息）。家长可语音追问"具体怎么辅导"→AI 简短前置回答+推送详细步骤链接（非语音长篇）。家长一键确认执行辅导计划→autopilot 记录并后续跟踪。
- **F. 内联分析（情境化内嵌非侧栏）**：家长在 app 内打开成绩详情，选中某次下降分数→当前页内联展开"为什么这次下降"生成式 UI 组件，不跳转不开聊天侧栏。
- **G. 闭环反馈**：2 周后系统再评估掌握度回升至78%→推送"进步确认"轻推（零打扰级，仅脉动圆点变色）。

**生成组件**：ambient 监控折叠态组件、学生快照卡、掌握度趋势时间线、风险预警卡（三级升级）、可行动 affordance 按钮组、可编辑草稿编辑器、计划审批面板、多模态推送壳、内联分析浮层、进步确认轻推组件。

**用户流程**（系统主动检测王同学数学下滑→触达教师→教师起草家校沟通→家长多模态接收并执行辅导→闭环跟踪）：
1. （autopilot 零打扰）学情监控 agent 检测到掌握度 3 周内 85%→62%，置信度 78%。app 折叠态仅脉动圆点；
2. （时机感知）系统判断当前 19:30 教师备课、家长晚饭后，选"建议级"轻推；
3. （生成式 UI）教师端推送学生快照卡：叙事正文+置信度 78% 黄虚线+来源角标+可展开推理+4个 affordance 按钮；
4. （copilot 草稿）教师点"起草给家长的沟通建议"→AI 生成话术草稿→教师编辑确认后发送；
5. （家长端多模态）家长晚饭后收到视觉卡片完整版；通勤收到语音简短版；
6. （家长追问）家长语音问"具体怎么辅导"→AI 简短前置回答+推送详细步骤链接；
7. （确认执行）家长一键确认→autopilot 记录；
8. （闭环）2 周后掌握度回升至 78%→"进步确认"轻推（零打扰级）；
9. （审计）全程决策与理由写入不可变审计表。

**协同分级**：
1. **autopilot+仅通知**：日常学情同步（作业完成率/出勤/课堂参与度趋势）→自动记录，仅"机会级"轻推；
2. **copilot（生成草稿，人确认后执行）**：起草家校沟通话术、生成针对性练习包、生成学情报告→可编辑草稿+确认发送门。**批量反馈走置信分层审核**：High>90% 自动下发仅通知、Medium 70-90% 进"待确认列表"配 diff 高亮、Low<70% 必须逐条审核；提供"抽样3份预览→确认风格→批量生成"工作流，避免40次盲审；
3. **确认门+审批（高风险/不可逆）**：请假审批、课后辅导安排、家访、处分建议、跨学期调整→durable 审批门+多审批人阈值；
- 置信度阈值驱动升级（>90% 绿自动+仅通知；70-90% 黄执行附注+轻推确认；<70% 红暂停请求人工）；
- **HOTL 监督仪表盘充实**：actions/hour 基线+异常告警、agent 能力边界可视化（可调）、撤销粒度（单动作/批量/时段）、紧急停止后恢复流程。

**聊天框陷阱检查**：整体**不会**退化为聊天框——默认态是 ambient 折叠脉动圆点而非输入框、AI 入口恒产出 tool-call→component 而非纯文本 chat reply、无聊天侧栏、对话深度受限并收敛到 artifact、批量/表格走结构化面板。但**3 处薄弱接缝需封堵**：
1. **语音追问是最易滑向"语音聊天"的接缝**：家长问"具体怎么辅导"→AI 简短前置回答+详情链接，若不足再问，第3轮强制收敛为"学情报告 artifact"。但收敛目标可能错配意图——家长问的是"怎么辅导"（开放式建议，原则7本就允许走对话），报告 artifact 是"学情数据"，一刀切2轮转报告对开放式建议类追问违背混合正解，反而把该走对话的综合推理硬塞进结构化成品；
2. 文本意图入口若外壳被设计成聊天框样式，视觉上即等同 ChatGPT——须确保默认入口是 ambient 主动推送，文本/语音只是次级入口之一；
3. 若为 demo 在每张卡下挂"和 AI 聊聊"且产出纯文本回复，则局部退化为 bolt-on 聊天侧栏（Marc Friedman"无目的新奇"）。

**对抗核验结论**：`truly_ai_native_interaction`，上述修正用于封堵语音追问接缝、补齐学生主动入口与精确编辑路径、强化批量与上下文兜底。

**修正**：
1. **语音追问分类收敛而非一刀切2轮**：追问分两类——(a)数据查询类（"这次考了多少"）→转结构化卡片；(b)开放式建议类（"孩子抗拒数学怎么办"）→允许深度对话但每轮产出微型 artifact（建议卡/方法卡）而非纯文本，设硬上限（如5轮）后转完整 artifact；
2. **默认入口锚定 ambient 推送**：首页/默认态是"折叠态脉动圆点+机会级轻推"，文本/语音意图输入是次级入口，输入框文案是"描述你想关注的学生/学科"而非"和AI对话"；所有 AI 入口产出 tool-call→component，禁绝纯文本 chat reply 作为终态；
3. **批量 copilot 草稿走置信分层审核**（见协同分级）；
4. **精确编辑走直连表格而非 AI**：成绩录入/修正、出勤标记、基础信息维护→直连表格/表单组件（web 强项），AI 仅用于异常检测与叙事生成，不接管确定性 CRUD；
5. **补齐学生主动入口**：学生端 ambient 入口（作业中标记"这里没听懂"→系统聚合形成知识缺口卡，反哺教师学情监控），让学生从被动接收者变为数据贡献者，修复"夹在中间"痛点；
6. 上下文兜底（双信号+降级）；
7. **置信度人话化**（三态徽章配自然语言注脚）；
8. 管理员 HOTL 充实。

**技术机制**：tool-call→component 渲染（Vercel AI SDK），工具 `generate_student_snapshot`/`draft_family_communication`/`schedule_tutoring`/`generate_practice_pack`，Zod inputSchema 强约束 props。streamUI async generator 渐进式流式 UI。inline AI 选中成绩→就地 `generate_inline_analysis`。durable 审批门（Cloudflare waitForApproval + schedule + 不可变 SQLite 审计表 + MultiApprovalAgent）。端侧：唤醒词激活语音（仅有意激活时处理音频，隐私），YAMNet 环境噪声分类驱动模态切换。置信度可视化三态。

## 五、四类角色的交互差异

| 维度 | 教师 | 家长 | 学生 | 管理员 |
|---|---|---|---|---|
| **核心诉求** | 高效管 40+ 学生、批量对比找风险 | 看自己孩子趋势+可执行辅导建议 | 自主查看进步、不被排名标签化 | 权限管理/数据质量/跨班全局风险 |
| **默认落地** | 探索画布/学生快照卡（风险/批量/草稿） | 预生成卡面（本周学习情况）+叙事卡 | 进步卡（正面导向） | 数据质量看板/跨班风险排名 |
| **置信度呈现** | 数值三态+SHAP top-3+sources 角标 | 自然语言三态（"较强把握/不太确定/建议和老师确认"）+一句话来源 | 不展示风险评分与排名 | 数值+全局异常告警 |
| **风险信息** | 完整风险时间线+因子 | 不展示风险分原始值与排名，进步标绿 | 行动建议而非诊断标签 | 跨班风险排名+actions/hour 异常 |
| **"问人"通道** | — | 红态徽章带"联系班主任"按钮 | — | HOTL 仪表盘 Emergency Stop |
| **协同分级侧重** | copilot 草稿+批量确认门 | 一键确认执行辅导计划 | 无审批门（只读+标记困惑点） | out-of-loop 审批门+多审批人阈值 |
| **移动端降级** | 完整组件 | 简化叙事卡+红绿灯 | 进步时间线 | 分层降密度（摘要→展开） |
| **角色变体核心差异** | 效率/批量/风险发现 | 信息+可执行指导 | 自主+进步导向 | 权限/全局/系统健康 |

**角色自适应关键点**：家长/学生变体不展示原始置信度数值与学术角标，仅自然语言三态+一句话来源；雷达/时间线默认折叠或仅教师/管理员可见；学生苏格拉底式"给提示非给答案"（Khanmigo 式）；管理员补缺位（权限变更走 autopilot 审批门）。

## 六、移动端与多模态入口（语音/相机/主动通知/端侧）

- **命令面板文本**（Cmd+K 桌面 / 长按移动端）：主入口之一，带建议提示词/历史查询/热门意图按钮填补空白页。
- **语音**：家长接送途中"孩子今天表现"→回答前置简短一两句最重要信息在前+随后推详情链接（非语音长篇）。语音只捕获意图（"给王同学办休学"），结果落结构化卡片+屏幕兜底，非全程对话。语音→意图草稿（可编辑预览）→确认再生成卡，避免识别歧义直接产出错误卡。
- **相机**：教师拍学生作业/纸质成绩单/缴费收据即时分析录入（VLM 场景理解）。
- **表格点击/选中**：inline AI 操作入口，操作长在选区（非侧栏）。
- **文件导入**：成绩单 Excel/批量照片→AI 提取草稿表。
- **触摸长按**：移动端意图入口+选区 inline AI。
- **音频环境感知**：YAMNet 噪声分类驱动模态切换（嘈杂→视觉卡片，安静→可语音），主动日历+被动 YAMNet 双信号，冲突降级视觉卡片（最稳态）。
- **端侧**：唤醒词激活语音处理（仅有意激活时处理音频，隐私保障），端侧推理保障用户数据安全。
- **移动端组件降级策略**：每组件定义移动端等价物（雷达→条形列表、热力图→简表、时间线→卡片流），保证不退化为纯文本聊天摘要。PlanDraftPanel 等高密度卡片在手机先显示摘要卡（3 行），展开后才显示完整步骤。

## 七、可访问性与非技术用户（教师/家长如何不被 AI 交互劝退）

1. **降权命令面板、提权表格**：数据密集场景以增强表格/智能表单/可编辑文档为一等点击入口（默认落地），任何人都不被强制打字——上下文锚（班级/学期/学科）选择即默认浏览视图，提供 always-available 的点击逃逸舱。命令面板仅作"模糊起点"快捷方式。
2. **置信度人话化**：对非技术用户，置信度三态显示为动作语义而非百分比——绿"可自动执行"/黄"需你确认后再执行"/红"建议多人会签或人工核验"，百分比作为次要 detail 折叠。家长/学生变体仅自然语言三态+一句话来源。
3. **空白页焦虑防御**：命令面板预填建议提示词+历史记录+热门查询按钮，不暴露空白聊天框；建议覆盖率 SLA>70%，低于阈值自动把高频自由文本提升为 chip。
4. **答案模式 vs 探索模式**：低 agency 用户（家长/学生）默认"答案模式"（单聚焦卡+建议下一步，或预生成卡面），高 agency 用户（教师/管理员）默认"探索模式"（全画布拖拽连线），一键切换。
5. **移动端分层降密度**：摘要卡→展开；置信度图例始终可见；默认视图"仅看需复核"降低滚动与盲目全批准风险。
6. **路由可解释**：每次路由升级显示"本请求因[风险因子/置信度]需 N 人会签"，让用户理解为何被升级而非视为系统"看心情"。
7. **学生主动入口**：学生端 ambient 标记困惑点→聚合知识缺口卡反哺教师，从被动接收者变为数据贡献者。
8. **渐进引导**：首次进入引导式示例查询降低教师迁移成本；Cmd+K 是高级 affordance，浮动"录入"按钮必须视觉上明显是采集工具（相机/文件图标）且与聊天输入框视觉区分。

## 八、交互自检清单（如何判断交互是否真 AI 原生而非聊天框）

**底线测试（Marc Friedman）**：每个动作都要能回答"这比用户今天能做的更快、更清晰、更可靠吗？"——查表格/录成绩结构化 UI 是，综合推理对话是。

- [ ] **1. 意图是否被路由到生成式 UI/结构化组件，而非直接进对话？** 数据密集/确定性事务（查/录/批/报）必须走组件，不能塞进对话。意图路由器是守门人。
- [ ] **2. AI 输出是否恒为 tool-call→受控组件（非任意 HTML、非纯文本 chat reply）？** Vercel AI SDK parts + Zod inputSchema 作 props 契约。全系统不得出现任何"问 AI"按钮。
- [ ] **3. 数据密集表格/批量操作/精确编辑是否保留 web 强项（AI 增强+inline 操作）？** 表格保持表格，精确 CRUD 直连组件 inline edit 不走 AI。
- [ ] **4. 对话是否仅用于模糊意图/综合推理/开放式探索？** 且对话产物落回快照组件（生成新卡片）而非留在聊天流；超阈值轮数未收敛给候选快照入口。
- [ ] **5. 命令面板是否为真·一次性（Linear 式，生成即关闭清空）？** 拒收续聊语义，追问走快照内组件交互或强制重生成整张快照。
- [ ] **6. 是否情境化内嵌（选中→inline 操作长在选区）而非另开聊天侧栏？** 无聊天侧栏。
- [ ] **7. 每个 AI 输出是否为可编辑草稿（可改可撤销可版本回退）？** 高风险/不可逆是否有 durable 确认门（可暂停跨天+超时升级+不可变审计表）？
- [ ] **8. 是否表达不确定性（置信度三态+推理披露+来源角标）？** 红<70% 是否带明确处置按钮/内联修复 chips 而非只暂停？
- [ ] **9. copilot/autopilot/out-of-loop 是否映射到对应 UI（确认门/审批/通知分级）？** 路由是否可解释？
- [ ] **10. ambient 主动推送是否三级升级+择时+可 dismiss+per-user 节流？** 默认态是否为折叠脉动圆点而非输入框？
- [ ] **11. 置信度是否对非技术用户降维（自然语言三态）？** 家长/学生是否不展示原始数值与排名？
- [ ] **12. 移动端是否有组件降级等价物（不退化为纯文本聊天摘要）？** 语音是否只捕获意图、回答前置简短？
- [ ] **13. Modify/纠错是否走结构化内联编辑（字段级）而非自然语言对话循环？** 计划迭代是否有界（≤3 次后引导直接编辑）？
- [ ] **14. 批量场景是否先生成聚合仪表盘再下钻（禁止卡片洪流）？** 批量反馈是否走置信分层审核？
- [ ] **15. Undo 是否按操作可逆性分级（不可逆禁用并提示线下冲正，禁止虚假回滚承诺）？**
