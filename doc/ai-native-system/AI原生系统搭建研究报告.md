# AI 原生系统搭建研究报告

> 基于 TaskingAI、Infinity、DeepBI 三个真实开源项目的深度数据获取与四维度标准定义撰写。所有项目分析均基于 GitHub API 真实获取的仓库元数据、README、目录结构与关键源码原文，未编造项目细节。

---

## 1. 执行摘要

AI 原生系统（AI-Native System）是指：以大模型/智能体的推理决策为系统的控制中枢、以语义表示为数据层第一公民、以声明式目标驱动业务流程编排、以模型能力为产品价值发生器的系统形态。其与 AI 赋能（AI-Enhanced）的本质区别，可用一句最锐利的判别概括——**移除大模型后，AI 原生系统的核心业务流程瘫痪（只剩空壳工具），AI 赋能系统只是丢掉一个功能（核心事务流照常运行）**。

本报告基于三个真实开源项目的代码级证据展开：**TaskingAI**（5382 stars，平台层，LLM Agent BaaS，以 pgvector 为唯一库 + 独立推理微服务 + 会话编排运行时）、**Infinity**（4582 stars，数据层，AI 原生数据库，以 C++23 从零构建 dense/sparse/tensor/full-text 四类检索为一等算子）、**DeepBI**（2357 stars，应用层，AI 原生数据分析，以 28 个 LLM Agent 的对话即业务逻辑）。

核心结论有三：第一，AI 原生不是全有或全无的二元标签，而是**分层的纯度概念**——三个项目分别在数据层、平台架构层、应用逻辑层达到了原生，但各自也都有"非原生层"作为务实基底；第二，**原生发生在哪一层，决定了系统的能力边界与代价**——数据层原生换来检索质量但无 LLM 编排，应用层原生换来业务自主但强依赖 LLM 成本与稳定性；第三，真正的工程智慧在于**在关键承重层做原生、在辅助层保留确定性**，而非追求全栈概率化。

## 2. 定义：什么是 AI 原生系统

### 2.1 精确定义（综合四维度）

AI 原生系统是在**架构、数据、流程、产品**四个维度上，均把大模型能力作为根基层（而非附加层）来设计的系统：

- **架构层面**：大模型/智能体是控制平面（control plane），请求路由、流程编排、状态推进由模型推理决策驱动，确定性代码下沉为工具与边界适配器。
- **数据层面**：语义表示（向量/嵌入）是第一公民，与原始数据同源共存、同事务写入，混合检索是默认读取路径。
- **流程层面**：流程以"目标 + 约束 + 验收标准"声明，由 AI 在运行时动态规划步骤序列，移除 AI 后流程无法运行。
- **产品层面**：AI 是价值发生器（engine）而非效率加速器，移除模型后核心价值闭环消失或退化为完全不同的非 AI 产品。

### 2.2 AI 原生 vs AI 赋能 核心判别准则

| 判别维度 | AI 原生系统 | AI 赋能系统 |
|---------|------------|------------|
| **移除模型测试** | 核心业务流程瘫痪（完成率≈0%） | 仅丢失某项功能，核心事务流照常（完成率>50%） |
| **路由决策归属** | 路由在模型推理（agent 依据上下文决定路径） | 路由在代码（controller/if-else/状态机） |
| **能力扩展方式** | 注册工具到 agent 目录 | 加 if-else 分支或微服务 |
| **模型角色定位** | 模型是控制中枢（orchestrator） | 模型是叶子函数（`result=llm(prompt)`） |
| **数据层脊梁** | 向量/语义检索为默认读路径，embedding 同事务写入 | 关系库 CRUD 为脊梁，向量是 opt-in 功能页 |
| **流程定义形态** | 声明式目标+约束，路径动态生成 | 命令式步骤序列，路径固定 |
| **输出形态** | 按上下文动态生成（千人千面） | 固定模板/页面，AI 仅填空润色 |
| **失败处理** | 为概率输出专门设计纠偏/置信度/兜底 | 把 AI 输出当确定结果直接展示 |
| **数据飞轮** | 采集数据服务于上下文/记忆/评估，越用越贴合 | 仅运营 CRUD，不复利 |

一句话分界：**AI 原生 = 移除 AI 即无产品；AI 赋能 = 移除 AI 仍有产品，仅降级。**

### 2.3 自检清单：5 个判别问题

回答"是"越多，越接近 AI 原生：

1. **移除测试**：若把大模型 stub 为返回空/报错，系统的 Top-5 核心任务还能完成几个？（原生目标 ≤ 5%）
2. **路由归属**：系统的主请求路径是否必须经过 agent loop（感知→推理→工具调用→反馈），还是 `controller→service→llm()` 把模型当函数调用？
3. **扩展方式**：新增一项能力，是只需注册一个 tool，还是要改控制流/加 if-else 分支？
4. **数据脊梁**：系统的核心读操作中，以语义相关性为选择标准的占比是否 >50%？embedding 是否同事务写入、版本化治理？
5. **产品价值来源**：产品的核心价值是否是一个"大模型出现前因成本/规模/质量而不可行"的能力？撤掉 AI，产品是塌缩还是仅降级？

## 3. 标准体系：AI 原生系统的四大维度标准

### 3.1 架构设计标准

**定义**：AI 原生架构把大模型/智能体作为系统的控制中枢，请求路由、流程编排、状态推进由模型推理决策驱动，确定性代码被下沉为工具与边界适配器。系统的能力边界等于智能体可调用的工具目录，记忆与评估是架构的一等子系统。

**标准清单**：

| 标准 | 核心要求 | 检查点 |
|------|---------|--------|
| 模型即编排器 | 智能体推理循环是顶层控制流 | 主请求路径必须经过 agent loop，非 controller→service→llm() |
| 确定性下沉到边界 | 数学计算、持久化、外部 API 下沉为工具 | happy path 分支决策来自模型推理，非硬编码路由表 |
| 记忆作为一等子系统 | 显式分层短期上下文/长期记忆/工作记忆/情景记忆 | 存在独立 memory layer，而非把历史全量拼进 prompt |
| 工具调用即集成平面 | 外部能力统一暴露为 agent 可调用工具 | 新增能力只需注册 tool，无需改控制流 |
| Schema 后置而非前置 | 模型先产出半结构化意图，schema 作输出约束 | 容忍非结构化输入，允许模型先推理再结构化 |
| 评估与可观测作为承重层 | 内建 trace、eval 数据集、golden 用例、guardrail | eval 是否纳入 CI |
| 工具幂等与补偿作为约束 | 工具层幂等或提供 saga/补偿语义 | 重复 tool call 安全，危险动作有 HITL 审批门 |

**反模式**：
- LLM 即微服务/边车（sidecar）：模型封装在 REST 端点后被当普通服务调用，编排路由全留在确定性代码。
- 把 prompt 塞当状态：用全量历史拼进 prompt 来"保存状态"，无工程化记忆层。
- 硬编码智能体图：agent 建成设计期固定的有向图（A 永远→B），路径由图作者决定而非模型推理。
- Schema 在入口扼杀模型：强制模型输出先过严格 JSON schema 校验才能进入推理。

**可量化指标**：控制流归属比（模型推理完成的路由决策占比）、移除模型可用性测试（核心流程完成率）、工具耦合覆盖度、评估面覆盖率、概率核心比（agent/reasoning 与确定性编排层的代码占比）。

### 3.2 数据层设计标准

**定义**：AI 原生数据层以语义表示为第一公民，原始数据与其向量/语义表示同源共存、同事务写入，混合检索是默认读取路径，并存在机器可内省的语义层供模型自主发现数据。治理对象从列级权限扩展到 chunk/embedding 级溯源与嵌入版本化。

**标准清单**：

| 标准 | 核心要求 |
|------|---------|
| 语义表示与原始数据同源共存、同事务写入 | embedding 是 first-class 持久化状态，与原始记录在同一写操作内落库 |
| 混合检索为默认读取路径 | 单次 pass 内融合向量相似度+词法（BM25）+结构化过滤+重排 |
| 存在机器可内省的语义层 | 本体/知识图谱/机器可读语义 schema，可被 agent 运行时查询发现 |
| 检索是 token 预算感知的 | 返回经排序、去重、上下文裁剪的 chunk，尊重模型上下文窗口 |
| Embedding 作为版本化受治理状态 | 携带溯源（源记录、chunk 边界、嵌入模型+版本、时间戳） |
| 写入即语义化 | 数据写入时语义索引在同一操作内更新（事务内或确定性 CDC） |
| 跨模态统一语义空间 | 文本/图像/音频/结构化数据投射到同一可查询语义空间 |
| 检索变更受评估门禁约束 | 每次变更通过 recall@k/precision/faithfulness/MMR 评估套件 |

**反模式**：Vector DB 螺栓（保留关系 schema 加个 pgvector 列便自称原生，但核心业务流从不走语义检索）、RAG 旁路服务（CRUD 与业务逻辑从不触及语义层，AI 只是只读消费者）、Embedding 黑箱（存了向量无版本无溯源）、语义层=数据字典（人写字段说明当语义层，静态不可内省）、关键词检索+AI 后处理（检索非语义，仅对结果用 LLM 重排）。

**可量化指标**：语义检索占核心读路径比例（建议 >50%）、Embedding 新鲜度滞后 P95（原生秒/分钟级）、Chunk 级溯源覆盖率（原生接近 100%）、检索评估门禁覆盖率、语义层可内省性。

### 3.3 业务流程标准

**定义**：AI 原生业务流程的控制流（步骤编排、分支决策、异常处理、资源调度）由 AI 模型在运行时基于输入与上下文动态生成，而非由预编码的确定性状态机驱动；流程以"目标+约束+验收标准"声明并交由 AI 端到端编排。

**标准清单**：

| 标准 | 核心要求 |
|------|---------|
| 声明式流程定义 | 流程以目标+约束+验收标准声明，AI 运行时规划步骤序列 |
| 运行时动态编排 | 步骤选取、顺序、是否调用由 AI 依据上下文决定 |
| AI 驱动分支决策 | 关键分支由 AI 判断（含置信度与依据），非 if-else 规则树 |
| 异常即重规划 | 处理不了的输入触发 AI 基于新信息重新规划，非固定错误码/人工队列 |
| 闭环自演进 | 执行结果回流改进提示/检索/策略/模型，有版本可追溯 |
| 人在判断节点 | 人参与目标校准/责任审核/伦理监督，非数据录入工位 |
| 语义化流程状态 | 状态以 AI 可消费的语义上下文管理，支撑跨步骤连续编排 |
| 决策链可观测可审计 | 每个 AI 决策点记录输入/输出/推理依据/模型版本/置信度 |

**反模式**：固定流水线+AI 填空（步骤顺序写死，仅某步调 AI）、规则树前置 AI（分支由 if-else 决定，AI 只在叶子节点执行）、AI 孤岛无闭环（结果不回流不改进）、人工兜底即异常处理（遇难题转人工队列，无 AI 重规划）、对话入口包装传统流程（包一层 AI 对话入口，底层控制流仍是后端状态机）。

**可量化指标**：动态编排率（AI 生成不同步骤序列的请求占比）、AI 决策可追溯覆盖率（目标 100%）、闭环改进时延、零代码适配率、AI 决策节点占比。

### 3.4 产品设计标准

**定义**：AI 原生产品设计的核心价值主张、交互范式与信息架构以大模型能力为"价值发生器"而非"效率加速器"——移除模型后，核心价值闭环即不复存在或退化为完全不同的非 AI 产品。

**标准清单**：

| 标准 | 核心要求 |
|------|---------|
| 价值主张溯源能力差距 | 核心价值必须是 AI 出现前不可行的能力，非对既有任务的便捷升级 |
| 意图优先的交互范式 | 主输入路径是自然语言/多模态意图，GUI 是可绕过的脚手架 |
| 移除模型即无产品 | 禁用模型后主流程无可用路径或退化为完全不同的非 AI 产品 |
| 生成式、千人千面的输出 | 交付物按用户与上下文实时生成，边际个性化成本趋近于零 |
| 为不确定性设计的 UX | 暴露置信度/备选，支持纠偏/编辑/撤销，幻觉兜底 |
| 以记忆/上下文为核心的数据飞轮 | 采集数据服务于上下文/记忆/反馈信号，AI 随使用变好 |
| 首次交互即 AI 主导 | 首启以 AI 行动开场，非空白仪表盘加手动配置 |

**反模式**：GUI 优先+AI 侧边栏（主流程点菜单填表单，AI 是抽屉里的聊天机器人）、单按钮 AI（挂一个"用 AI 总结/生成/搜索"入口，核心价值链与 AI 无关）、固定提示词模板冒充 AI（输入固定输出固定无上下文无个性化）、把模型输出当确定性真相（无置信度无纠错无撤销）、无状态单轮调用（每次交互与历史割裂不记忆不学习）。

**可量化指标**：模型移除后核心任务完成率（原生 ≤5%）、自然语言意图完成覆盖率（原生 ≥70%）、上下文复用/记忆命中率（原生 ≥50% 且随使用单调上升）、动态生成界面/输出占比（原生 ≥60%）、AI 输出反馈闭环覆盖率（原生 ≥80%）。

## 4. 如何实现：从 0 到 1 搭建 AI 原生系统

### 4.1 分阶段落地路径

#### 阶段一：需求重塑——找到"AI 出现前不可行"的能力缺口

- **关键动作**：不要问"现有业务哪里能用 AI"，而要问"哪个任务在大模型出现前因成本/规模/质量而不可行"。若该任务已有成熟解决方案且体验不差，则属增强而非原生。
- **验证标准**：通过"移除模型测试"——若移除 LLM，这个产品是否塌缩为完全不同的东西或根本不存在。原生产品存在的理由是填补 AI 才闭合的能力缺口。
- **产出物**：能力缺口声明书（一句话讲清"AI 之前为什么做不到"）。
- **真实印证**：DeepBI 的 autopilot 模块（端到端无人介入生成 HTML 分析报告）是 AI 出现前不可行的能力——传统 BI 不可能"自动驾驶"完成全链路分析，这就是它的原生理由。

#### 阶段二：架构设计——确立 agent loop 为控制中枢

- **关键动作**：让主请求路径必须经过 agent loop（感知→推理→工具调用→反馈），而非 controller→service→llm()。把确定性逻辑（计算、持久化、外部 API、格式校验）下沉为工具或边界适配器。
- **验证标准**：happy path 的分支决策应来自模型推理，而非硬编码路由表。新增能力只需注册一个 tool，无需改控制流。
- **产出物**：控制流归属比埋点（统计由模型推理完成的请求路由决策占比）。
- **真实印证**：DeepBI 的 task 路由由 TaskSelectorAgent（另一个 LLM Agent）决策而非规则分支；TaskingAI 的 assistant/generation/session.py 是以会话为核心的 LLM 编排运行时，区分 stateful/stateless × stream/normal 四种会话。

#### 阶段三：数据底座——语义表示作为第一公民

- **关键动作**：从存储选型起就以向量/语义检索为一等公民，而非关系库事后加 AI 搜索。embedding 与原始数据同源共存、同事务写入。建立机器可内省的语义层。
- **验证标准**：移除 AI 能力后核心读写是否瘫痪。embedding 是否版本化受治理（携带源记录、chunk 边界、嵌入模型+版本、时间戳）。语义检索占核心读路径比例 >50%。
- **产出物**：混合检索默认读取路径（向量+词法+结构化过滤+重排，单次 pass 完成）。
- **真实印证**：TaskingAI 选 pgvector 为唯一持久化库、database_ops/retrieval 作为独立持久化子层与 auth 并列；Infinity 把 dense/sparse/tensor/full-text 作为表的一等列类型与 integer/varchar 同级，存储引擎内建 9 种索引族。

#### 阶段四：流程编排——声明式目标驱动运行时动态规划

- **关键动作**：流程以"目标+约束+验收标准"声明，由 AI 在运行时规划步骤序列。关键分支由 AI 判断（含置信度与依据）。异常触发 AI 重规划而非跳到固定错误码。
- **验证标准**：动态编排率（同类业务不同输入产生不同路径）。AI 决策可追溯覆盖率 100%（记录输入/输出/推理依据/模型版本/置信度）。
- **产出物**：决策链可观测可审计的日志体系。
- **真实印证**：DeepBI 的每一步（理解 schema→生成 SQL→执行→可视化→分析）都是 LLM Agent 之间的 initiate_chat 对话，流程编排权在 LLM 不在 if/else。

#### 阶段五：产品交互——意图优先与为不确定性设计

- **关键动作**：主输入路径是自然语言意图，GUI 是可绕过的脚手架。为概率输出专门设计纠偏/置信度/备选/撤销/幻觉兜底。首启以 AI 行动开场。
- **验证标准**：把 GUI 关掉只留意图输入，主流程仍可完成。模型移除后核心任务完成率 ≤5%。
- **产出物**：动态生成界面/输出占比统计（原生 ≥60%）。
- **真实印证**：DeepBI 的对话即产品（唯一交互范式），README 三大功能全部以"Conversational"开头；TaskingAI 的 Assistant 配置项 memory="naive" 把"记忆策略"作为一等产品参数。

#### 阶段六：评测迭代——闭环自演进

- **关键动作**：内建 trace、eval 数据集、golden 用例、guardrail，eval 纳入 CI。执行结果回流改进提示/检索/策略/模型，且有版本可追溯。
- **验证标准**：评估面覆盖率（模型治理输出被 eval/golden trace/guardrail 覆盖的比例）。闭环改进时延。
- **产出物**：检索变更评估门禁（recall@k/precision/faithfulness/MMR）。
- **真实印证**：Infinity 把 benchmark 作为一等文档（0.1ms/15K QPS），Roadmap 2025 持续围绕检索能力演进；DeepBI 的 version.md 显示迭代重心（token 优化、Agent 重试、TaskSelector、多 LLM）全在 AI 层。

### 4.2 技术选型建议（结合三个真实项目）

| 层次 | 推荐选型 | 参考项目 | 选型理由 |
|------|---------|---------|---------|
| 向量数据库 | PostgreSQL + pgvector（中等规模）/ 自研 C++ 引擎（极致性能） | TaskingAI / Infinity | pgvector 事务完整易上手；Infinity 的 9 族索引+MLAS 矩阵乘适合 ColBERT 张量检索 |
| LLM 推理抽象 | 独立微服务封装多提供商统一接口 | TaskingAI inference 服务 | 30+ 提供商统一为 chat completion+embedding+function call，消除厂商锁定 |
| 多智能体编排 | AutoGen / LangGraph | DeepBI (AutoGen) | initiate_chat 对话协议驱动 Agent 间协作 |
| 混合检索融合 | RRF / 加权求和 / ColBERT 重排 | Infinity | 多路召回→粗排→精排在存储执行层完成 |
| 记忆层 | 独立 memory layer（短期上下文+长期向量+工作记忆） | DeepBI ChatMemoryManager | 按 user_id 维护消息池，非全量拼 prompt |
| token 预算管理 | tiktoken 实时统计+裁剪 | DeepBI base_config.py | max_token_num 约束上下文，数据组织首要约束是 token 预算 |
| 评估面 | trace + eval 数据集 + golden 用例 + guardrail 入 CI | — | 概率系统必须用评估面兜底正确性 |
| 部署形态 | Docker Compose 微服务编排 | 三项目均采用 | 一键拉起多服务进入生产 |

### 4.3 常见陷阱与避坑

1. **Vector DB 螺栓陷阱**：保留关系 schema、加一个 pgvector 列便自称原生，但核心业务流从不走语义检索。避坑：检查"语义检索占核心读路径比例"，而非看有没有装向量库。
2. **CRUD 优先+AI 外挂层**：数据模型、API、持久化是经典 CRUD，在其上加一层"AI 服务"。避坑：移除 AI 后产品是否毫无变化？若是，架构脊梁是 CRUD 数据库，AI 是功能模块。
3. **硬编码智能体图**：把 agent 建成设计期固定的有向图（A 永远→B），模型只填充节点内容。避坑：路径是否由模型推理决定？若由图作者决定，这是"LLM 节点的状态机"。
4. **对话入口包装传统流程**：包一层 AI 对话入口，底层控制流仍是后端状态机。避坑：AI 只做意图识别后调用既有接口吗？若是，控制流仍在确定性代码手里。
5. **Embedding 黑箱**：存了向量但无版本、无溯源、无 re-embedding 策略。避坑：嵌入模型升级时检索静默退化无人知晓——必须版本化受治理。
6. **强 LLM 依赖的成本与稳定性风险**：每步分析触发多轮 Agent 对话，token 消耗大、延迟高，受 LLM 输出规范性影响（DeepBI 的 version.md 多次修"Agent 响应不规范导致对话中断"）。避坑：建立代码规范检测+重试机制，为概率输出设计纠偏。
7. **LLM 权重全外部依赖**：平台本身不承载模型推理（TaskingAI 调 provider API），推理延迟与成本受外部厂商制约。避坑：明确区分"AI 编排基座"与"AI 计算基座"，本地模型支持（Ollama/LM Studio/LocalAI）作为降级方案。

### 5.3 三个项目分别教会我们什么

#### TaskingAI 的启示

1. **AI 原生体现在"管理对象是 LLM/向量/Agent"而非架构范式创新**：后端采用常规 FastAPI 分层（routes/services/models），但领域模块全部围绕 AI 原语切分（assistant/inference/model/retrieval/tool），LLM 推理拆为独立可部署微服务——原生不要求架构范式颠覆，而要求 AI 是被管理的核心对象。
2. **多 LLM 提供商统一抽象消除厂商锁定**：30+ 提供商统一为 chat completion+embedding+function call 接口 + 本地模型支持（Ollama/LM Studio/LocalAI），这是 AI 原生架构"模型可替换核心底座"的产品化验证。
3. **直击 LangChain 无状态、OpenAI Assistant API 耦合两大痛点**：通过 stateful/stateless × stream/normal 四种会话运行时 + OpenAI 兼容路由，天然支持多租户与"AI 逻辑与产品开发分离"的 BaaS 范式。

#### Infinity 的启示

1. **AI 原生数据底座：检索算子是查询计划器的一等公民**：kKnnScan、kMatchTensorScan、kMatchSparseScan、kFusion 是与 kTableScan、kJoin 同级的物理算子，可被 planner 统一优化、并行调度——向量/张量检索不是插件函数，而是查询引擎的原生算子，移除则系统不成立。
2. **多路召回→粗排→精排全在存储执行层完成**：9 族索引按数据形态分治（HNSW/IVF/DiskANN/sparse/multivector/ColBERT 张量），内嵌 MLAS 矩阵乘把张量检索矩阵运算下沉到 SIMD/BLAS 级优化——混合检索融合不是应用层拼接，而是数据库内置算子。
3. **AI 原生有边界：它是 RAG 数据底座而非全栈**：项目不含 LLM 推理/Agent 编排，端到端 AI 业务流程需上游应用（如 RAGFlow）补齐——原生纯度高不等于全栈覆盖，要诚实标注原生的层次边界。

#### DeepBI 的启示

1. **Agent 即逻辑：移除 AI 引擎产品价值归零**：28 个 LLM Agent 的 initiate_chat 对话承载"理解 schema→生成 SQL→执行→可视化→分析"全部业务逻辑，autopilot 端到端无人介入生成 HTML 报告是 AI 原生的标志性能力——这是判断 AI 原生最强的证据。
2. **LLM 后端可插拔证明 LLM 被抽象为核心底座**：2.0.0 起支持 DeepSeek/glm-4/OpenAI/Azure/Claude3/百度千帆/阿里百炼可替换，换底层 LLM 不影响上层 Agent 编排——这是 AI 原生架构的关键验证点。
3. **token 预算驱动的数据组织**：max_token_num + tiktoken 实时裁剪上下文，数据层为 LLM 对话而非报表优化——AI 原生数据层首要约束是 token 预算而非表结构，这是与 BI 赋能系统的本质区别。

## 6. 结论与建议

### 6.1 核心洞察

1. **AI 原生是分层纯度概念，不是全有或全无**。三个项目分别证明了数据层（Infinity）、平台架构层（TaskingAI）、应用逻辑层（DeepBI）的可行原生路径。真正的原生发生在哪一层，决定了系统的能力边界与代价。不要追求"全栈概率化"，而要在关键承重层做原生、在辅助层保留确定性——DeepBI 保留 Redash 作为执行基底、TaskingAI 用常规 FastAPI 分层承载 AI 原语，都是务实的工程取舍。

2. **"移除 LLM 测试"是最锐利且最廉价的判别工具**。无论架构多复杂、用了多少向量库，只需做一个故障注入：把大模型 stub 为返回空，测量核心业务流程完成率。原生 = 0%（瘫痪），赋能 > 50%（仅降级）。三个项目都通过了这个测试——TaskingAI 没有 LLM 推理微服务则 generate_message 主链断裂，Infinity 移除 AI 检索算子则系统失去存在意义，DeepBI 移除 ai/ 引擎则退化为普通查询器。

3. **原生的代价与边界必须诚实标注**。纯度高也意味着强 LLM 依赖带来的成本（每步多轮 Agent 对话 token 消耗大）与稳定性风险（受 LLM 输出规范性影响，DeepBI 多次修"Agent 响应不规范导致对话中断"）。且各项目都有"非原生层"：DeepBI 的 Redash 基底、Infinity 无 LLM 编排、TaskingAI 的 LLM 权重全外部依赖。诚实的原生不是假装全栈 AI，而是明确"原生发生在哪一层、辅助层为何保留确定性"。

4. **评估面与闭环自演进是三个项目的共同短板**。三个项目均未建立完整的 eval plane（trace+eval 数据集+golden 用例+guardrail 入 CI）与闭环自演进（结果回流改进提示/检索/策略/模型）。这意味着它们在"架构"与"数据"维度原生度高，但在"流程闭环自演进"维度仍是未完成的原生——这是从 0 到 1 搭建时最容易被忽视、却最决定长期质量的承重层。

5. **数据层原生是地基，但常被低估**。Infinity 证明：把 dense/sparse/tensor/full-text 四类检索作为存储、索引、查询计划、执行算子的根基全部重写，而非给关系库加向量插件，是 AI 原生最硬的证据。TaskingAI 从存储选型起选 pgvector 而非关系库+AI 插件，也体现了这一原则。数据层是否原生，决定了系统的语义检索能力上限。

### 6.2 给从 0 到 1 搭建 AI 原生系统的团队的落地建议

1. **先做移除测试再做架构**：在设计阶段就回答"移除 LLM 后这个产品是什么"。若答案是"一个体验稍差但完整可用的产品"，回去重新找能力缺口。原生产品存在的理由必须是"AI 才闭合的能力缺口"，而非对既有任务的便捷升级。

2. **数据层优先于架构层原生**：从存储选型起就以语义检索为一等公民。中等规模选 pgvector（事务完整易上手），极致性能需求再考虑自研引擎。embedding 与原始数据同源同事务写入、版本化治理（携带源记录、chunk 边界、嵌入模型+版本、时间戳）。避免"Vector DB 螺栓"——装了向量库但核心业务流从不走语义检索。

3. **把 agent loop 设为主请求路径，确定性逻辑下沉为工具**：主流程必须经过感知→推理→工具调用→反馈。新增能力只需注册一个 tool，无需改控制流。建立"控制流归属比"埋点，量化由模型推理完成的请求路由决策占比。

4. **从第一天就建评估面，而非事后补**：概率系统必须用 eval 兜底正确性。建立 trace（每个 AI 决策点记录输入/输出/推理依据/模型版本/置信度）、eval 数据集、golden 用例、guardrail，并纳入 CI。检索变更必须通过 recall@k/precision/faithfulness/MMR 评估门禁。这是三个真实项目的共同短板，也是从 0 到 1 最该做对的事。

5. **为 LLM 的不确定性与成本设计**：暴露置信度/备选，支持纠偏/编辑/撤销，设幻觉兜底。建立代码规范检测+重试机制（DeepBI 的 Echart 代码规范检测+重试值得借鉴）。token 预算作为数据组织的一等约束（tiktoken 实时统计+裁剪）。多 LLM 后端可插拔（DeepSeek/glm-4/OpenAI/Azure 等）作为降级与成本控制手段。

6. **诚实标注原生的层次边界**：明确"原生发生在哪一层、辅助层为何保留确定性"。不要假装全栈 AI——DeepBI 保留 Redash 作为执行基底是务实的，TaskingAI 的 LLM 权重全外部依赖也是合理的（它是"AI 编排基座"而非"AI 计算基座"）。原生不等于无确定性，而等于"确定性服务于概率性核心，而非相反"。

7. **闭环自演进是原生系统的复利来源**：执行结果（成功/失败/人工修正）回流改进提示/检索/策略/模型，且有版本可追溯。无闭环则 AI 调用一次性、流程静态——这是"无状态单轮调用"反模式的根治之道，也是原生产品"越用越贴合用户"飞轮的引擎。
