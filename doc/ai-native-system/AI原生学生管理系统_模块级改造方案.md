# 学生管理系统 AI 原生改造方案：模块级标准与实现

## 一、背景与目标：从"AI 赋能"到"AI 原生"

传统学生管理系统是一套事务型 CRUD 软件的集合——档案录入、成绩登记、考勤点名、排课调课、选课抢课、评教问卷、通知广播、报表导出、缴费审批。它的本质是"记录已发生的事"：分数、出勤、异动、缴费，都被当作需要持久化的静态事实存进数据库，再按固定模板取出来给人看。教师最大的时间黑洞是"录入与统计"，管理者最大的痛点是"看数据却不懂业务"，学生最大的感受是"被排名与被催缴"。

"AI 赋能"的做法是在这套 CRUD 之上外挂一个聊天框或一个"智能助手"按钮：录入完分数后点一下生成评语、查完成绩后问一下"为什么下滑"。这是典型的 bolt-on——核心流程照常运转，AI 只是多了个可选的便利层。它的特征是：移除 AI 系统仍然完整可用，AI 的价值是"锦上添花"而非"承重墙"。

"AI 原生"的根本不同在于：模型在请求路径最前端，概率性逻辑是核心交付物，移除模型则核心价值归零、无可行回退。在本案例中，这意味着——成绩不再是"录入的分数"而是"由作答事件流经知识追踪模型实时计算出的掌握度概率分布"；排课不再是"手工填表"而是"自然语言意图经 LLM 解析为约束、再由求解器与偏好学习共同产出"；预警不再是"缺勤≥3 次的红黄灯"而是"多源融合的未来 4 周挂科/辍学概率预测+因果归因"。

本方案覆盖 10 个模块：学生档案与学籍管理、成绩与学业评价管理、考勤与行为管理、排课与课表管理、选课与培养方案管理、教师与教学评价管理（评教）、通知与家校沟通、学业预警与干预、报表与决策分析、缴费与事务办理。每个模块都经过设计+对抗核验两轮：先给 AI 原生改造设计，再用"今晚移除所有 AI 调用"的移除测试逐流程推演，判定是 truly_native（真原生）、赋能伪装（AI 织入叙事但核心 CRUD 仍可独立存活）、还是 bolt_on（可拔插外挂）。核验暴露的最大反模式不是单模块缺陷，而是"范围借用"——6 个模块都把"风险预测/EWS"当作自己的心脏。因此整体架构的首要决策是把被重复宣称的"心脏"提升为共享内核，让模块从"各建孤岛"变为"订阅内核 + 提供垂直能力"。

## 二、AI 原生的根本判据（应用到此案例）

AI 原生有三条根本判据，缺一不可：

**判据一：移除测试（最硬的标准）。** 今晚移除所有 AI 调用后，模块是否崩溃？这里"崩溃"不是指"退化为旧版本"，而是指"核心价值交付路径归零、无可行的确定性回退"。判定细则必须严格：若移除后模块退化为"一个仍能独立售卖部署、功能完整的传统系统"，则属赋能伪装；若退化为"无智能内容填充的空壳"，则属真原生。

**判据二：概率性逻辑为核心。** 核心交付物必须是模型推理产生的概率性输出（掌握度概率、风险概率、置信度判定），而非确定性规则分支。若决策层是"风险分阈值→分层→工具"的查表，则 ML 退化为上游特征抽取，决策层是确定性 if-else，不满足此判据。

**判据三：第一性原理。** 从原始需求出发而非从惯例出发。学生管理的原始需求是什么？不是"录入分数"，而是"理解每个学生的学业状态并据此行动"；不是"群发通知"，而是"让家长在正确时机获得关于自己孩子的可行动信息"；不是"排一张无冲突课表"，而是"自适应地满足师生偏好并随变更演化"。从这些原始需求出发，AI 原生改造才能触及真正的价值心脏。

**"如果还是赋能会怎样"的对照：** 若本系统停留在赋能层，则教师仍逐条录入分数、排名仍持久化存在、异动仍靠跑 5 个部门、预警仍等缺勤≥3 次才触发（干预成功率从 80% 跌到 20%）、家长仍被淹没在模板通知里、评教仍是期末打分问卷。AI 只是"多了个智能助手"，飞轮不转、合规靠人工把关、headcount 仍随学生数线性增长。这正是传统学生管理系统数十年未解的痛点，也是赋能伪装承诺解决却无法解决的根本原因。

## 三、整体 AI 原生参考架构

### 3.1 七层分层架构

自下而上七层。关键设计：**学生长期记忆 + 学业状态语义层 + 共享风险融合引擎** 是横切三层的"统一 AI 内核"，10 个模块不再各自重建，而是订阅内核并贡献垂直信号。

| 层 | 组成 | 关键决策 |
|---|---|---|
| **基础设施层** | 云-边-端三层；模型 API 网关（多 provider 路由+语义缓存+故障转移+成本上限）；异步队列（Temporal/Inngest 承载跨系统联动+限流重试）；对象存储（材料 artifact，per-tenant 命名空间隔离） | 跨系统联动（宿舍/校园卡/选课/财务/图书馆）由确定性工作流引擎承载，诚实标注为"数字化非 AI" |
| **模型层** | LLM 路由（按复杂度+风险双维度，非纯成本：高端 Sonnet 10%/中端 Haiku 30%/廉价 Qwen2.5-7B 60%，合规关键决策强制走高端）；概率性 ML（DKT/BKT 知识追踪、双层 LSTM 时序、XGBoost 分层、自托管 BERT 自杀意念检测边缘部署数据不出校）；OCR/IDP（PaddleOCR+MinerU 抽取为结构化数据隔离注入面）；harness governs | 权重按 cohort 学习校准非手调 0.7/0.3；输出校准概率+置信区间+SHAP；风险感知路由——成本服从风险 |
| **数据与知识层（统一内核主体）** | 学生长期记忆（单一学生状态真源，各模块读写而非各自存本地副本）；双知识图谱（掌握度图谱+培养方案图谱，共享图引擎多图实例）；向量库（pgvector 起步→Milvus，per-tenant 命名空间）；语义层数仓（StarRocks/Doris，text-to-SQL 防幻觉层）；事件流（Kafka+Flink 四源采集管道）；专有数据壁垒 | 消除孤岛的关键层；专有壁垒=题-知识点映射+错因库+历史异动结局+评审决策库+干预效果库 |
| **Agent 编排层（内核执行体）** | 共享 agent 运行时（LangGraph/自研 harness 状态机，通用 agentic loop 感知→规划→工具→反思）；路由层（最高杠杆组件：意图路由+复杂度路由+copilot/autopilot/out-of-loop 三级分级）；**共享风险融合引擎（消除范围借用的核心决策）**；模块即 agent | 风险融合引擎宿主于"学业预警与干预"模块，其他模块订阅消费而非重建 |
| **工具层（ACI 工程化）** | 检索类（retrieve_student_profile 最小高信号 token 集、rag_search、知识图谱查询、retrieve_similar_case）；计算类（CP-SAT/GA 求解器诚实标注为非 AI 工具、run_forecast、explain_attribution）；IDP 类；系统联动类（API/RPA）；生成类；合规类（check_consent 分层同意、guardrails_check、entitlement_validate 服务端二次校验防 LLM06）；升级类（escalate_to_human+临床复核） | schema 化+服务端二次权限校验 |
| **应用层（10 模块）** | 全部采用生成式 UI+copilot/autopilot/out-of-loop 分级+可纠错可撤销 | 各模块承重方式与回退策略见第 5 章 |
| **治理与可观测层** | evals（CI/CD 部署硬门禁非报告型：分层 golden 集+对抗样本+双轨 gate 核心ML指标+LLM-as-judge+公平性审计）；trace（Langfuse+provenance 可复现）；安全 guardrails；合规引擎；成本护栏（六行成本分解） | 自杀意念召回≥0.95 硬下限；偏差审计 AI Act 附件 III 义务 |

### 3.2 统一 AI 内核

**统一 AI 内核 = 以学生为中心的长期记忆 + 学业状态语义层 + 多 agent 编排运行时 + 共享风险融合引擎 + 共享飞轮。**

**以学生为中心的长期记忆**：单一学生状态真源，贯穿入学→在校→离校→归档/删除全生命周期。包含基础画像、掌握度状态向量（BKT/DKT 概率分布）、行为风险轨迹、校园生活信号、受限文本信号（NLP 情绪标记仅 embedding/标记不留原文）、干预历史与结局。各模块读写同一记忆而非各自存本地副本，离校按 PIPL 第47条自动触发删除/归档。

**学业状态语义层**：数据模型倒置——唯一持久化核心=作答事件流+学生知识状态向量；分数/排名/等级/成绩单全部成为按需生成的派生投影，不存在持久化 CRUD 副本。移除 AI 即无法生成任何学业输出。双知识图谱共享图引擎分别实例化。

**多 agent 编排运行时**：共享 LangGraph/harness 状态机，模块即 agent/skill。路由层为最高杠杆，动态选模型与检索范式（非硬编码 60/30/10 配比，按 eval+成本数据持续调优）。跨 agent 协作通过共享学生长期记忆与事件流协同而非点对点集成。

**共享风险融合引擎——消除范围借用**：核验发现学籍、成绩、考勤、预警、沟通、报表 6 个模块都把"风险预测/EWS"当自己的心脏，这是"范围借用"反模式。架构决策：风险融合引擎作为平台能力宿主于"学业预警与干预"模块，输出多源融合风险分（权重按 cohort 学习校准非手调）+ LLM 主导因果归因（SHAP 特征作证据输入，单一真源）+ MTSS 分层。其他模块订阅消费。LLM 承重位前移——根因归因由 LLM 主导而非"传统 ML 预测+LLM 外周生成"两层割裂。

**共享飞轮**：统一回流所有模块交互信号到学生长期记忆+专有数据壁垒。飞轮因果化——干预效果用反事实/倾向分匹配/ITS 归因，仅沉淀通过因果检验的策略-学生类型映射。飞轮合规化——只在家长单独同意+目的限定+最小化的数据上运行，未成年人数据默认排除训练或脱敏后入飞轮。

### 3.3 模块协同关系表

| 模块 | 主要落位层 | 内核消费关系 | 内核贡献关系 |
|---|---|---|---|
| 学生档案与学籍（修正：流失预测+留存编排） | 应用层（异动办理 agent） | 订阅共享风险引擎+学生长期记忆 | 写入异动结局→飞轮；剥离 EWS 到共享引擎 |
| 成绩与学业评价（修正：掌握度引擎） | 应用层+知识层（掌握度图谱） | 消费共享预测 | **写入掌握度向量→学生长期记忆核心组成** |
| 考勤与行为 | 应用层+边缘采集 | 订阅共享风险引擎（行为风险） | 写入行为信号→学生长期记忆；边缘 CV 采集 |
| 排课与课表（修正：LLM 承重） | 应用层（排课 agent）+工具（求解器） | 消费选课预测 | 写入排课偏好→飞轮 |
| 选课与培养方案（修正：图谱唯一模型） | 应用层+知识层（培养方案图谱） | 消费掌握度+毕业预测 | 写入选课效果→飞轮 |
| 教师与教学评价 | 应用层（评教诊断 agent） | 消费学生学业 delta+掌握度 | 写入教学法知识库（经 curation gate） |
| 通知与家校沟通 | 应用层（沟通 agent） | 订阅共享风险引擎（主动触达）+学生长期记忆 | 写入沟通反馈→飞轮 |
| 学业预警与干预 | 应用层（EWS 守护 agent） | **宿主共享风险融合引擎** | 写入干预效果→飞轮（因果归因） |
| 报表与决策分析 | 应用层（分析 agent）+语义层数仓 | 消费所有模块数据+共享风险引擎洞察 | 写入 query-insight-action 三元组→飞轮 |
| 缴费与事务办理 | 应用层（缴费管家/事务 agent） | 消费学生画像+欠费辍学风险 | 写入评审决策库→飞轮+专有壁垒 |

**协同三机制**：共享 agent 运行时（复用工具集/guardrails/路由/trace）；共享知识库与图谱（多模块读写消除孤岛）；共享数据飞轮（统一回流到学生长期记忆+专有壁垒）。

## 四、12 维标准在本系统的落地映射

| 维度 | 本系统怎么做 | 落在哪个模块/层 |
|---|---|---|
| **1. 架构标准**（Intelligence-First/harness governs/概率核心） | 模型在请求路径最前端（学生一句话/每次作答/每条家长消息都过 LLM 或 ML 推理）；harness 治理工具权限白名单+模型路由+成本上限；概率性逻辑为核心（掌握度=概率、风险分=校准概率+置信区间、资格判定=置信度非布尔）；诚实分层——确定性部分（求解器/工作流引擎/支付）标注为非 AI | 全部模块/模型层+Agent 编排层 |
| **2. 数据与知识标准** | 统一学生长期记忆+双知识图谱+语义层数仓；retrieve_student_profile 最小高信号 token；检索按复杂度动态选（规则/向量/RAG/求解器/时序模型）；专有壁垒=题-知识点映射+错因库+历史异动结局+评审决策库+干预效果库 | 数据与知识层（共享）+成绩/选课/学籍/缴费模块 |
| **3. Agent 系统标准** | 共享 agent 运行时通用 loop；简洁性优先（简单事务不进 agent 直批、SQL 能解不用 LLM）；ACI 工具 schema 化+服务端二次校验；规划模式按任务选 | Agent 编排层（共享）+全部模块 |
| **4. 产品设计标准** | 生成式 UI 按意图动态组装（字段级动态生成非聊天框跳表单）；三级分级（在读证明 autopilot/沟通诊断 copilot/危机退学 out-of-loop）；可纠错可撤销；结果导向（交付异动办结/诊断报告/缴费方案而非工具） | 应用层（全部模块） |
| **5. 业务流程标准** | AI 在价值交付关键路径（异动办结/风险挽留/掌握度诊断/决策洞察）；人机协同分级（材料归档 out-of-loop/沟通休学 on-the-loop/退学学位奖助 in-the-loop）；headcount decoupling | 学籍/成绩/排课/选课/沟通/缴费/报表模块 |
| **6. 评估与可观测性标准** | 分层 golden 集+对抗样本+双轨 gate（核心 ML AUC/RMSE+LLM-as-judge）+公平性审计+自杀意念召回≥0.95 硬下限；Langfuse trace+provenance 可复现；成熟度 foundational→conversational→agentic→multi-agent | 治理与可观测层（共享）+全部模块 |
| **7. 成本与经济模型标准** | 六行成本分解（embedding/LLM 推理/检索/eval/observability/guardrails 均单列可观测）；多模型路由按复杂度+风险双维度；语义缓存降本 30-50%；Opus 战略路由预算护栏；诚实拆分"数字化收益 vs AI 收益" | 模型层+治理层+缴费/排课模块 |
| **8. 安全与对齐标准** | LLM06 过度代理：高风险动作服务端二次权限校验+人工 gate；LLM03 供应链：第三方求解器/OCR/支付供应商 SBOM+准入审计；prompt injection：材料经 IDP 抽取隔离注入面+学生文本输入消毒；guardrails：Moderation 双向+文本蕴含防幻觉+IDK 兜底；NIST 对抗 ML：公平性/偏差审计 | 治理层（共享）+学籍/缴费/评教/沟通模块 |
| **9. 多模态标准** | 材料/课堂转写/作业图片外置为 artifact 不入 LLM 上下文；工具按意图动态挂载；稳定前缀缓存（系统提示/校规/MTSS 规则/证明模板） | 基础设施层+学籍/评教/缴费模块 |
| **10. 边缘与端侧标准** | 边缘 CV（YOLO/ArcFace 原始视频本地丢弃）+自托管 BERT（自杀意念数据不出校）+本地学籍数据网关；云端重推理+重训；端侧轻量意图分类；路由层按风险等级+数据敏感度选推理路径 | 基础设施层+考勤/学业预警模块 |
| **11. 合规标准** | PIPL Art31（<14 岁家长单独同意+PIA+专门规则）；GDPR Art22（重大权益人工复核+申诉）；AI Act 附件 III（教育高风险：数据治理/人工监督/日志/FRIA/偏差缓解）；境内存储 Art38/40；删除权 Art47；可解释（SHAP/LIME 归因+诊断引证据页码）；**自杀意念检测全自建 BERT 边缘部署解决"不出校 vs Moderation 云端"合规矛盾** | 治理层（共享）+全部模块 |
| **12. 组织与团队标准** | 水平工作（学生长期记忆/共享风险引擎/agent 运行时/知识图谱/evals/guardrails 基建）归平台团队；垂直工作（学籍异动业务规则/培养方案/奖助评审/各校干预工作流）归产品团队 | 组织设计横跨全部层 |

## 五、模块级改造方案

### 5.1 学生档案与学籍管理

**传统形态与痛点**：学生基本信息 CRUD、学籍异动（休学/转学/退学/复学/降级/跳级）多级审批、异动跨部门手工联动（人工通知宿舍退宿、校园卡停用、选课退课、学费结算、图书馆权限收回）、档案归档（纸质+电子双套人工录入）、报表导出、证明开具人工跑盖章。最痛痛点是异动办理是"跑 5 个部门"的纸面马拉松，状态变更后下游常不同步——学生已退学但宿舍未退、校园卡未停；异动时机识别严重滞后，错过挽留黄金期（EWS 数据显示 Yellow 区 3-5 次缺勤干预成功率 80%，Red 区 10+ 次仅 20%）；未成年人数据访问无最小化、无审计。

**AI 原生改造设计**：

- **core_ai_capability**：学生动态画像引擎（多模态信号→学籍风险预测）+ 学籍异动智能体编排（感知→规划→工具→反思办理全流程）+ 档案语义化维护。声明价值交付关键路径的心脏是"异动办理智能体编排"——移除它异动办理退回多部门纸面流程。
- **redesign**：异动办理重构为 agent 工作流而非表单审批链——学生/家长用自然语言发起"我想休学一年"→agent 感知意图并聚合该生最小高信号画像→规划子任务序列（材料核验→学籍状态校验→跨系统联动清单→审批流→档案归档→通知）→逐工具执行→反思校验。材料拍照上传，OCR 自动提取结构化字段，LLM 核验材料完整性与校规符合性（缺项自动追问而非退回重填）。跨系统联动由 agent 自动调用 API 完成，仅例外转人工，进度实时可视化。学籍风险前置识别——多模态信号用集成模型（随机森林+双层 LSTM 时序）算综合风险分，三色分级对应 MTSS 干预层级；agent 每日 8AM 前推送"晨间异动风险简报"。退学等重大异动，agent 生成挽留方案建议，counselor on-the-loop 审核。
- **用户体验**：学生/家长不再填一堆表，自然语言发起异动→系统生成异动方案预览卡（需备材料+预估办理时长+对学业/住宿/选课影响+合规提示）→按提示拍照上传→实时看办理进度卡；<14 岁学生异动触发家长单独同意。班主任收到晨间异动风险简报（本周需关注学生+信号+建议谈话方向）；审批时看到学生综合画像+风险分+SHAP 可解释依据。教务管理员异动办理从"跑 5 个部门"变成"看一张进度卡"。证明开具在线秒开（模板稳定前缀缓存+电子签章）。
- **agent 循环**：异动办理为完整 agentic loop——感知（解析异动意图+主动感知画像风险信号触发主动异动建议）；规划（LLM 按意图+画像+校规+合规约束分解子任务，简单异动单次规划、复杂多部门多步规划+反思）；工具（ACI 工具集——OCR 材料核验/学籍查询变更/宿舍/校园卡/选课/财务/档案归档/通知/合规校验，每工具服务端二次权限校验防过度代理）；反思（每步结果校验，材料缺失追问、状态冲突回退规划、审批被拒记录原因）。
- **数据飞轮**：每次异动办理（申请材料/审批决策/处理时长/跨系统联动结果/最终留校或流失结局）→沉淀标注数据；风险预测模型用历史异动结局训练；counselor 对挽留方案的采纳/修改反馈→形成"报告→训练→提升"闭环。每 30 天用新数据重训（仅验证准确率提升才更新）。

**移除测试结果（设计自判）**：抽掉 AI——异动办理退回多部门纸质跑流程、跨系统联动退回人工逐个系统操作、学籍风险识别退回事后人工发现（错过 80% 挽留黄金期，只剩 20%）、档案退回人工录入+翻柜检索、材料核验退回肉眼辨真伪、证明开具退回人工跑盖章。模块退化为 CRUD 学籍表+异动登记表+档案柜的空壳。设计自判 native。

**应用的标准**：架构标准（模型在请求路径上、harness 治理、风险评分是概率性逻辑）；数据与知识标准（飞轮、上下文工程按异动类型动态裁剪、检索范式按复杂度动态选、专有数据壁垒=历史异动结局+校规知识库）；Agent 系统标准（异动办理为完整 agentic loop、ACI 工具接口工程化）；产品设计标准（生成式 UI、copilot/autopilot 分级、可纠错可撤销、结果导向）；业务流程标准（异动办理全流程在 AI 价值交付关键路径、人机协同分级、headcount decoupling）；评估与可观测性标准；成本与经济模型标准；安全与对齐标准（LLM06 过度代理服务端二次校验、LLM03 供应链、材料核验 LLM 输出不直接落库、学生材料文本 prompt injection 输入消毒）；多模态标准（材料外置 artifact、证明模板稳定前缀缓存）；边缘与端侧标准（端侧自助查询 Qwen2.5-7B、边侧本地学籍数据网关、云侧复杂异动规划）；合规标准（PIPL 第31条/GDPR 第22条/AI Act 附件 III、SHAP/LIME 可解释）；组织与团队标准。

**对抗核验结论：赋能伪装。** 这是最重要的诚实呈现。

逐流程推演"今晚移除所有 AI 调用"：(1) 学生基本信息 CRUD → PostgreSQL 直查直改，零影响。(2) 报表导出（花名册/异动统计/在校生/毕业生）→ SQL 聚合，零 AI 依赖。(3) 证明开具 → 退回人工模板+盖章，仍可办结仅慢；"稳定前缀缓存"是优化非必需。(4) 档案归档与检索 → 退回人工录入+关键字/SQL 检索，仍可用。(5) 学籍异动办理 → agent 消失后：NL 入口退回表单、LLM 材料核验退回肉眼、风险感知规划退回固定审批链。但异动仍然能办结：表单提交→逐级签字→人工通知 5 部门→归档。学校数十年无 AI 照常运转。关键：设计把"跨系统联动（宿舍/校园卡/选课/财务/图书馆同步）"算作 agent 价值，但该能力由确定性工作流引擎（Temporal/Camunda）+ API 集成即可实现，零 AI。agent 真正 AI 贡献仅 NL 解析+材料模糊核验+边缘 case 判断，对 90% 标准异动是便利层而非承载层。(6) 风险预测/晨间简报/挽留建议 → 这部分确实坍塌，无确定性替代能"提前 3 周预测谁会退学"。这是唯一真正原生能力。

结论：模块实际声明范围（档案与学籍管理）下，CRUD/报表/证明/归档/异动办理全部有可用非 AI 回退，移除 AI 后模块退化为"还能用的档案+异动登记系统"——设计自承"退化为 CRUD 学籍表+异动登记表+档案柜的空壳"。"还能用"恰是赋能伪装签名，非真原生。唯一原生能力（风险预测）属 EWS/学业预警范畴，是从另一模块范围借用来撑场面。且最痛的"跑 5 个部门"实为工作流自动化收益，被错误归因给 AI。

**反模式风险**：(1) AI 洗（HIGH）——把"跨系统联动/跑 5 个部门"的解决归给 agent，实则工作流引擎+API 即可零 AI 实现。(2) 范围借用（HIGH）——风险预测/EWS 是真正原生能力，但属独立学业预警模块范畴，被折叠进来抬高原生性。(3) Agent 戏剧化（MEDIUM）——标准异动子任务序列由校规固定，LLM"规划"对 90% 标准用例是表演，状态机足矣。(4) 固定权重伪概率（MEDIUM）——风险分=0.7×学业+0.3×行为+固定阈值 0.6/0.75/0.9 接近"调参 if-else"而非学习型决策器。(5) NL 入口套壳风险（MEDIUM）——"我想休学一年"若下游映射到固定表单则沦为聊天框套壳。(6) Flywheel 部分空转（LOW-MEDIUM）——agent 编排"飞轮"实为 LLM-as-judge 监控，agent 不按结局微调模型，非真飞轮。

**修正后的设计要点**：(1) 重新定位模块主体价值——以"学生流失风险预测+留存编排"为主体输出，档案/异动记录降为喂给预测的数据基础设施。移除模型→主体价值归零而非退回"还能用的档案柜"，才算真移除测试通过。或诚实把 EWS 拆为独立原生子模块、档案学籍部分承认赋能伪装。(2) 异动办理分层并诚实归因——确定性部分（材料清单校验、5 系统同步、固定审批链）由 Temporal/Camunda+API 承载标注"数字化非 AI"；AI 仅接管真正概率性部分（政策模糊/状态冲突/多因素权衡、材料 LLM 核验、风险感知异动时机建议）。量化"AI 不可替代 case 占比"目标 >40%。(3) 预测模型改为决策器非打分器——输出校准概率+置信区间，权重按年级/专业 cohort 学习而非手调 0.7/0.3，SHAP 挂在概率上而非固定阈值带。(4) Agent 飞轮实化——规划 agent 按"决策-结局"对在次优 case 上微调更新模型，而非仅 LLM-as-judge 监控。(5) 端到端可证伪度量——发布"移除 AI 后各核心价值下降 X%"：记录类≈0%、风险预测≈100%，按此分裂如实标注模块原生度，不把数字化收益混入 AI ROI。

**技术实现**：意图路由与材料核验用中端模型（Claude Haiku/GPT-4o-mini 或本地 Qwen2.5-7B 边缘部署）；复杂异动规划与画像摘要用高端（Claude Sonnet/GPT-4o）；学籍风险预测用表格 ML（XGBoost/随机森林 100 棵树最大深度10）+时序（双层 LSTM 64/32 单元，权重按 cohort 学习校准）；OCR 用 PaddleOCR/MinerU。档案语义检索用 Milvus/pgvector（BGE-M3 embedding+bge-reranker，top-20 分段）；校规 RAG 同款检索管线。agent 框架 LangGraph 或自研 harness；异步跨系统联动用 Temporal/Inngest 做队列+限流重试（诚实标注为非 AI 工作流引擎）。数据层 PostgreSQL 主库+pgvector+对象存储（per-tenant 命名空间隔离）+多模态信号采集管道。境内云满足 PIPL 本地化；模型 API 网关多 provider 路由+语义缓存+故障转移；CI/CD 内建 evals 门禁（50-100 条 golden 异动案例）。trace 用 Langfuse。离线 golden 异动案例集；LLM-as-judge 评估异动方案与挽留建议质量；在线监控风险预测 precision/recall+材料核验准确率+异动办理时长（对标缩短 30%）。

### 5.2 成绩与学业评价管理

**传统形态与痛点**：成绩录入（教师逐条录分、Excel 批量导入易错）、查询、统计（均分/标准差/及格率/优秀率/分数段）、排名（班级/年级/加权总分）、等级评价（A/B/C/D 换算）、评语（手写或套模板）、成绩单 PDF 导出、家长端查询。核心痛点：统计排名是"事后尸体解剖"——只描述结果不回答为什么差、差在哪、怎么补；排名制造焦虑且单一维度，总分排名掩盖知识结构差异（同是 80 分，学生 A 丢分在计算、学生 B 丢分在概念，补救路径完全不同）；评价=打标签无诊断深度，评语模板化空洞；知识掌握度近乎空白，分数≠掌握；无预测无预警，挂科发生才知道。

**AI 原生改造设计**：

- **core_ai_capability**：基于知识图谱的技能级知识追踪（skill-level Knowledge Tracing）——把"分数"转化为"知识掌握度概率分布"。声明是模块心脏，移除即退化为分数仓库，同时驱动诊断、补救、预测、评价四件事，知识掌握度这一职责在纯 CRUD 下根本无法存在。
- **redesign**：(1) 录入即诊断——教师在录入分数的同时系统把每道题/每个知识点作答数据（题-知识点映射、用时、错误类型、修正轨迹）映射到知识图谱节点，实时更新学生知识状态向量（BKT/DKT 概率）。(2) 知识掌握度建模——学科拆解为纳米级知识点（松鼠AI 初中数学 10000+ 点范式），每个知识点维护掌握概率 p。(3) 个性化诊断生成——LLM 基于知识状态向量+薄弱点+历史轨迹生成自然语言诊断，遵循 Khanmigo"给提示而非给答案"。(4) 个性化补救路径——知识图谱引擎定位前置依赖与最近发展区 ZPD，由系统而非学生决定下一步补什么。(5) 学业预测——APLSE 式预测模型综合风险=0.7×学业+0.3×行为三色分级。(6) 教师端班级诊断——LLM 生成班级共性薄弱点报告+教学建议（每周省约 37 分钟）。(7) 学业评价重构——从 A/B/C/D 标签→多维学业画像，评语由 LLM 起草、教师可编辑可拒绝。
- **用户体验**：学生不再看冷排名表，而是"本周知识掌握度变化"卡+个性化诊断+下一步该补什么。教师录入分数后自动获得班级诊断与教学建议。家长看到可解释学业画像。管理员看三色分级学业预警看板。
- **agent 循环**：模型成为核心路由+教师侧 agent。感知（采集每次作答的题-知识点-用时-错误类型-修正轨迹→实时更新知识状态向量）。规划（知识图谱引擎定位掌握/未掌握节点，计算前置依赖路径上的 ZPD，规划补救序列；教师侧 agent 感知班级聚合数据→规划干预）。工具调用（诊断报告生成、补救练习集生成、家长沟通起草、班级共性薄弱点看板、分层教学分组推荐、学业预警触发，ACI schema 化）。反思（对比预测与实际下次表现，更新知识追踪参数与图谱权重，校准风险阈值；LLM-as-judge 评估生成诊断/评语质量；教师"接受/编辑/拒绝"反馈进入飞轮）。
- **数据飞轮**：每次作答→题-知识点-作答数据→更新知识状态+知识图谱参数+错因库→更精准个性化诊断/路径→更好学习效果→更多学生使用/更多数据→更优算法。教师对 AI 诊断/评语的反馈训练生成质量。错因库与题-知识点映射表是专有数据壁垒。

**移除测试结果（设计自判）**：抽掉知识追踪+知识图谱+LLM 诊断+预测，系统退化为"分数录入+排名表+等级标签+手写评语"——无法建模知识掌握度、无法诊断薄弱点、无法生成补救路径、无法预测挂科风险。设计自判 native。

**应用的标准**：架构标准（模型在请求路径上、概率性逻辑为核心知识掌握度=概率估计）；数据与知识标准（飞轮、上下文工程传知识状态向量+薄弱点+轨迹不传整张试卷、检索按复杂度、专有数据壁垒题-知识点映射+错因库+知识图谱）；Agent 系统标准（agentic loop、简洁性优先知识追踪用确定性概率模型而非一切皆 LLM）；产品设计标准（意图驱动生成式 UI、copilot vs autopilot 诊断/路径=autopilot 建议评分/留级=human-in-the-loop、可纠错可撤销教师可编辑 AI 评语）；业务流程标准（AI 在价值交付关键路径、人机协同、headcount decoupling）；评估与可观测性标准；成本与经济模型标准；安全与对齐标准（LLM06 AI 不直接改分/留级须人工）；多模态标准（试卷扫描件外置 artifact）；边缘与端侧标准（云-边-端三层）；合规标准（PIPL Art31、GDPR Art22、AI Act 附件 III、SHAP/LIME 可解释）；组织与团队标准。

**对抗核验结论：赋能伪装。** 诚实呈现如下。

严格重做移除测试：模块传统核心流程（按 traditional_functions）= 录入→查询→统计→排名→评语→成绩单导出→家长查询。今晚移除全部 AI（BKT/DKT、知识图谱推理、LLM 诊断/评语、预测模型）后逐项推演：(1) 成绩录入：教师仍可按课程/学期逐条录分、Excel 批量导入——纯 CRUD 持久化，不依赖模型，照常运行；(2) 查询/统计/排名：均分/标准差/及格率/优秀率/分数段、班级与年级排名——纯 SQL 聚合与确定性计算，照常运行；(3) 等级评价：百分制查表换 A/B/C/D，照常运行；(4) 评语管理：退回手写/模板评语，照常运行（AI 起草只是便利层）；(5) 成绩单 PDF 导出/归档：模板渲染，照常运行；(6) 家长端查询：退回裸分与排名数字展示，照常运行。

结论：模块核心流程（学校赖以运转的录分—出榜—出成绩单—家长看分）完全不崩溃，仅退化为传统分数仓库——而这恰是一个功能完整、可独立售卖部署的系统。决定性证据：录入仍以"分数"为持久化主数据、知识追踪作录入"同时"的副作用触发（录入即诊断=CRUD+enrichment），证明 AI 富化传统数据模型而非取代之；模型不在传统核心操作请求路径上，仅在新增能力路径上。真正崩溃的只有诊断/补救/预测/知识掌握度这层"新加能力"——但它们是 augmentation，非模块既有契约核心。设计的自移除测试作用域错误：它测的是"某个新能力是否消失"而非"模块核心流程是否崩溃"。

**反模式风险**：(1) 录入即诊断实为 bolt-on 增强模式——保留"教师逐条录入分数"传统输入与"分数"为持久化主数据，知识追踪作录入副作用触发，AI 富化传统数据模型而非取代之（标准1：模型未在录入核心请求路径上，仅为 side effect）。(2) 传统 CRUD 核心可独立存活——录入/查询/统计/排名/成绩单导出/家长查询移除 AI 后照常运行。(3) 排名未根除仅"不展示给学生"——排名计算仍持久化存在，半吊子改造。(4) 硬编码阈值=AI 当 if-else——综合风险=0.7×学业+0.3×行为、三色 0.6/0.75/0.9 为魔法数字。(5) 评语 LLM 起草属典型 copilot 套壳。(6) 知识掌握度是"新加能力"非模块既有职责——传统系统该职责"近乎空白"，AI 创造新功能而非转化既有核心。(7) 未成年人数据张力未解——10000+ 纳米知识点细粒度掌握度建模=收集更多未成年人敏感心理画像，与"数据最小化"冲突。(8) DKT/LSTM 黑盒与可解释性矛盾。(9) 过度代理权风险（LLM06）——"系统决定学习顺序"对未成年人是重大代理权委托，缺人工 override。(10) eval 门禁偏生成质量轻核心模型——evals 集中于诊断/评语 LLM-as-judge，未将掌握度预测 AUC/RMSE 作为 CI 硬门禁。

**修正后的设计要点**：(1) 数据模型倒置（最关键）——唯一持久化核心数据=作答事件流+学生知识状态向量（概率分布）；分数/排名/等级/成绩单全部成为按需生成的派生投影，不存在持久化 CRUD 副本。如此移除 AI 后无法生成任何学业输出，模型真正在每次核心请求路径上（满足标准1 Intelligence-First）。(2) 录入即建模、废除"教师逐条录入分数"——作答数据从自适应练习/自动批改/题级捕获直接流入，每次记录=知识状态更新，分数不再作为输入只作为投影输出。(3) 评价=生成式画像、废除 A/B/C/D 标签与模板评语——学业画像每次请求实时生成，教师可编辑可拒绝，无模板回退。(4) 排名根除——不存储排名。(5) 家长端仅展示 AI 生成画像与行动建议——AI 不可用时显示"画像生成中"而非分数表。(6) 模块成功指标 AI 原生化——掌握度收敛率、预测 AUC/RMSE、诊断接受率，而非"录分条数"。(7) 评估门禁双轨——掌握度预测 AUC/RMSE held-out+诊断 LLM-as-judge 双 gate 进 CI/CD。(8) 解未成年人数据张力——知识状态向量作为敏感心理画像单独同意+PIA；定义掌握度轨迹留存/删除/模型遗忘（被遗忘权）；细粒度建模与数据最小化通过"按需展开"调和。(9) 学习路径人工 override——系统决定学习顺序须有显式、可审计的教师/家长 override 与日志。

**技术实现**：知识追踪 DKT（LSTM）/BKT，技能级细粒度，参考 Carnegie MATHia ACT-R 认知模型。知识图谱 Neo4j/NebulaGraph，纳米级知识点拆解，节点=知识点、边=前置依赖/包含/关联。预测模型 APLSE 式 LightGBM/随机森林（权重按 cohort 学习校准）。LLM 诊断与评语多模型路由。RAG 题-知识点映射表+课程标准+错因库检索增强，DPR 双编码器。Embedding bge-m3/text-embedding-3-large，pgvector 起步规模大迁 Milvus，per-tenant 命名空间隔离。Agent 框架 LangGraph/自研状态机。数据层 PostgreSQL+Neo4j+pgvector/Milvus+Redis（实时知识状态缓存）。评估 50-100 条 golden 诊断样例+掌握度预测 AUC/RMSE held-out 双 gate+LLM-as-judge，CI/CD 评测门禁。合规境内存储、<14 岁家长同意+PIA、自动化决策人工复核通道、SHAP/LIME 可解释。

### 5.3 考勤与行为管理

**传统形态与痛点**：考勤记录（人工点名/刷卡，按日/周/月生成考勤率统计表与班级出勤排名）、行为记录（手工填写违纪/奖励事件单，累计行为分，学期生成行为档案与品德评语）、异常行为识别（固定阈值规则触发预警如缺勤≥3 次）、报表导出。痛点：考勤采集靠人工点名占用课堂时间或刷卡只能签到无法识别行为模式（周一/周五规律性缺勤、上课注意力涣散）；行为记录主观、滞后、稀疏，只记大事件；异常识别是事后阈值法——等缺勤≥3 次才预警，此时干预成功率已从 80% 跌到 20%，AI 本可比阈值法提前约 3 周识别风险；无闭环，预警名单发出后无干预跟踪；考勤、行为、成绩、校园卡、LMS 数据孤岛无法多模态融合。

**AI 原生改造设计**：

- **core_ai_capability**：多模态行为时序推理引擎（multimodal behavioral risk inference engine）——持续融合考勤模式、校园数字足迹（校园卡消费/门禁/图书馆、LMS 登录节奏含深夜提交）、行为事件、讨论区 NLP 情绪等信号，输出动态风险轨迹+分层干预建议。声明是模块心脏：移除后异常行为识别退化为静态阈值（晚 3 周、干预成功率 20%），考勤/行为采集退回人工录入，模块从行为智能降级为记录本。
- **redesign**：考勤从人工点名/刷卡转为边缘端多模态信号采集——闸机/教室边缘盒子做人脸检测+ArcFace 嵌入（不上传原始视频，只传事件与 embedding artifact），叠加校园卡门禁与 LMS 登录时间戳，自动捕获完整出勤行为模式。行为从手工事件单转为持续行为信号流——校园卡消费规律性、图书馆/食堂/宿舍使用、LMS 学习时长分布与深夜提交、讨论区文本情绪（NLP），事件自动聚类上下文化而非孤立打分。异常识别从阈值法转为 EWS agentic loop——ABC 指标+MTSS 绿/黄/橙/红分层，AI 提前约 3 周识别，干预在 Yellow 区（80% 成功率）而非 Red 区（20%）触发。干预闭环：检测→优先级（每日 8AM 前晨间优先简报：风险学生+识别模式+建议下一步）→行动（AI 起草个性化家长沟通，一键审核发送）→分层建议→跟踪。可解释——每个风险评分附 SHAP/LIME 因子归因。
- **用户体验**：教师每天 8AM 前收到晨间优先简报——风险学生名单、识别出的行为模式、建议下一步、AI 起草好的家长沟通草稿一键审核发送，免去点名。学生看到参与度仪表盘（非惩罚式），温和助推。家长收到个性化有上下文的沟通。辅导员看风险热力图、干预跟踪看板、干预 ROI 分析。所有关键动作可纠错可撤销。
- **agent 循环**：EWS agent（agentic loop）：感知——每日/实时摄入多模态信号，上下文工程抽取最小高信号 token 集。规划——按复杂度路由：简单模式（已知缺勤序列）走规则快路径；细微/新型风险走 LLM 推理+检索学生历史轨迹，规划干预类型与层级。工具调用——retrieve_student_history（Milvus 检索相似历史案例）、compute_risk（LSTM 时序+XGBoost 分类器）、draft_communication（LLM 起草沟通）、schedule_meeting、update_crm；ACI 工程化：工具接口稳定、参数服务端校验、LLM 不可直接授权高风险动作（LLM06）。反思——干预后跟踪出勤/参与度是否改善；LLM-as-judge 评估沟通消息质量；哪些干预对哪类画像有效写入机构知识库；模型更新经 eval 门禁。
- **数据飞轮**：每次考勤/行为事件→多模态信号捕获→更新风险轨迹模型→更早更准预测→更多干预在 Yellow 区成功（80%）而非 Red 区（20%）→更多成功结果→教师信任与使用增加→更多数据捕获→模型更强。叠加：干预结果跟踪→沉淀机构知识库→未来案例检索复用→预测+干预双轮闭环。

**移除测试结果（设计自判）**：考勤退回人工点名/刷卡（仍可记录但失去模式识别），行为退回手工事件单（稀疏孤立），异常识别退回缺勤≥3 次静态阈值——晚 3 周才发现，干预成功率从 80% 跌到 20%；晨间简报、个性化家长沟通、干预跟踪、机构知识库沉淀全部消失。设计自判 native。

**应用的标准**：架构标准（Intelligence-First 风险推理引擎在请求路径上、harness governs、风险分是概率输出而非确定性规则）；数据与知识标准（飞轮、上下文工程最小高信号 token、检索按复杂度、专有数据壁垒多模态行为信号）；Agent 系统标准（agentic loop、简洁性优先简单走规则复杂走 LLM、ACI）；产品设计标准（生成式 UI 晨间简报、copilot vs autopilot 采集 autopilot 干预 copilot、可纠错可撤销、结果导向交付干预结果而非仅告警）；业务流程标准（AI 在关键路径、人机协同、headcount decoupling）；评估与可观测性标准；成本与经济模型标准；安全与对齐标准（LLM06、prompt injection NLP 输入防注入、guardrails 生成家长消息过 Moderation）；合规标准（PIPL 未成年人敏感、AI Act 附件 III 考试行为监测高风险、GDPR Art 22、FERPA、数据境内存储）；多模态标准（视频帧外置 artifact）；边缘与端侧标准（云-边-端三层边缘 CV 首轮推理）；组织与团队标准。

**对抗核验结论：truly_native（条件式）。** 诚实呈现透明度问题。

分两层严格推演。(1) 按"移除所有 AI 调用"全口径（含经典 ML：compute_risk/LSTM/XGBoost、NLP 情绪、LLM agent/沟通/judge/语义缓存）：考勤采集退回人工点名/刷卡（边缘 CV 被移除），行为退回手工事件单，异常识别退回缺勤≥3 次静态阈值——晚约 3 周、干预成功率 20%；晨间生成式简报、个性化家长沟通、干预跟踪、机构知识库、多模态预测全部消失。CRUD/Excel 报表仍能手动跑，但模块声称的身份"行为智能/早期干预"被掏空，退化为传统记录本+阈值告警。→ 核心身份崩溃，通过移除测试。

(2) 但若仅移除 LLM/agentic 层、保留经典 ML（LSTM/XGBoost/NLP 情绪）：边缘 CV 考勤采集照常、风险预测仍由 LSTM/XGBoost 产出（提前约 3 周识别仍成立），只丢失晨间简报/起草沟通/LLM-as-judge/案例检索等 co-pilot 能力。这揭示一个透明度问题：设计的"心脏"实为经典 ML 推理引擎，LLM agent 是编排/沟通 co-pilot 而非 load-bearing 核心；设计将二者打包宣称"移除即崩"略有过度宣称。

结论：在全 AI 移除口径下核心身份崩溃、判 native；但须修正措辞，承认 LLM 层是 co-pilot、ML 推理才是 load-bearing，避免以 agentic 外衣包装经典 ML EWS 的赋能伪装倾向。

**反模式风险**：(1) AI 洗/过度宣称——"多模态融合无法做"——特征融合本身是数据工程，真正需要 AI 的是"从融合特征做概率预测"，措辞把数据工程算作 AI 能力。(2) AI 当 if-else——综合风险=0.7×学业+0.3×行为是手调线性公式，若直接驱动分层与干预决策则决策层是确定性规则而非概率 AI。(3) LLM 层非 load-bearing 却被包装为心脏。(4) 忽视 fairness/bias eval——对未成年人做风险分层却无显式偏见/公平性审计。(5) 数据飞轮 vs PIPL 最小化冲突未调和。(6) NLP 抓取未成年人讨论区情绪的合规与伦理风险——将青少年正常情绪波动病理化。(7) 外部指标当事实——"提前3周/80%成功率"引自外部案例未在本系统自有数据上在线 eval 验证。(8) autopilot 采集与 AI Act 高风险监督张力。

**修正后的设计要点**：(1) 透明化分层——明确 load-bearing 核心=多模态 ML 推理引擎（LSTM/XGBoost/NLP 情绪），LLM agent=编排与沟通 co-pilot；删除"多模态融合需 AI"的过度宣称，改为"从融合特征做概率预测需 ML 模型"。(2) 决策可纠错——风险分层影响未成年人重大权益，须强制人工复核（GDPR Art 22+AI Act 附件 III）；0.7/0.3 仅作简单快路径启发式，复杂路径走 ML+LLM 并附人工复核；分层阈值不得全自动施加干预。(3) 补 fairness eval——新增人口学公平性/差异影响审计作为 eval 门禁，与 golden 样本、LLM-as-judge 并列；偏见超标阻断部署。(4) 飞轮合规化——飞轮只在家长单独同意+目的限定+最小化的未成年人数据上运行，定义留存期与目的边界，<14 岁走 PIPL Art 31 敏感信息专门规则+PIA。(5) NLP 情绪治理——讨论区文本纳入敏感信息、须单独同意与目的限定、设公平性 guardrail 避免病理化正常情绪、情绪驱动 flag 须人工复核。(6) 自验指标——"提前3周/80%"改为本系统在线 eval/shadow-mode A/B 目标而非引用外部数字当既成事实；上线前 shadow-mode 跑通。(7) 采集监督——行为监测高风险场景采集不全 autopilot，设人工监督与日志（AI Act 附件 III）。

**技术实现**：考勤采集边缘端 YOLO/MediaPipe 人脸检测+ArcFace embedding（国产化部署于闸机/教室边缘盒子，只传事件与 embedding，原始视频本地丢弃）；非 CV 路径用校园卡门禁+LMS 时间戳。风险预测双层 LSTM（64/32 单元）时序+XGBoost/Random Forest（100 棵深度 10）分层分类；K-means 聚类行为模式。权重按 cohort 学习校准。SHAP/LIME 归因。NLP/LLM 国产小模型（GLM-4-Flash/Qwen2.5-7B 境内部署）做情绪抽取、家长沟通起草、LLM-as-judge；多模型路由；语义缓存。检索 Milvus 向量库存学生历史轨迹与干预案例库，per-tenant 命名空间隔离；DPR 式检索。Agent 框架 LangGraph 编排。数据层 PostgreSQL（行级安全 per-school 隔离）+Kafka+Flink 实时算特征+MLflow 模型版本+eval 门禁。基础设施阿里云/腾讯云境内存储；边缘推理盒；Inngest 式异步队列。评估 50-100 golden 风险识别样例+LLM-as-judge 沟通质量+fairness/偏见审计三重 eval 门禁+在线 shadow-mode 自验提前期与成功率。

### 5.4 排课与课表管理

**传统形态与痛点**：手工或 Excel 排课（教师录入可用时间表，管理员手动分配课程-教师-教室-时段四元组）、调课（表单申请，管理员手动查找替代）、教室/教师/班级资源调度（资源台账 CRUD，冲突靠人工或简单规则校验）、课表查询、走班制组合（3+1+2 选课结果手动组合，跨班冲突人工核对）、冲突检测（仅硬冲突规则校验，软约束无法表达）、报表。痛点：排课是 NP-hard 组合优化，传统手工/Excel 在大校耗时长、冲突频发、易错；软约束（教师偏好"数学排上午"、学生最优认知时段、教室设备匹配、实验室检修）无法表达；临变更响应慢，调课牵一发动全身；走班制组合爆炸；排课与选课、考勤、财务割裂；教务老师的自然语言需求须翻译成系统参数；资源利用率低；无学习闭环。SCIRP 论文实证：纯 GA 自动求解能力不足，实践退化为"半自动"。

**AI 原生改造设计**：

- **core_ai_capability**：LLM 自然语言偏好解析+任务编排（LLM-as-Orchestrator）与约束优化求解器（CP-SAT/GA）的融合是模块心脏。LLM 负责"感知→偏好解析→任务分解→编排"，CP-SAT/GA 负责"约束求解"。移除 LLM→教务需求无法自然语言化、退回手工参数；移除求解器→大规模组合优化无法完成。叠加预测性预分配（ML 预测选课需求）与动态调课 agent 形成闭环。
- **redesign**：(1) 自然语言需求录入——教务老师说"高三数学尽量排上午前两节、王老师周二下午有教研不排、物理实验室周三检修不可用"，LLM 解析为结构化约束（硬约束：实验室周三不可用；软约束：数学偏好上午权重0.8；王老师周二下午排除）。(2) 两阶段求解——第一阶段 LLM 偏好解析生成约束集→第二阶段 CP-SAT（中小规模精确求解）或 GA（大规模走班制，repair 机制维持可行性）求解，硬约束零冲突、软约束加权最优。(3) 预测性资源预分配——基于历史选课数据用 LSTM/随机森林预测下学期选课需求趋势。(4) 动态自适应调课——教师请假/教室占用触发时，agent 自动重排受影响子图（非全量重排），给出多个备选方案+影响评估。(5) LLM-as-Orchestrator 对接异构系统——老旧教务系统无 API 时用屏幕语义理解/RPA 打通。(6) 走班制智能组合——3+1+2 选课结果自动生成走班组合，最小化跨班冲突。(7) 数据中台联动——排课结果通过 RESTful API/Webhook 推送考勤系统核销课时，财务按核销自动结算。(8) 生成式 UI——按意图动态组装"王老师下周课表""高三2班实验室缺口""本周调课影响范围"。
- **用户体验**：教务管理员不再手工拖拽排课，用自然语言描述需求，系统生成候选课表+冲突预警+资源利用率+软约束满足度，在生成式 UI 上审阅/微调（human-on-the-loop），调课方案可一键采纳/撤销/回退。教师自然语言提偏好自动并入软约束。学生/家长课表自动推送，调课影响主动通知。
- **agent 循环**：感知（监听选课数据变更、教师请假单、教室占用/检修、考勤核销回流、历史排课偏好）→规划（LLM 把教务自然语言需求解析为结构化约束集；按规模分层选求解器；分解为子任务：预分配→主排课→调课→核销）→工具调用（调用 CP-SAT/GA 求解器 OR-Tools、冲突检测 API、教室/教师可用性查询、课表发布 API、考勤/财务联动 Webhook，ACI 工程化）→反思（求解后评估软约束满足度/资源利用率/教师负载均衡，不达标则调整权重重求解；调课后回溯采纳/拒绝形成偏好学习）。
- **数据飞轮**：每次排课/调课记录：教务需求(NL)→解析的约束→求解方案→管理员采纳/修改/拒绝→实际执行→考勤核销→教师负载。闭环：偏好学习（哪些软约束被频繁要求/拒绝→自动调整权重）、选课预测、调课方案采纳率、资源利用率回溯。每 30 天用新数据重训预测模型。

**移除测试结果（设计自判）**：抽掉 AI 后自然语言需求无法解析退回手工参数配置表单；大规模组合优化无法自动求解退回 Excel 手工排课；临变更无自动重排方案；无预测性预分配；无学习闭环；无生成式 UI。剩下的只是课表 CRUD+硬冲突规则校验+利用率报表=空壳。设计自判 native。**诚实补充**：若只移除 LLM 保留 CP-SAT/GA 求解器，退化为传统优化排课软件（仍能排课但无自然语言录入/预测预分配/动态调课 agent/学习闭环），属"赋能级"。

**应用的标准**：架构标准（模型在请求路径上 LLM 偏好解析+任务编排、harness governs LLM 不直接求解而是调用求解器工具、概率性逻辑为核心）；数据与知识标准（飞轮偏好学习/选课预测、上下文工程最小高信号约束集、检索按规模分层、专有数据壁垒本校排课偏好权重）；Agent 系统标准（agentic loop、简洁性优先 LLM 只做编排不做求解、ACI、规划模式按任务选）；产品设计标准（意图驱动生成式 UI、copilot vs autopilot 全量排课 autopilot+审阅调课 copilot、可纠错可撤销、结果导向）；业务流程标准（AI 在关键路径、人机协同 human-on-the-loop、headcount decoupling）；评估与可观测性标准；成本与经济模型标准；安全与对齐标准（LLM06、LLM03、prompt injection 教务 NL 输入过滤、guardrails 课表变更须人工审阅）；多模态标准（课表导出外置 artifact）；边缘与端侧标准（云-边-端三层、路由层最高杠杆）；合规标准（PIPL 第31条 K-9 排课数据须家长同意、GDPR Art 22 类比、未成年人数据最小化、可解释性）；组织与团队标准。

**对抗核验结论：赋能伪装。** 诚实呈现如下。

移除测试须严格按"移除所有 AI 调用"执行，而 AI=模型驱动的概率性系统（LLM/ML/飞轮学习），不含 CP-SAT/GA（经典运筹学确定性/随机搜索算法，无数据学习、无概率推理，传统排课软件早已使用，设计自承"部分学校用 GA/CSP 排课软件但参数须手工配置"）。逐项推演（移除 LLM+ML+飞轮，保留非 AI 求解器）：①排课核心（生成无冲突课表）——仍可用 CP-SAT/GA+手工配置硬/软约束完成，这正是传统排课软件现状，核心交付不崩溃；②冲突检测——硬约束校验本就是确定性规则，照常；③课表查询/报表——静态查询+确定性统计，照常；④调课——退回人工查找替代（降级但可运行）；⑤走班制——GA 可解，退回手工配置参数。故核心流程不崩溃，仅退化为传统排课软件。

决定性证据是设计自身的诚实补充："若只移除 LLM 保留 CP-SAT/GA 求解器，退化为传统优化排课软件（仍能排课）……属'赋能级'"。"赋能级"即赋能伪装。设计的"native"自判与之矛盾：它把非 AI 求解器计入"AI 整体"，再称"移除任一关键组件即退化"，但求解器不是 AI，移除所有 AI 后求解器仍在、核心排课仍跑通。

结论：核心价值交付（解 NP-hard 排课）由确定性求解器承载，概率性逻辑（LLM/ML/飞轮）是输入翻译层+预测增强+UX 包装，非负载核心，违反维度1"概率性逻辑为核心"。设计把"模型在请求路径上"做成"模型在请求路径的入口做翻译然后交给确定性求解器"，这是成熟工程但非 native。判赋能伪装（非 bolt_on，因 AI 已织入 agent loop/飞轮/生成式 UI 多层，非可拆卸 addon；非 truly_native，因移除 AI 后核心排课仍可交付）。

**反模式风险**：(1) AI 套壳聊天框——NL 输入本质是套在确定性求解器外的聊天框，若 LLM 仅把"数学排上午"映射为 weight=0.8 则是 glorified 表单。(2) LLM 当 if-else 路由器——"按规模分层选求解器"可能退化为确定性 if-else 分支。(3) mega-prompt 脆性。(4) 忽视在线 eval/漂移。(5) 未成年人数据合规风险（高）——数据飞轮积累 K-9 未成年人偏好/选课数据，未指定最小化/PIA/家长对学习环路单独同意。(6) 过度代理权 LLM06——LLM 编排"考勤核销→财务结算"Webhook，若 LLM 可直接触发财务结算而无人工 gate。(7) 供应链 LLM03。(8) "空壳"过度宣称。(9) legacy 回退削弱 native 性。(10) 成本可辩护性缺口。(11) 预测模型 bolt-on。(12) autopilot vs GDPR Art22 冲突。

**修正后的设计要点**：要达真正 AI 原生，须让概率性/模型逻辑对核心交付负载承载，而非套在确定性求解器外。(1) LLM 真正负载——除 NL→约束翻译外，LLM 须承担求解器做不到的概率推理：隐含偏好推断（从历史推断教师未言明偏好）、课表质量评判（LLM-as-judge 评判课表"好"而非固定适应度）、冲突不可避免时的权衡推理（牺牲哪个偏好及为何）。这些必须在请求路径上且不可被手动配置等价替代。(2) 移除手动配置逃生舱——通过 eval+A/B 证明 AI 推断的软约束权重显著优于手动配置，并 deprecate 手动配置路径。(3) Agent loop 为默认路径——调课由 agent 自主执行（human-on-the-loop 例外），手动降级为异常回退而非默认。(4) 数据飞轮负载——可证系统质量依赖积累偏好数据（fresh install 无历史时质量可测下降），飞轮为核心而非增强；未成年人飞轮数据聚合化存储+PIA+家长对学习环路单独同意。(5) 生成式 UI 为唯一界面，静态表格仅降级回退。(6) 评估闭环——在线 eval+漂移检测+A/B（证明 AI 优于手动）作为持续部署门禁。(7) LLM06 过度代理权——财务结算须人工 gate。(8) 成本可辩护——六行成本分解数值化。(9) 可解释性+申诉通道——对未成年人自动化排课决定提供人工复核+申诉。

**技术实现**：LLM 偏好解析/任务编排用 Claude Sonnet 4.6（复杂多步推理），轻量约束分类用 Haiku 4.5，多模型路由按复杂度分流。求解器 Google OR-Tools CP-SAT（中小规模精确求解）+GA（大规模走班制，TimetablingAI repair 机制），按规模分层。约束建模硬/软约束分离，GA 基因编码=课程-教师-教室-时间四元组，适应度=冲突数+教师负载均衡+教室利用率加权。Agent 框架 LangGraph，工具化求解器/冲突检测/发布 API/考勤财务联动 Webhook。数据层 PostgreSQL+pgvector（排课偏好向量检索，per-tenant 命名空间隔离）；选课预测 LSTM/随机森林。基础设施后台队列 Inngest/Trigger.dev 做异步求解+限流重试；语义缓存；Strangler Fig 增量替换传统排课系统（上游代理拦截，新旧并行，AI 失败回退 legacy）。数据中台 RESTful API/Webhook 推送考勤/财务联动。可观测性每次求解 trace（约束数/耗时/利用率/软约束满足度），LLM-as-judge 评估约束解析准确率，50-100 条 golden 样例排课质量测试套件作为部署门禁。

### 5.5 选课与培养方案管理

**传统形态与痛点**：选课（课程目录浏览检索、加退选、先修课校验、容量与时间冲突检测、候补队列、按年级/院系分时段开放）、培养方案（课程树/模块维护、学分分布规则、按届版本管理、课程与毕业要求映射表）、学分（学分计算与分类、GPA 计算、学期学分上限校验、学分类型核算）、毕业审核（学位要求 checklist、学分完成度核对、先修链核验、缺项报告、人工审核流程）、报表。痛点：选课"凑学分"而非"规划学业"——学生面对几百门课目录不知如何选，靠口碑/水课攻略；热门课秒空，抢课靠手速。培养方案是"静态文档"——规则一旦定下难适配转专业/辅修/海外交流/跨届差异。学分只是"数字累加"——满足学分≠满足培养目标。毕业审核"事后一次性"——到毕业前才跑 checklist，发现缺项已来不及补救。规则变更影响不可见。"满足学分但能力结构畸形"无法被发现。

**AI 原生改造设计**：

- **core_ai_capability**：个性化培养路径生成与动态约束求解引擎——把培养方案重构为可计算的知识图谱（课程-先修-学分规则-能力达成），由 LLM 解析学生意图与教务变更需求并翻译为结构化约束，由约束求解器在图谱上求解最优个性化路径，并对毕业可达性做持续预测。LLM 是意图→约束的翻译与任务路由核心，求解器是优化心脏，二者移除任一即崩溃。
- **redesign**：(1) 培养方案知识图谱化——将培养方案从课程列表/规则文档重构为属性图——课程节点+先修关系边+学分类型边+课程→培养目标/毕业要求（对标工程教育认证 OBE）的能力达成映射。(2) 学生学业状态实时建模——维护"能力达成图谱"——已修课程、成绩、掌握度、按学分类型的缺口、剩余学期容量。(3) 选课 Copilot（意图驱动）——学生用自然语言表达意图（"考研AI方向""下学期要实习想少选""想辅修金融"），LLM 解析为结构化偏好与软约束；约束求解器（OR-Tools CP-SAT/GA，硬约束=先修链/学分类型/容量/时间，软约束=偏好/能力达成均衡/避免水课扎堆）求解，生成 2-3 个候选方案，每个含理由、风险、毕业审核预测影响，可一键应用/撤销重排。(4) 毕业审核前置化与预测——从毕业前一次性 checklist 变为持续性"毕业可达性预测"——每学期末模拟"按当前路径预计第几学期达成所有毕业要求"，提前预警。(5) 培养方案智能演化（管理员侧）——教务用自然语言描述变更，LLM 翻译为约束修改+影响分析。(6) 培养方案健康度看板——基于历史选课数据识别结构性问题。
- **用户体验**：学生不再面对课程目录搜索框+手动凑学分，打开模块见对话式规划器——"你上学期数据结构 B+，离考研AI目标还差这些能力点，本学期建议这样选…"；可对话调整，方案实时重排。教师/教务看培养方案健康度看板、批量毕业审核自动化。家长看孩子学业路径概览与风险（非明细成绩）。
- **agent 循环**：混合架构：模型为核心路由+agentic loop。感知（实时读取学生学业状态、培养方案知识图谱、课程容量与时间表、历史选课效果数据）。规划（LLM 解析意图→结构化偏好/软约束；路由器按复杂度选路径：简单学分核算→规则引擎直接返回；中复杂度路径推荐→GA/CP-SAT；高复杂度跨学期规划→多步 agent+求解器）。工具调用（OR-Tools CP-SAT 约束求解、GA 求解器、知识图谱查询 Cypher、容量查询、毕业审核模拟器、成绩趋势预测 LightGBM）。反思（生成方案后自检；学生采纳/拒绝/调整作强化信号回收；历史"选了推荐课后成绩"归因分析改进推荐）。
- **数据飞轮**：学生采纳/拒绝/调整方案→方案效果数据（选了推荐课后的成绩/能力达成/毕业进度）→路径推荐模型改进+毕业预测准确率 vs 实际毕业→预测模型校准+培养方案变更影响分析 vs 实际影响→影响分析模型改进→更准个性化路径。专有数据壁垒：本校历史选课效果数据。

**移除测试结果（设计自判）**：培养方案退回静态规则文档+课程列表；选课退回目录搜索框+手动凑学分+抢课；毕业审核退回毕业前一次性手动 checklist；学分退回数字累加。设计自判 native。**诚实说明**：选课执行层（抢课/容量/退选/落库）本身是确定性事务，不需要 AI 也必须保留；但模块核心价值主张——路径规划、毕业预测、方案演化、意图驱动选课——完全由 AI 承载，移除则模块退化为 CRUD 空壳。

**应用的标准**：架构标准（Intelligence-First LLM 在选课请求路径上做意图解析与路由、harness governs 选课落库必须服务端二次校验容量与先修权限 LLM 不可直接授权落库、概率性逻辑为核心意图解析概率性+硬约束求解确定性混合架构）；数据与知识标准（培养方案知识图谱化是飞轮基础、上下文工程只送学生状态+相关子图+目标、检索按复杂度、专有数据壁垒本校选课效果数据）；Agent 系统标准（agentic loop、简洁性优先执行层用确定性逻辑、ACI）；产品设计标准（意图驱动生成式 UI 对话式规划器而非搜索框、copilot vs autopilot 路径推荐=copilot 容量抢课执行=autopilot、可纠错可撤销、结果导向）；业务流程标准（AI 在价值交付关键路径、人机协同、headcount decoupling）；评估与可观测性标准；成本与经济模型标准；安全与对齐标准（LLM06 选课落库服务端二次校验、prompt injection 学生意图输入 sanitize）；多模态标准（课程大纲 PDF 外置 artifact）；边缘与端侧标准（路由层最高杠杆）；合规标准（PIPL 未成年人单独同意、GDPR Art22 自动化决策人工复核、AI Act 附件 III 高风险教育水平评估、未成年人数据最小化、可解释性）；组织与团队标准。

**对抗核验结论：赋能伪装。** 诚实呈现如下。

移除全部 LLM+求解器（CP-SAT/GA）后逐项推演：1) 选课执行层（目录浏览/加退选/先修校验/容量与时间冲突/候补/分时段开放）——设计明言"确定性事务，不需要 AI 也必须保留"→照常运行，学生仍能选课、抢课、退选、落库；2) 学分计算/GPA/学期学分上限/学分类型核算——纯算术→照常；3) 培养方案维护（课程树/模块/规则/版本/课程-毕业映射）——确定性 CRUD→照常，教务仍可手动改规则；4) 毕业审核——设计自承"退回毕业前一次性手动 checklist"且"须人工复核"（GDPR Art22/AI Act Annex III）→照常（仅从事前预测退化为期末一次性核对），学位仍能授予；5) 报表——确定性 SQL 聚合→照常。仅移除 LLM（保留求解器）：NL 意图解析断→学生退回目录搜索框（设计承认该 fallback 存在）；毕业可达性模拟（求解器）仍可跑；教务变更退回手动改规则——模块主体照常。

结论：机构强制性法定/事务性核心动作全部存活，"模块退化为 CRUD 空壳"表述误导——一个能正确选课、算学分、授予学位的"CRUD"就是该模块的真实产品本体，AI 层（路径规划/毕业预测/变更影响/健康分析）是高价值但可选的顾问层且有完整确定性 fallback，故未通过严格移除测试。叠加域级法律上限：学位授予与毕业审核依 GDPR Art22/AI Act Annex III 不得纯自动化、须人工复核+确定性可审计，该域法律层面禁止把核心动作做成概率性 AI，AI 被锁死在 copilot 位置，无法达到 translation-app 级的 truly_native。

**反模式风险**：(1) 套壳聊天框——保留'目录搜索框'作为 fallback，对话式规划器可能沦为套在旧目录之上的聊天 UI。(2) AI 当 if-else/路由器硬编码。(3) mega-prompt 风险。(4) evals 覆盖面不足。(5) 未成年人数据合规缺口——数据飞轮用未成年人选课效果数据训练推荐模型，属 PIPL Art31 敏感个人信息。(6) 过度代理权（LLM06）变更路径——LLM 把教务 NL 翻译为培养方案图谱约束修改，影响面巨大，缺 canary/爆炸半径限制/灰度。(7) 供应链（LLM03）。(8) 预测模型误报成本——LightGBM 70-85% 准确率的毕业预测假阴性代价高。

**修正后的设计要点**：(1) 让培养方案知识图谱成为毕业审核的唯一数据模型，退役并行的静态 checklist——人工是对图谱计算结果的复核，而非另跑一套确定性 checklist。(2) 退役"目录搜索+手动凑学分"作为主入口，使 AI 路径规划成为选课的唯一发起路径；确定性执行（容量/先修/落库）作为规划结果的下游服务端强制校验，而非并列替代入口（消除 fallback 即消除套壳退路）。(3) 培养方案变更只能经"NL→图谱约束突变→人工批准→canary 灰度"路径，取消可手动直接改规则表的后门；图谱即规则，无并行规则表可手编。(4) 把数据飞轮接入审核校准（预测准确率反馈进图谱能力达成权重），使审核质量结构依赖反馈环；并对未成年人飞轮训练补 PIPL 家长同意/PIA/单独同意/离校删除/训练数据最小化。(5) 培养方案健康分析作为唯一治理信号，退役静态"执行率报表"。(6) 给变更路径加 canary+爆炸半径限制，补 SBOM/供应链扫描与预测误报人工升级通道。(7) 明确 LLM 工具调用 schema 与边界防 mega-prompt 脆化；扩展 evals 覆盖确定性核心与线上 LLM-as-judge gating。**诚实边界**：即使做到以上，学位终审仍须人工法定复核（GDPR Art22/AI Act 域约束不可解除），AI 可成承重计算但非终审，这是该域能达到的"最接近 native"上限。

**技术实现**：模型选型意图解析与 NL→约束翻译用 Claude Haiku（简单分类/抽取）与 Sonnet（复杂多步规划）；多模型路由按复杂度分流。检索培养方案知识图谱 Neo4j 存课程-先修-学分规则-能力达成关系；课程语义检索 pgvector 存课程描述 embedding。agent 框架 LangGraph 自研 agent loop；工具含 OR-Tools CP-SAT、GA 求解器（对标 TimetablingAI 的 repair 机制）、知识图谱查询 Cypher、容量查询、毕业审核模拟器、成绩趋势预测 LightGBM。数据层关系库（选课记录/成绩/容量）+图库（培养方案）+向量库（课程语义）+事件流（选课变更推送考勤/财务）。基础设施语义缓存（Redis+embedding 相似度>0.95 命中降本 30-50%）、后台异步队列（Inngest/Trigger.dev 做批量毕业审核与方案影响分析）、评测门禁（golden 路径方案集+毕业预测准确率作 CI 门禁）、Strangler Fig 增量替换（选课执行层先 replatform，路径规划新服务并行，失败回退 legacy，严禁 Big Bang）。合规境内存储默认、跨境模型调用走标准合同、离校后按留存期自动删除（对标 PIPL 第38/47条）。

### 5.6 教师与教学评价管理（评教）

**传统形态与痛点**：评教问卷设计（固定量表教学态度/内容/方法/效果若干维度，每维1-5分+开放文本框）、评教任务下发（期末统一开放）、评教数据回收（统计均分、标准差、班级/院系排名）、教师查看评教报告、评教结果存档（作为教师考核/晋升参考）、异常评教人工排查。痛点：期末一次性、滞后——学期结束才出结果，教师已无法对本届学生改进；量表维度固定、打分主观——"教学态度4.5分"无法告诉教师具体问题在哪、归因到哪个可干预维度；大量开放性评教文本无人读、无结构化利用；评教与改进脱节——教师拿到分数排名却无改进建议；学生敷衍/恶意——刷好评或报复性差评；教师抗拒——被考核感强；无过程性证据。

**AI 原生改造设计**：

- **core_ai_capability**：LLM 对多模态教学证据（评教文本+课堂转写要点+学生学业表现delta+互动信号）的语义理解、归因诊断与个性化改进建议生成——把"评教"从"事后打分问卷"重构为"持续的教学质量理解与教学改进闭环"。移除该能力，模块退化为问卷录入+分数统计，"教师教学质量分析"与"教学改进"两项职责直接消失。
- **redesign**：(1) 评教从"期末量表"改为"持续微型反馈"——学期中分多次、以对话式/触发性方式采集（某节课后学生可一句话反馈），LLM 实时聚类成主题（如"概念讲解跳跃""板书太快""作业反馈及时"），教师周期性看到"本周学生关注主题"而非分数。(2) 文本证据归因——LLM 对评教文本做主题建模+归因到可干预维度（讲解清晰度/互动设计/反馈时效/难度梯度），并关联课堂转写与学生学业表现变化做交叉验证。(3) 个性化改进建议生成——参考 Khanmigo for Teachers 教师侧定位，检索教学法知识库（RAG），为每位教师生成"下周可做的2-3件具体改进动作"而非抽象建议，标注证据来源页码供追溯。(4) 改进效果追踪——教师采纳改进动作后，系统追踪下个反馈周期学生主题变化与学业产出delta，评估改进有效性形成闭环。(5) 异常评教识别——LLM 语义识别敷衍填写/恶意差评/刷分模式，标记低置信数据不进入诊断。(6) 班级/学科共性诊断——聚合多教师诊断，给院系管理者"共性教学薄弱点+资源调配建议"。
- **用户体验**：学生不再填长问卷，可在任意课后用一句话/情绪标签即时反馈，看到"你的反馈已被教师看到并改进"的闭环回执。教师不再看冷分数排名，而是周期性收到"教学质量画像+本周学生关注主题+2-3条可执行改进建议+证据链接"，可一键采纳/修订/拒绝建议，改进效果有追踪回显。院系管理者看到聚合的教学质量趋势与共性诊断而非个人排名博弈。教师考核：AI 诊断作为"改进支持"输入，但评价判定权在人工委员会。
- **agent 循环**：感知→规划→工具→反思循环：感知层持续采集评教文本片段、课堂转写要点、学生学业表现时序delta、出勤/互动信号、历史改进记录；规划层构建教师教学质量画像，识别强项/薄弱维度并归因到可干预维度，按复杂度选规划模式（单点反馈→轻量诊断；跨学期多证据→多步推理）；工具层 RAG 检索教学法知识库找改进策略、对比同类课程/教师基线、生成结构化改进计划（含可执行动作、预期影响、证据引用）；反思层追踪教师采纳的改进动作→下周期信号变化→LLM-as-judge 评估诊断质量与建议有效性→更新画像与知识库，对错误诊断降置信。
- **数据飞轮**：评教文本/课堂证据→LLM 诊断与建议→教师采纳改进动作→下周期学生反馈与学业产出变化→系统评估改进有效性→有效策略沉淀进教学法知识库、无效/错误诊断降权→更准的下次诊断。教师"采纳/修订/拒绝"建议的显式反馈是关键训练信号，形成"更多教师使用→更多改进效果数据→更优诊断建议"的闭环。

**移除测试结果（设计自判）**：抽掉 LLM 语义理解与诊断能力后，模块退化为"问卷录入+分数统计+排名报表"——评教文本无人读、无法归因到可干预维度、无法生成个性化改进建议、无法追踪改进效果闭环，"教师教学质量分析"与"教学改进"两项模块职责直接消失。设计自判 native。

**应用的标准**：架构标准（模型在请求路径上、概率性逻辑为核心评教诊断与改进建议生成由 LLM 在请求路径上完成）；数据与知识标准（飞轮、上下文工程、检索按复杂度、专有数据壁垒教学法知识库）；Agent 系统标准（agentic loop、简洁性优先、ACI）；产品设计标准（生成式 UI 教师端画像+主题+建议而非静态报表、copilot 改进建议可采纳/修订/拒绝、可纠错可撤销、结果导向交付改进结果而非评教工具）；业务流程标准（AI 在关键路径、人机协同评价判定 human-in-the-loop 诊断改进 on-the-loop、headcount decoupling）；评估与可观测性标准；成本与经济模型标准；安全与对齐标准（LLM06 AI 诊断不可直接作为教师考核授权决策、prompt injection 学生评教文本输入过滤、文本蕴含防幻觉）；多模态标准（课堂转写外置 artifact）；边缘与端侧标准（云-边-端三层）；合规标准（AI Act 附件 III 评价学习成果高风险、GDPR Art22 人工复核、PIPL、可解释性诊断引用证据页码、未成年人数据最小化）；组织与团队标准。

**对抗核验结论：truly_native（条件修正）。** 诚实呈现关键自相矛盾。

逐层移除 AI 调用后的推演：(1) 评教采集层——redesign 已将"期末统一量表"替换为"课后一句话/情绪标签持续微型反馈"。移除 LLM 后，CRUD 仍能落库一句话文本，但"LLM 实时聚类成主题"消失，教师拿到的是一堆无结构原始文本而非"本周学生关注主题"。关键：由于期末量表已被 redesign 移除，不存在"退回打分量表兜底"的路径。(2) 文本证据归因——LLM 主题建模+归因+课堂转写与学业 delta 交叉验证——整条链路由 LLM 完成，移除后完全消失，无确定性规则可替代语义归因。(3) 个性化改进建议——RAG 检索教学法知识库+生成"下周 2-3 件具体改进动作+证据页码"——移除后教师无任何改进输入，该职责归零。(4) 改进效果追踪——数据采集可留，但"评估改进有效性"的语义判断消失，闭环断裂。(5) 异常评教识别——可退化为纯规则但语义识别消失（属辅助非核心）。(6) 班级/学科共性诊断——聚合统计可做但"共性教学薄弱点+资源调配建议"的语义诊断消失。

核心结论：四项用户交付物——教学质量画像、本周关注主题、可执行改进建议+证据追溯、改进效果闭环——移除 AI 后全部无法生成。设计声明的两项核心职责"教师教学质量分析"与"教学改进"直接消失，只剩无结构一句话文本的原始采集壳。期末量表既已被移除，无 AI 时不存在可用的"分数统计+排名报表"兜底交付。核心流程崩溃→通过移除测试。

**但必须修正设计中的自相矛盾（最致命）**：removal_test 文字称无 AI"退化为问卷录入+分数统计+排名报表"，这与 redesign_detail"改为持续微型反馈"冲突。若团队按字面实现（保留传统量表+排名作为兜底交付），AI 将退化为传统评教之上的增强层→滑入赋能伪装。必须澄清并锁定：分数统计/排名不是独立用户交付物，仅作 AI 诊断的输入证据。

**反模式风险**：(1) removal_test 自相矛盾风险（最致命）——若按"分数统计仍照常运行"落地，传统报表兜底存在→AI 沦为增强层→赋能伪装。(2) 未成年人数据合规盲区——评教主体为学生（中国语境下大量未成年），缺失未满14岁儿童评教需监护人同意、课堂转写采集学生声音/行为属敏感个人信息需单独明示同意、学业表现 delta 跨界用于评教教师违反 PIPL 目的限制原则。(3) 改进效果归因科学性风险——用"学生主题变化+学业 delta"评估教师改进有效性，但学业 delta 受多重混淆变量影响，反事实归因困难，飞轮可能沉淀错误策略污染知识库形成负飞轮。(4) 知识库自动沉淀污染风险——未指定人工 curation gate。(5) 在线 eval 指标薄弱。(6) 套壳聊天框残余风险。(7) 课堂转写 artifact 外置未明确。(8) mega-prompt 退化风险。

**修正后的设计要点**：(1) 修正 removal_test 自相矛盾（最高优先级）——在架构图中明确标注"期末量表+分数统计+排名报表"已从用户交付物中移除，分数仅作 AI 诊断的输入证据而非独立输出。无 AI 时不产生任何可用分数/排名交付，只剩无结构一句话文本采集壳；四项核心交付物的生成路径必须经过 LLM，无确定性 fallback。(2) 补齐未成年人/敏感数据合规——未满14岁学生评教须监护人同意流程或采用不采集身份的匿名化反馈通道；课堂转写作为敏感个人信息单独同意、最小化留存、定期销毁；学业 delta 仅以聚合/脱敏形式参与教师诊断不得回溯具体学生身份。(3) 改进效果归因置信度处理——引入对照基线，改进有效性以"置信区间+效应量"输出而非二元判定；低置信不自动沉淀知识库，标记"待人工复核"。(4) 知识库 curation gate——教学法知识库新增/更新须经教学专家人工审核或 LLM-as-judge+人工抽检双闸门，禁止 LLM 无监督自动写入。(5) 在线 eval 量化指标——定义改进采纳率、建议被执行率、下周期负面主题信号下降率、教师 NPS、学业 delta 对照基线效应量等在线指标并接入 trace 看板。(6) 教师端强制生成式 UI——交付为结构化卡片而非聊天框；copilot 控制为显式操作。(7) 课堂转写外置 artifact——长转写/音视频以 artifact 引用、按需检索加载，prompt 中只放高信号摘要+引用，稳定前缀缓存。(8) 成本六行分解落地。

**技术实现**：模型选型多模型路由——评教文本聚类/抽取/异常检测用便宜模型（GPT-4o-mini/Claude Haiku 约60%流量）；归因诊断与改进建议生成用中端（Claude Sonnet 约30%）；跨学期多证据复杂推理用高端（约10%）。检索教学法知识库用 RAG，pgvector 起步、百万级向量迁 Milvus/Qdrant；参考 Jill Watson 的 DPR 双编码器+多表示存储；top-k 检索分批+文本蕴含幻觉检测。Agent 框架感知层（持续采集评教文本/课堂转写/学业 delta，后台队列 Inngest/Trigger.dev 异步处理）→规划层→工具层（RAG 检索、同类基线对比、改进计划生成）→反思层（效果追踪+LLM-as-judge）。数据层评教文本向量库+教师教学质量时序画像+教学法知识库+改进效果追踪表；per-tenant（院系）命名空间隔离。基础设施语义缓存（相似度>0.95 命中降 30-50% 调用）、provider 故障转移、流式响应、CI/CD 内建评测门禁（50-100 条 golden 诊断样例）。guardrails：学生评教文本 prompt injection 检测、Moderation API 输入输出双向、文本蕴含防幻觉、诊断必须引用证据页码、低置信兜底"建议人工复核"。可解释性每条诊断/建议附证据追溯链。

### 5.7 通知与家校沟通

**传统形态与痛点**：通知管理（创建/编辑/删除通知、手动选择接收人、定时/即时发送、多通道推送、已读状态统计报表）、家校沟通（教师-家长一对一消息收发、班级群/年级群聊、消息模板、离线消息与未读提醒）、消息中心（收件箱列表、已读/未读标记、消息分类/置顶、关键词搜索、消息存档与导出）、答疑（人工维护 FAQ 知识库、家长/学生提问→教师排队回答）、通知模板与统计。痛点：通知千人一面、家长被淹没（教师群发同一模板通知，关键信息淹没在事务性通知中，已读率与有效阅读率极低）；沟通成本随学生数线性增长（一名班主任面对几十个家长，重复回答相同问题）；答疑排队、响应滞后（教师无法7×24在线）；通知与孩子真实情况脱钩；沟通内容零沉淀；危机沟通被动滞后。

**AI 原生改造设计**：

- **core_ai_capability**："意图理解+基于学生全息画像的个性化沟通生成+RAG 自适应答疑+风险信号主动触发"四位一体的对话式 AI 引擎。声明是模块心脏：移除它，通知退回模板广播、沟通退回人海一对一、答疑退回 FAQ 排队、主动预警沟通消失。模型在请求路径上而非旁路外挂——家长每条消息都过 LLM 做意图理解与回复生成。
- **redesign**：四个子能力按 copilot/autopilot/out-of-the-loop 三级重构。(1) 通知：从"教师手写广播"到"意图驱动的个性化生成式通知"——教师只描述意图，AI 基于每个学生画像生成个性化通知草稿（对成绩下滑学生家长强调沟通重点与建议，对进步学生家长表达肯定，对正常学生家长简洁传达）；通知优先级由 AI 基于风险信号自动排序；同一意图自动生成多渠道版本。(2) 家校沟通：从"教师全手动回复"到"AI copilot 草拟+关键对话路由人工"——家长消息进来先经 AI 意图理解，检索学生画像后生成草拟回复供教师一键审核/编辑/发送；涉及心理危机、投诉、敏感话题的对话 AI 自动升级给人工并标注风险等级。(3) 答疑：从"FAQ 表+教师排队"到"AI 助教7×24答疑+苏格拉底式引导"——采用 Jill Watson 模块化技能路由架构（共指消解→技能分类→RAG 检索→答案生成→文本蕴含幻觉检测→Moderation 双向过滤）；学业类问题采用 Khanmigo 苏格拉底式"给提示而非直接给答案"；置信度低的路由给教师。(4) 消息：从"收件箱列表"到"意图聚合+智能摘要"——家长端 AI 把多来源消息按主题聚合并生成每日"今日关注"摘要，关键信息置顶；教师端 AI 聚合班级共性消息生成"今日待办"。
- **用户体验**：家长每天收到一条 AI 生成的"今日关注"摘要而非数十条零散通知；可直接问 AI 助教"孩子这周作业完成情况/下周考试安排/请假流程"；收到的是与孩子真实情况相关的个性化沟通而非模板；可长按消息报告不准确。教师不再手写通知，描述意图即可，AI 生成个性化草稿供审核；常见家长问题由 AI 助教自动应答，教师只处理 AI 路由上来的关键对话。学生随时向 AI 助教提问作业疑难，获苏格拉底式引导。管理员看板显示全校沟通健康度、AI 答疑覆盖率与解决率、家长满意度、人工升级率、成本与质量趋势。
- **agent 循环**：对话式 AI Agent 的 agentic loop：感知（监听学生信号源→实时计算学生风险状态；监听家长/学生消息流→共指消解+意图分类+情感检测；监听通知事件→通知意图）。规划（路由为最高杠杆：意图路由（通知/答疑/沟通/升级）；复杂度路由——简单 FAQ→单步 RAG 直接答廉价模型，开放性学业问题→多步苏格拉底引导高端模型，敏感/危机→升级人工；copilot/autopilot/out-of-the-loop 三级）。工具调用（ACI 工程化接口：retrieve_student_profile 获取最小高信号画像、rag_search 检索知识库 DPR+BM25 混合+重排、generate_notification 生成多渠道通知、check_consent 合规检查、guardrails_check Moderation 双向+prompt injection 清洗、escalate_to_human 升级人工、entitlement_validate 服务端权限二次校验防 LLM06）。反思（答疑后做文本蕴含幻觉检测；沟通草稿生成后追踪教师采纳率；家长长按报告不准确→进数据飞轮）。
- **数据飞轮**：交互→数据：家长/学生提问→意图分类标注与未覆盖问题识别；教师审核 AI 草稿→采纳/编辑/拒绝信号；家长阅读/回复行为→通知效果与渠道偏好数据；AI 答疑满意度评分与"报告不准确"→质量信号；人工升级对话→关键对话模式。数据→改进：意图分类模型按真实分布 fine-tune；通知生成 prompt 按采纳率优化；RAG 知识库按未覆盖问题自动扩充。更好体验→更多交互：AI 答得准→家长更愿用 AI 助教而非排队等教师→更多真实提问→更多标注数据→更准。闭环关键：人工升级的"教师实际如何回答"作为高质量监督样本回流，持续提升 AI 自主应答边界。

**移除测试结果（设计自判）**：抽掉 AI 后通知退化为模板广播+手动选接收人，千人一面、关键信息淹没，已读率塌陷；家校沟通退化为教师一对一手动回复，无法应对规模；答疑退化为 FAQ 表+教师排队，7×24 响应消失；消息退化为收件箱列表无聚合无摘要；学业预警/考勤异常/心理危机无法主动触发个性化家长沟通，错过 EWS 干预黄金窗口；沟通数据无法结构化反哺学生画像与预警系统。模块退化为传统 CRUD 通讯工具。设计自判 native。

**应用的标准**：架构标准（模型在请求路径上家长每条消息都过 LLM、harness governs、概率性逻辑为核心意图分类/情感检测/风险路由均为概率性判定）；数据与知识标准（飞轮、上下文工程 retrieve_student_profile 仅取沟通所需最小高信号 token、检索按复杂度、专有数据壁垒学生全息画像+历史沟通+校规知识库）；Agent 系统标准（agentic loop、简洁性优先技能路由先分流简单问题、ACI 工程化 7 个规范化工具接口）；产品设计标准（意图驱动生成式 UI 家长端"今日关注"摘要动态生成、copilot/autopilot/out-of-the-loop 三级分明、可纠错可撤销、结果导向交付"家长已知晓并理解关键信息"而非"已发送通知"）；业务流程标准（AI 在价值交付关键路径、人机协同教师 on-the-loop 审核、headcount decoupling AI 承担80%常见沟通）；评估与可观测性标准；成本与经济模型标准；安全与对齐标准（LLM06 AI 不自主发送危机通知/授权动作、LLM03 供应链、prompt injection 防护家长消息输入清洗、guardrails Moderation API 输入输出双向+文本蕴含防幻觉+IDK 兜底）；多模态标准（作业图片/通知附件外置 artifact）；边缘与端侧标准（云-边-端三层、路由层最高杠杆）；合规标准（PIPL 第31条不满14岁未成年人单独同意、第38条境内存储与跨境合规、GDPR 第22条重大影响决定人工复核、可解释性 AI 答疑引用文档页码）；组织与团队标准。

**对抗核验结论：truly_native。** 诚实呈现滑落风险与修正。

逐流程严格推演（今晚移除全部 LLM 调用）：(1) 通知：传输层 CRUD（标题/正文/接收人/定时/多通道发送/已读统计）仍照常运行——这是赋能伪装的"照常运行"信号。但意图驱动个性化生成、风险信号自动排序、多渠道版本自适应全部消失；对200人班级手写200条个性化通知人工不可行（headcount 不可替代），千人千面能力实质崩溃。(2) 家校沟通：消息收发传输存活，AI copilot 草拟+对话风险升级消失，教师重回人海一对一，"AI 承担80%"的去人头解耦失效。(3) 答疑：FAQ 表+教师排队存活但 7×24 AI 助教消失——而 7×24 是教师物理上无法规模化提供的（不可由堆人力替代），晚间/周末辅导归零。(4) 消息：收件箱/搜索/存档 CRUD 存活，"今日关注"智能聚合摘要消失，家长重回被数十条通知淹没。(5) 主动预警：EWS 风险检测若为规则可部分存活，但 AI 个性化家长沟通草稿+自动触达消失，错过干预黄金窗口。(6) 报表：已读率/发送成功率存活，AI 原生 outcome 指标消失。

判定：传输/CRUD 确实"照常运行"（赋能伪装信号存在），但三大差异化核心价值（千人千面沟通、7×24 答疑、主动风险触达）全部崩溃，且其中 7×24 答疑与规模化个性化是"堆人力也无法替代"的 AI 负载核心，非"多个智能助手"。模型在请求路径（家长每条消息过 LLM）、agent loop 为架构骨架、飞轮/eval/copilot-autopilot-out-of-loop 分级均到位。故不满足"AI 只是多了个智能助手"的赋能伪装/bolt-on 判定条件，判 truly_native；但存在向赋能伪装滑落的实现风险，须按修正项修正以坐实原生性。

**反模式风险**：(1) 赋能伪装滑落风险——设计自述移除 AI 后"退化为模板广播/FAQ 表/收件箱"，即把传统路径当作可运行的降级地板。若实现中保留手动模板编辑器/FAQ 表/收件箱为默认主界面、AI 生成仅作可选"智能"按钮，则滑为"传统系统+AI 润色层"的赋能伪装。(2) 未成年人敏感数据飞轮训练合规盲区——飞轮用师生家长对话（含心理危机/投诉标记对话）fine-tune 模型，但 check_consent 仅是渠道同意，缺"AI 处理同意"与"数据用于训练同意"分层。(3) 主动风险触达误报伤害（LLM06 邻近）——AI 基于风险信号自动生成家长沟通，误报将"你家孩子有心理危机"错送家长，对未成年人造成严重伤害；copilot 审核不足以控制"已审核的假阳性"伤害，且缺精度优先阈值与临床复核。(4) 飞轮采纳信号造假——教师对 AI 草稿"一键发送"橡皮图章式采纳，采纳率成为无意义噪声。(5) eval 形同虚设——50-100 条 golden 对多校多学科答疑覆盖过薄，且未明确为发布阻断门禁，缺红队/学生越狱对抗样本。(6) 6-role Jill Watson 流水线过度工程。(7) 硬编码路由 60% Haiku/30% Sonnet/10% Opus 为固定配比非数据驱动。(8) 错误成功指标——以传统"已读率/发送成功率"为 KPI 会优化阅读回执而非沟通有效性。

**修正后的设计要点**：(1) AI 生成设为通知与答疑的默认主路径，手动模板/FAQ 表/收件箱降为显式"降级模式"（仅 AI 不可用或人工显式覆盖时启用并标注降级），从产品形态上消除"传统系统为地板、AI 为天花板"的赋能结构。(2) 合规分层——拆分"同意 AI 处理并响应"与"同意数据用于模型训练"两套同意；心理危机/投诉等敏感数据默认排除训练，需单独显式同意+DPIA；设危机数据留存上限与删除通道；风险判定对家长可解释（PIPL 第24条/GDPR 第22条）。(3) 风险触达精度优先+临床复核——心理危机类主动沟通须辅导员/临床人员复核（非仅教师），精度优先（宁漏报勿误报），家长话术受控模板化，提供 opt-out，并设假阳性率趋零 KPI。(4) 飞轮信号去噪——按教师审核编辑距离区分"实质审核"与"橡皮图章"，仅前者入训练；样本审计；追踪"未采纳/重写"负信号。(5) eval 加厚并阻断发布——每校每知识域分层 golden 集+红队/学生越狱对抗样本+LLM-as-judge 双维（准确度与共情与安全分别评分）；eval 作为发布阻断门禁（不达标不上线），非报告型。(6) 流水线精简——起步 retrieve→generate→verify+guardrail 最小 agent loop，仅当 eval 证明必要才加角色。(7) 路由动态化——60/30/10 仅作起点，路由决策按置信度/复杂度动态+eval+成本数据持续调优，A/B 路由策略，trace 暴露路由依据。(8) 成功指标 AI 原生化——主 KPI 改为 AI 自主解决率（≥70% 目标）、家长理解并行动率、风险触达精度/召回、假阳性危机沟通率（趋零）、教师生均节省时间；已读率降为辅指标。

**技术实现**：模型选型与多模型路由——意图分类/情感检测/简单 FAQ 答疑→Claude Haiku 4.5 或 GPT-4o-mini（约60%流量）；个性化沟通生成/苏格拉底学业引导→Claude Sonnet 4.6（约30%）；复杂多步推理/危机判断→Opus 4.5（约10%）。provider 故障转移（Anthropic/OpenAI 双供应商）。检索 DPR 双编码器+BM25 关键词检索混合+cross-encoder 重排 top-20→top-5；知识库分≥500 字符段落、50% 重叠，每段存多表示（原文/标题/摘要/embedding）。Agent 框架参考 Jill Watson 模块化技能路由，6 个 LLM 角色流水线；Context Loop 迭代检索至多4批；工具调用规范化为7个 ACI 接口。数据层 pgvector 起步，向量超1M后迁 Milvus/Qdrant，per-tenant 命名空间隔离；学生画像实时特征存 Redis；沟通历史时序存储供飞轮训练。基础设施语义缓存（embedding 相似度>0.95 命中降本30-50%）；后台队列（Inngest/Trigger.dev）异步 embedding 与批量通知生成+限流重试；流式响应（time-to-first-token 优先）；CI/CD 内建 eval 门禁（50-100 条 golden 样例回归）。安全与合规 OpenAI Moderation API 输入输出双向；文本蕴含幻觉检测；prompt injection 输入清洗（家长消息去指令）；entitlement_validate 服务端权限二次校验；境内模型部署/调用（PIPL 第38/40条）；数据最小化中间层；PIA 评估。可观测每次调用 trace 记录（意图/模型/token/成本/采纳率/升级结果）；LLM-as-judge 离线评估回复质量与共情度。

### 5.8 学业预警与干预

**传统形态与痛点**：挂科阈值预警（教师/辅导员手工录入成绩、考勤后系统按固定阈值 GPA<2.0、单科<60、缺勤>N次触发红黄灯标记）、重点关注名单 CRUD（辅导员手工维护重点学生台账，手工录入学生状态与谈话记录）、辍学风险事后统计（按休学/退学率做学期末统计报表，无法预测）、心理风险人工上报（心理测评问卷一年一次+辅导员线下上报+纸质归档）、干预工单手工流转（手工创建干预任务，线下执行后回系统补录结果）、预警报表导出、辅导员纸质/Excel 台账。痛点：滞后性致命——阈值法只能事后告警，挂科已发生才触发，错过干预黄金窗口（AI 比阈值法提前约3周识别风险，干预成功率随阶段骤降 80%→20%）；单一数据源——仅靠成绩/考勤，忽略 LMS 数字足迹、校园生活、NLP 文本信号，无法识别"表面正常但内部已脱离"的学生，错过约70%高信号数据；规则僵化误报漏报并存；干预断裂无闭环；心理风险盲区；不可解释无行动性；人海战术低效。

**AI 原生改造设计**：

- **core_ai_capability**：多模态风险综合画像引擎——以学业+LMS行为+校园生活+NLP文本四源异构数据实时融合，输出"可解释、可溯源、随时间演化"的学生风险综合分与根因归因，并驱动 MTSS 分层干预闭环。这是预警→干预→跟踪全链的心脏，移除即退化为阈值告警表格。
- **redesign**：(1) 数据感知层（四源融合）——教务（成绩趋势/选课/考勤）、LMS（登录频次/深夜提交/视频时长/作业提交时序）、校园生活（图书馆/食堂/宿舍门禁/一卡通消费规律性与经济压力指数）、NLP文本（课程讨论帖/作业/求助台工单中抑郁-焦虑-孤独-自杀意念的语言标记）。每类数据按知识图谱对齐到"学生风险画像"实体。(2) 分层风险建模——学业风险=双层 LSTM 时序趋势预测+随机森林挂科分类；行为风险=K-means 聚类+学术参与度与学习规律性；心理风险=NLP 文本分类器检测负性情绪与自杀意念标记；综合风险=0.7×学业+0.3×行为，心理风险独立通道触发优先级提升。每条风险分强制 SHAP/LIME 给 top-3 归因特征。(3) 预测而非告警——不只告警已挂科，而是预测"未来4周挂科/辍学概率"——在 Yellow 区干预可预防的辍学是 Red 区的4-5倍。(4) MTSS 五步闭环——Detection 模型每日增量识别→Prioritization 每日8AM前生成"晨间优先简报"→Action AI 起草个性化沟通一键审核发送→Tier Recommendation 自动按风险+历史建议干预层级→Tracking 跟踪哪些干预对哪些学生有效并沉淀机构知识。(5) 多模型路由降本——风险分类/抽取走轻量模型，根因分析与沟通草稿走中端，自杀意念检测走审核过的专用 BERT/Moderation。语义缓存相似案例归因结果降本30-50%。
- **用户体验**：学生侧低介入非惩罚性——不打分不给标签，在恰当时机通过熟悉渠道收到"本周学习状态微调建议"，主动求助时即时响应；心理风险学生被专业顾问"恰好出现"地外联，系统绝不告知学生"你有心理风险"。教师/辅导员侧从"查名单补台账"变为"审核干预决策"——每天收到 AI 生成的晨间优先简报（3-5个最需关注学生+每人可解释根因+一键审核发送的个性化干预建议）。家长侧在 Yellow/Orange 区收到 AI 起草、辅导员审核的个性化沟通，内容聚焦"如何支持"而非"你孩子有问题"。管理员侧看到"干预有效性看板"。
- **agent 循环**：Agent化设计——"学生风险守护 agent"闭环：感知（每日增量拉取四源数据更新学生风险画像，向量库 embedding 检索历史相似干预案例）。规划（对进入风险阈值的学生，agent 按 MTSS 分层选择工具组合；检索范式按复杂度动态选择：简单单点风险查询用向量召回相似历史案例；复杂根因归因用多步 RAG+LLM 推理）。工具调用（5个 ACI 工程化工具：生成个性化沟通草稿、创建干预工单并派单、触达学生、转介心理咨询/学业辅导、拉取学生最近 NLP 文本做情绪复核。高风险动作必须 human-in-the-loop，agent 起草、辅导员一键审核后才执行，服务端二次权限校验防过度代理权 LLM06）。反思（跟踪干预后2-4周风险分变化，评估干预有效性，更新"干预策略→学生类型→效果"知识库，失败案例触发策略调整）。
- **数据飞轮**：每次风险识别→干预执行→效果跟踪产生闭环数据：干预前后风险分变化成为模型正负样本；辅导员对 AI 建议的接受/编辑/拒绝成为偏好学习信号；哪类干预策略对哪类学生有效沉淀为机构知识库；学生主动求助与响应模式修正触达策略；每30天用新数据重训风险模型（仅验证准确率提升才更新）。更多学生交互→更精准风险归因→更有效干预建议→更高留存→更多数据，形成飞轮。

**移除测试结果（设计自判）**：抽掉 AI 后多源数据融合能力消失，只能用成绩/考勤阈值，丢失约70%高信号数据；风险分退化为固定阈值红黄灯，无法预测未来4周挂科/辍学概率，只能事后告警，错过黄金干预窗口；可解释根因+个性化干预建议消失，辅导员重新面对无原因的红点无法精准行动；晨间优先简报、AI 起草沟通、自动 MTSS 分层建议、效果跟踪闭环全部消失，退回手工台账；心理风险盲区重现。模块退化为"挂科阈值告警+手工干预台账"空壳。设计自判 native。

**应用的标准**：架构标准（Intelligence-First 每次风险识别、根因归因、干预建议生成都是模型推理而非固定规则分支、harness governs 框架治理 agent 工具调用边界、概率性逻辑为核心风险分是概率预测非确定阈值）；数据与知识标准（飞轮、上下文工程每生只取top-3高信号归因特征、检索按复杂度、专有数据壁垒四源融合+干预效果知识库）；Agent 系统标准（agentic loop、简洁性优先 agent 只做风险守护单职责、ACI 5个工具明确签名与权限边界、规划模式按任务选）；产品设计标准（意图驱动生成式 UI 晨间优先简报、copilot vs autopilot 学习建议 autopilot 家长沟通/心理转介 copilot、可纠错可撤销、结果导向）；业务流程标准（AI 在价值交付关键路径、人机协同 on-the-loop、headcount decoupling 辅导员管理学生数从百级扩到数百级）；评估与可观测性标准；成本与经济模型标准；安全与对齐标准（LLM06 高风险动作必须人工审核、LLM03 供应链、prompt injection 学生 NLP 文本输入做注入检测、guardrails 心理风险触达话术受 Moderation 双向过滤）；多模态标准（学生长文本/作业原文外置为 artifact）；合规标准（GDPR Article 22 不得纯自动化做对学生产生重大影响的决定、PIPL 第28/31条未成年人敏感信息单独同意+专门规则+PIA、数据最小化禁用种族/收入等人口统计变量、可解释性 SHAP/LIME 归因满足 EU AI Act Annex III 透明度义务、数据境内化、删除权）；组织与团队标准。

**对抗核验结论：truly_native（条件式）。** 诚实呈现关键透明度问题与合规矛盾。

分两层移除推演。【广义移除：抽掉全部 ML+LLM】四源数据摄入管道（教务/LMS/校园生活/NLP文本）属数据工程非 AI，仍能运行；但风险综合分（双层 LSTM+随机森林）、未来4周挂科/辍学概率预测、NLP自杀意念检测（BERT）、SHAP/LIME根因归因全部失效→MTSS自动分层、晨间优先简报、AI起草沟通、效果跟踪闭环全部消失。重点名单/工单/报表的 CRUD 骨架仍可运行，但沦为"无智能内容填充的空数据库"。模块退化为设计自身已判定"滞后失效"的传统阈值告警+手工台账空壳。→ AI 是价值承重墙，通过原生性测试。

【狭义移除：仅抽掉 LLM，保留 LSTM/RF/BERT】风险评分与"提前3周预测/85%准确率"仍工作、SHAP归因保留、自杀意念 BERT 检测保留→核心风险识别价值存活。失效的是：沟通草稿生成、LLM-as-judge 质量评估、多步 RAG+LLM 复杂根因推理、agent Plan 步的 LLM 工具选择。即 LLM 主要承重"沟通/评判/复杂推理增值层"，而"风险预测核心"由传统监督式 ML 承担。

关键判定：CRUD/报表骨架确实仍照常运行，但无 AI 填充后是无智能内容的空壳（重点名单无人自动识别、工单无 AI 起草、报表无预测洞察）——这恰是原生信号：产品价值=智能而非 CRUD。传统产品在设计自身的痛点陈述中已被判定"滞后失效"，故不存在"工作良好的传统产品+AI 助手"的赋能伪装结构。综合：广义移除通过原生性测试；狭义移除暴露 LLM 部分外周化的实情——核心预测靠传统 ML，LLM 承重位需前移。故判 truly_native，但为"条件式原生"：架构原生成立，合规与评测完整性需修正后方可部署。

**反模式风险**：(1) AI洗/术语通胀——把"多源数据管道+集成ML"包装成"多模态风险综合画像引擎"——"多模态"在AI原生语境通常指视觉/音频/文本跨模态基础模型，此处实为multi-source多数据源，非multimodal；知识图谱对齐引用Squirrel AI纳米级知识点拆解属名称借用。(2) LLM外周化（核心智能非LLM）——决定设计引以为傲的"提前3周预测/85%准确率"的核心是传统监督式ML，LLM承重的主要是沟通草稿生成与LLM-as-judge评判，属增值层而非价值核心。(3) AI当if-else风险——MTSS四级分层与agent Plan步"按MTSS分层选择工具组合"若实现为"风险分阈值→分层→工具"的规则查表。(4) 合规自相矛盾（一·数据出境）——自杀意念检测同时列了"审核过的专用BERT"与"OpenAI Moderation API"，又声称"敏感数据不出校走边缘模型"，OpenAI Moderation是云端调用即把学生文本出境，与"不出校"直接冲突。(5) 合规自相矛盾（二·文本内容vs行为数据）——声称"数据最小化、只采集行为数据、禁用人口统计变量"，但同时做NLP文本分析（课程讨论帖/作业/求助台工单的情绪与自杀意念标记）——文本内容分析≠行为数据。(6) 忽视公平性/偏差evals——教育风险评分系统在Annex III下属高风险AI，强制要求偏差检测与缓解。(7) 生命安全级评测缺位——自杀意念检测的假阴性门槛未明确指定召回率/假阴率阈值与人工兜底复核流程。(8) 数据飞轮因果归因薄弱——飞轮依赖"干预后2-4周风险分变化=干预效果"作正负样本，但风险分变化可能由诸多混淆因素导致。(9) 生成式UI过度宣称。(10) 归因机制冗余/真源不明。

**修正后的设计要点**：(1) LLM承重位前移——把根因归因从"SHAP为主、LLM复杂推理为辅"改为LLM主导的因果叙事归因（SHAP特征作为LLM输入证据），使LLM位于核心推理路径而非外周生成层；agent Plan步的工具选择明确由LLM推理驱动而非规则查表。狭义LLM移除回归测试须验证此时核心归因退化。(2) 消除合规矛盾一（数据出境）——自杀意念检测二选一：要么全自建/自托管BERT（边缘部署数据不出校符合PIPL），要么在取得法定前置条件下用Moderation API；不得并列宣称二者可互换。(3) 消除合规矛盾二（文本内容处理）——明确NLP文本分析的lawful basis，对讨论帖/作业/求助台内容做情绪与自杀意念挖掘属敏感个人信息处理，须PIPL单独同意+PIA+数据最小化（仅保留情绪标记与风险信号embedding，原文不留存或不进入训练），并明确告知学生；将"只采集行为数据"措辞统一修正为"采集行为数据与受限文本信号，禁用人口统计变量"。(4) 补齐公平性evals——新增偏差审计维度，分群准确率/假阳假阴率差异监控、Equalized Odds评测、定期偏差报告；纳入EU AI Act Annex III高风险AI的偏差缓解义务。(5) 生命安全级评测与兜底——自杀意念检测设定硬性召回率下限（如≥0.95）与假阴率上限，所有模型判定为"高风险/疑似自杀意念"的案例强制人工复核兜底，并接入校园心理危机应急通道。(6) 飞轮因果归因强化——引入反事实/倾向分匹配或ITS中断时序分析来归因干预效果，区分"干预致效"与"自然波动"，知识库只沉淀通过因果检验的策略-学生类型映射。(7) 术语去通胀——将"多模态风险综合画像引擎"正名为"多源风险融合与归因引擎"。(8) 生成式UI落实或降阶。(9) 归因真源统一——明确SHAP/LIME提供特征级证据，LLM基于SHAP+历史案例生成因果叙事归因，二者为"证据→叙事"链路而非并列，冲突时以SHAP数值为锚、LLM不得编造未被证据支持的归因（guardrails约束）。(10) 狭义移除可观测——建立LLM-only移除的回归测试（保留ML、抽掉LLM），监控核心风险预测不退化、仅沟通/评判层降级。

**技术实现**：模型选型风险分类/归因用随机森林（100树深度10）+双层LSTM（64/32单元）时序预测（CSDN范式准确率约85%）；NLP文本情绪/自杀意念检测用BERT微调模型（**自托管边缘部署数据不出校，解决合规矛盾一**）；根因可解释性用SHAP/LIME（特征级证据）；个性化沟通草稿生成用Claude Haiku/GPT-4o-mini（多模型路由）；LLM-as-judge用GPT-4评估草稿质量与归因合理性。检索pgvector起步（每租户命名空间隔离），百万级向量迁Milvus/Qdrant；DPR双编码器检索历史相似案例与干预策略知识库。Agent框架自建轻量agent loop，5个ACI工具（生成沟通草稿/创建干预工单/触达学生/转介/文本复核），高风险动作服务端二次权限校验；后台队列Inngest/Trigger.dev做每日批量风险扫描与embedding。数据层四源数据RESTful API/Webhook摄入，统一数据层向量化存储，风险画像对齐知识图谱；每30天重训仅准确率提升才更新。基础设施云-边-端三层——云端做大模型推理与批量重训，边缘（校内服务器）做实时风险扫描与向量检索（低延迟+数据境内化），端侧（辅导员APP/学生端）做轻量展示；路由层按风险等级与数据敏感度选推理路径（敏感数据不出校走边缘模型）。可观测性LangSmith/Langfuse trace，成本分解到embedding/推理/eval/observability六行，50-100条golden样例做CI/CD门禁。合规基础设施境内存储默认；未成年人数据单独同意流；离校自动删除定时任务；人工复核工作流引擎（心理转介/家长沟通工单强制human-in-the-loop签字）；SHAP归因附在每条预警供审计。

### 5.9 报表与决策分析

**传统形态与痛点**：固定模板统计报表（学生人数/性别比/年级分布/出勤率/及格率/优秀率等预置报表，导出 Excel/PDF）、领导决策汇总报表（定期生成跨部门跨学年对比表）、静态数据看板（预配置KPI大屏，固定图表与固定下钻路径，手动刷新）、趋势分析（同比/环比折线图+简单线性趋势线，需人工对比与解读）、预警阈值报表（基于固定阈值触发的风险名单报表，人工逐条排查）、报表订阅与定时推送、自定义查询取数（领导提需求→分析师写SQL临时取数→手工制表，周期以天计）。痛点："看数据"而非"懂业务"——静态报表只呈现数字，需人工跨多张表交叉对比、靠经验主观归因才能发现问题和原因，决策严重滞后；固定模板无法回答即席提问；描述性而非预测/处方性；滞后指标——报表生成时问题已发生，错过干预窗口（阈值法比AI晚约3周识别风险，干预成功率从80%骤降到20%）；千人一面；数据孤岛；看板堆砌几十个指标却不突出高信号；趋势分析依赖个人经验，隐性拐点常被漏判，归因主观且因人而异。

**AI 原生改造设计**：

- **core_ai_capability**：自然语言→洞察→行动的生成式分析（generative analytics）：意图理解+多源检索+推理归因+预测+处方生成，模型是核心路由——把领导"问什么"翻译成"查什么、怎么算、怎么解释、该做什么"。移除该能力，模块退化为静态报表导出空壳。
- **redesign**：(1) 报表→生成式叙述报告——取消固定模板为核心，改为按意图动态生成。领导自然语言提问（"本学期哪些班级数学下滑最明显，可能原因是什么"），系统自动跨表跨库检索（成绩+考勤+校园卡+LMS+教师请假+家校沟通）、计算、生成带数据、可视化、根因归因、影响范围、建议干预的叙述性报告；报表不预存，按意图组装（生成式 UI）。(2) 决策→主动巡检型分析agent——参考Brainbridge"晨间优先简报"，每日/每周主动生成"今日要关注"简报（3-5条高信号），每条含异常、根因假设、建议动作、影响范围；建议可一键派发为任务并跟踪闭环，对接EWS五步闭环。(3) 看板→意图驱动动态看板——看板按角色和当前关注焦点动态组装，主动推送高信号洞察而非堆砌指标；静态KPI大屏降级为兜底视图。(4) 趋势→预测+归因——从同比环比折线升级为基于历史数据训练的时序预测模型预测下月/期末挂科率、留存率、各班排名走势，并给出"为什么"的因果归因（SHAP/LIME）和"做什么"的处方。(5) 多模型路由——简单聚合统计走廉价模型，复杂归因与叙述走高端推理模型；语义缓存降本30-50%。(6) 安全——输出经文本蕴含幻觉检测+Moderation双向过滤+低置信度降级为"我不确定"，高危动作（派发任务）服务端权限二次校验不直接落库。
- **用户体验**：校长早上登录即看到AI生成的"今日要关注"简报（3-5条），每条含异常+根因假设+建议动作+影响范围，可一键展开为完整报告或派发任务，无需翻多个报表找问题。班主任问"我班这学期哪些科目退步了"，得到带数据、图表、原因分析和补救建议的叙述报告，而非自己导Excel算。教务问"下学期哪个年级教室资源最紧张"，得到预测+排课建议。所有人能追问、能纠错（标记AI解读不准进入反馈飞轮）、能撤销派发的任务——交付的是"洞察报告+建议"而非"查询工具"。
- **agent 循环**：分析型agent，感知→规划→工具→反思循环：感知（定时每日/每周+事件驱动数据更新触发，扫描全量学情；接收用户NL提问）→规划（意图分类：描述/诊断/预测/处方；拆解子查询与计算步骤；按复杂度选检索范式——简单聚合走语义层/SQL，复杂归因走RAG+推理，预测走时序模型；按复杂度路由模型）→工具调用（query_warehouse 文本转SQL经语义模型防幻觉、semantic_search跨源向量检索、run_forecast时序预测、explain_attribution SHAP/LIME归因、generate_chart图表、dispatch_action派发任务）→反思（数字一致性自检、因果是否站得住脚、幻觉检测；低置信标注并降级为"我不确定"+请求人工；记录用户纠错）。模型在请求路径上为核心路由，归因与建议为概率性逻辑而非确定性规则。
- **数据飞轮**：交互→数据→改进→更好体验闭环：交互（用户提问、追问、纠错、采纳/拒绝建议）记录为带标注的query-insight-action三元组；数据沉淀为"问过的好问题+采纳的解读+验证有效的干预"golden set；改进用golden set做eval（LLM-as-judge评估解读质量、用干预结果评估建议有效性）、微调检索与路由、训练更好的预测模型；更好体验使系统越来越准地预判"领导想问什么"并主动推送（从copilot走向autopilot）。闭环校验：派发的干预建议被跟踪结果，干预有效则强化该模式、无效则修正，形成"检测→建议→行动→结果→改进"闭环。学情数据+干预结果反馈构成专有数据壁垒。

**移除测试结果（设计自判）**：抽掉AI后报表退化为固定模板导出，无法回答即席提问、无法跨表归因；决策退化为人工翻多份报表自己找问题，无主动预警、无根因、无建议；看板退化为静态KPI大屏，只有"发生了什么"没有"为什么和怎么办"；趋势退化为同比环比折线图，无预测无归因。核心价值（洞察、归因、预测、行动建议）全部消失，只剩数据展示空壳。设计自判 native。

**应用的标准**：架构标准（模型在请求路径上、harness governs、概率性逻辑为核心归因与干预建议是概率推断而非if-else规则）；数据与知识标准（飞轮闭环、上下文工程跨源聚合后裁剪为最小高信号token集、检索按复杂度、专有数据壁垒学情数据+干预结果反馈）；Agent 系统标准（agentic loop、简洁性优先SQL能解决就不用LLM推理、ACI工具接口工程化、规划模式按任务选）；产品设计标准（意图驱动生成式 UI、copilot vs autopilot 分级人提问=copilot AI主动推送异常=autopilot待审批、可纠错可撤销、结果导向）；业务流程标准（AI在价值交付关键路径、人机协同、headcount decoupling）；评估与可观测性标准；成本与经济模型标准；安全与对齐标准（LLM06派发任务需权限校验、LLM03供应链、prompt injection防护NL提问清洗、guardrails输出经Moderation双向过滤+文本蕴含防幻觉）；多模态标准（生成的图表/PDF报告外置artifact）；边缘与端侧标准（云-边-端三层、路由层最高杠杆）；合规标准（PIPL未成年人敏感信息单独同意与最小化、GDPR Art22自动化决策保留人工复核与申诉、AI Act Annex III教育水平评估属高风险需可解释与人工监督、可解释性SHAP/LIME归因）；组织与团队标准。

**对抗核验结论：truly_native。** 诚实呈现残留风险与修正。

重做移除测试（今晚移除所有AI调用，逐层推演核心流程是否崩溃）：入口层——NL意图理解消失→用户无法用自然语言提问；意图路由消失→系统无法决定"查什么/算什么/生成什么/该做什么"。入口从"问什么"退化为必须从固定菜单选。报表层——生成式叙述报告消失——跨表跨库自动检索、根因归因、影响范围、干预建议全部消失；退化为"静态KPI兜底视图"，只能看预置指标，无法回答即席提问。决策层——晨间主动巡检简报消失——无主动预警、无根因假设、无建议动作、无影响范围、无派发任务闭环（dispatch_action断链）；决策退化为人工翻多份兜底报表自己找问题。看板层——意图驱动动态看板消失，退化为固定KPI大屏。趋势层——时序预测消失；SHAP/LIME因果归因消失；退化为同比环比折线+简单线性趋势线。预警层——阈值法回归，比AI预测晚约3周识别风险，干预成功率从80%骤降到20%。行动层——EWS五步闭环断链。飞轮层——query-insight-action三元组不再沉淀，golden set停止增长，LLM-as-judge eval、检索/路由微调、预测模型重训练全部停滞。

结论：模块"支持决策与驱动行动"这一核心价值交付路径全面崩溃，仅剩数据仓库基础设施（StarRocks/Doris语义层，属平台团队水平工作）+静态KPI兜底视图。AI明确在价值交付关键路径上、模型在请求路径上为核心路由、归因与建议为概率性逻辑而非确定性规则→移除测试通过。残留保留点辨析：数据仓库+静态KPI兜底仍可运行，但设计明确定位其为degraded fallback（非核心、非默认），且"可纠错可撤销/优雅降级"本就是产品设计标准要求。保留兜底不构成bolt-on信号。唯一须警惕：若实践中用户默认使用兜底、AI路径不被采纳与纠错，飞轮停转，则会实际退化为赋能伪装。

**反模式风险**：(1) 兜底看板默认化风险——静态KPI兜底视图若过于完整舒适，老领导会习惯性回到熟悉大屏，AI生成式路径不被采纳、不被纠错，数据飞轮停转——实践中退化为"AI赋能伪装"（数据仓库+看板照常跑，AI沦为可选助手），移除测试在实践层面失效。(2) text-to-SQL静默错误风险——NL→SQL经语义层仍可能生成错误SQL，产出看似合理实为错误的洞察；若golden set只评叙述质量不评SQL正确性，幻觉数字会静默通过且难被发现。(3) autopilot规则硬化风险（AI当if-else）——晨间简报预警逻辑若不再依赖重训练的预测模型、退回固定阈值，AI沦为if-else硬塞反模式，且丧失"比阈值早3周"的核心优势。(4) eval覆盖与分层不足风险——50-100条golden样例对描述/诊断/预测/处方四类意图可能不够分层覆盖；缺SQL正确性eval、缺归因有效性eval。(5) 归因不可复现风险——SHAP/LIME仅给局部特征解释，整条跨源归因链若不存证SQL/检索片段/模型版本/trace，重蹈传统"人工拼凑且不可复现"覆辙。(6) 未成年人/AI Act合规深度不足风险——AI Act Annex III(4)教育水平评估属高风险，需FRIA基本权利影响评估与高风险系统注册，设计仅提"可解释与人工监督"。(7) 成本边界漂移风险——Opus 4.5（约10%战略决策）若被高频触发，cost-per-query失去可辩护性；语义缓存命中率需在线监控。

**修正后的设计要点**：(1) 兜底视图降权——静态KPI兜底显式定位为"原始数据视图"并访问受限（非默认首页），AI生成式简报/报告为唯一默认入口；确保飞轮持续转动，防止实践退化为赋能伪装。(2) eval分层与SQL正确性——golden set按描述/诊断/预测/处方四类意图分层扩充，新增text-to-SQL正确性eval（执行结果比对）与归因有效性eval（干预结果回灌评分），作为CI/CD部署硬门禁。(3) 归因链可复现存证——每份生成报告挂载provenance（执行SQL、检索片段、模型版本、SHAP值、trace id），支持"同问题重放"——直接回应传统"人工拼凑且不可复现"痛点。(4) 预测模型防漂移——预测模型按周期重训练+数据漂移检测触发重训；预警逻辑必须保持模型驱动，禁止退化为固定阈值if-else（守护核心优势）。(5) 未成年人/高风险合规落地——EU部署执行AI Act Annex III(4) FRIA+高风险注册+人工监督日志；PIPL未满14岁走监护人同意+最小化；GDPR Art22自动化决策保留人工复核与申诉通道并记审计。(6) 成本在线护栏——对Opus战略路由设频次上限与预算告警；语义缓存命中率/缓存失效纳入可观测面板，cost-per-query按意图类型对外可辩护（六行成本分解：embedding检索/路由推理/预测模型/eval/observability/语义缓存）。

**技术实现**：模型选型意图理解/路由/简单聚合解读用Claude Haiku 4.5（廉价高频约60%查询）；复杂归因与叙述报告用Claude Sonnet 4.6（约30%）；深度战略决策推理可选Opus 4.5（约10%）。预测模型挂科率/留存率/排名走势用LightGBM或时序模型Prophet/DeepAR训练（参考Carnegie APLSE年终进度预测）。检索text-to-SQL经语义层（预定义指标口径的语义模型防幻觉SQL）查数据仓库；跨系统非结构化用pgvector起步、规模上来迁Milvus做DPR双编码器语义检索（参考Jill Watson RAG管线，段落≥500字符50%重叠、top-20分批检索）。agent框架自建轻量agent loop或LangGraph编排，工具集query_warehouse/semantic_search/run_forecast/explain_attribution(SHAP)/generate_chart/dispatch_action，工具调用受服务端权限二次校验。数据层StarRocks/Doris做指标预聚合数仓+pgvector/Milvus向量库+时序预测模型独立服务化。基础设施语义缓存（embedding相似度>0.95命中降本30-50%）、多模型路由层、CI/CD内建eval门禁（50-100条golden样例自动化测试，AI质量作部署门禁）、LangSmith/自建trace。合规境内存储，跨境模型调用走网关脱敏（PIPL Art38）；未成年人数据查询走单独同意与最小化（PIPL Art31）；自动化决策建议保留人工复核与申诉通道（GDPR Art22、AI Act Annex III）；归因可解释用SHAP/LIME。改造路径用Strangler Fig增量替换（上游代理拦截，AI分析服务与旧报表并行，失败回退legacy），严禁Big Bang一次性切换。

### 5.10 缴费与事务办理

**传统形态与痛点**：【学费缴纳】缴费项配置、按学年生成缴费账单、对接支付网关、缴费记录查询、人工群发欠费催缴短信/邮件、退费手工录单、财政票据开具、欠费名单报表。【奖助学金】奖助项目配置、学生填申请表+上传证明材料、资助中心逐份人工看材料核资格、评审委员会打分、名额人工分配、发放记录登记、公示。【事务申请与审批】申请类型表单（请假/休学复学/转专业/在读证明/宿舍调换/困难认定等）、固定多级审批流配置、逐节点人工流转、催办、归档。【报表】缴费统计、奖助发放统计、事务办理时效统计。痛点：催缴靠群发短信，学生忽略/错过截止，欠费累积影响注册与选课；经济困难学生真实诉求淹没在群发里无人识别；奖助评审是人海战术——资格条件散落在几十份红头文件里（"资格条件活在非结构化文本中"），评审人凭记忆+逐份翻文件判资格，遗漏/不公平/低效；家庭经济困难认定主观，名额分配靠拍脑袋；事务审批表单字段死板，学生填错反复退回；简单事务（开在读证明）也要走完整审批链；跨系统靠人肉搬运；学生侧信息不对称；合规盲区——财务数据敏感，奖助名额/退费是影响学生重大权益的自动化决策，传统系统无人工复核门槛与申诉通道意识。

**AI 原生改造设计**：

- **core_ai_capability**：LLM作为需求路由与资格判定的核心推理引擎：把学生用自然语言表达的诉求+上传的任意非结构化材料，直接转化为结构化申请+资格证据链+带置信度的审批建议，并按风险等级在autopilot/copilot/人工三级间动态路由。移除它后系统只剩表单填写与人工逐份审批。**注意边界**：资金划拨、账务记账、票据开具、最终授权决策保留为确定性服务与人工，AI心脏在"需求理解+资格判定+个性化助推+风险预判+审批路由"这一智能编排层。
- **redesign**：①学费缴纳→AI主动缴费管家（proactive payment copilot）。不再等学生登录查账单：agent感知缴费状态（来自SIS/财务）+学生画像（经济压力指数、历史缴费行为、是否已申请助学贷款），主动生成个性化缴费方案——分期/绿色通道缓缴/助学贷款对接/临时困难补助申请；催缴从群发短信升级为Mainstay式行为助推对话，按学生情况生成"为什么缴/怎么缴/截止/可选项"的个性化消息，对话中一旦出现经济困难信号即路由资助中心人工跟进；聚合洞察预判欠费辍学风险群体。退费/票据：材料上传后agent抽取关键信息生成退费申请草稿供人工复核。②奖助学金→资格匹配与评审助手。学生侧：学生无需在几百个项目里找资格，agent基于档案（学业/经济/特长/民族/生源地）自动匹配可申请项目，生成"你符合这5项、这2项还差X条件"清单+一键发起申请——直接利用"资格条件活在非结构化文本中"这一痛点，LLM把红头文件条款解析为结构化校验规则入库。评审侧：申请材料由IDP+LLM抽取关键事实+交叉核验（学籍/成绩/经济认定库）→生成带置信度的资格判定建议+风险标记（材料矛盾/疑似造假）；ML经济困难度评分辅助名额分配；评审委员会人在环上复核——AI只给"建议+置信度+依据"，不直接落库生效（反LLM-as-authorization防护）。③事务申请与审批→意图驱动办理agent。学生不再从表单列表找类型，用自然语言/语音说"我要请三天病假""开在读证明""转专业""休学一年"→agent意图分类→自动选申请类型→智能预填表单（从学籍档案拉已知字段）→只问缺的字段→生成申请；审批路由按事务类型+风险等级动态选链（在读证明→自动直批；请假→辅导员一签；休学/退学/转专业→多级+家长确认）；LLM-as-Orchestrator把审批任务拆解为对财务/学籍/宿舍等异构系统的操作序列；上传证明用IDP抽取核验生成证据摘要给审批人；界面按意图动态生成（生成式UI）而非固定表单列表。
- **用户体验**：学生不登录找表单，一句话说诉求，agent预填、只问缺的字段、给缴费方案/可申请奖助清单、主动提醒截止。例：学生说"家里最近出事交不起学费"→agent识别经济困难信号→主动推荐缓缴/助学贷款/困难补助并路由资助中心人工跟进，而非冷冰冰催缴。教师/辅导员不再逐份看材料，审批台显示agent生成的"资格证据链+置信度+风险标记"，只复核高风险/低置信度案例，简单事务自动通过，省出时间做高接触advising。管理员不再手写催缴短信、不手动分名额，看聚合洞察仪表盘（欠费风险学生、奖助匹配缺口、事务办理瓶颈），配策略后agent执行。家长：涉及未成年人重大事务（休学/退学/大额缴费）触发家长确认通道（PIPL Art31合规）。
- **agent 循环**：感知（接收学生自然语言诉求+上下文快照学籍/缴费/奖助/学业状态最小高信号token集非全量档案+上传材料二进制外置artifact）。规划（意图分类缴费/奖助/事务→子任务分解→检索范式按复杂度动态选择：简单资格匹配用规则/关键词检索；复杂材料理解用RAG+LLM；经济困难度/辍学风险用ML结构化模型→copilot/autopilot分级：低风险事务autopilot直批；奖助名额等影响重大权益copilot必经人工复核）。工具调用（资格规则检索器向量库top-k、IDP材料抽取器PaddleOCR/MinerU、交叉核验器查学籍/成绩/经济认定库、缴费方案生成器、审批路由器、对接SIS/财务/宿舍的API或RPA、催缴消息生成器、票据生成器）。反思（置信度评估材料矛盾/缺失→降置信度→转人工；幻觉检测文本蕴含校验Jill Watson模式；服务端二次权限校验LLM输出不可直接授权反OWASP LLM06；AI服务失败流量回退legacy表单Strangler Fig）。
- **数据飞轮**：每次交互沉淀为标注数据：学生诉求表述（意图识别训练）→上传材料+抽取结果（材料抽取模型微调）→资格判定建议 vs 评审委员会最终决策（采纳/修改/否决，这是关键人类反馈信号，Duolingo Max报告→训练→提升模式）→催缴话术A/B（哪种助推最有效）→缴费/辍学结局（欠费辍学风险模型重训）。闭环：交互→数据→规则库越滚越准（LLM把新文件条款解析为结构化规则）+模型微调→更准匹配/更少人工/更低欠费辍学→更好体验→更多交互。专有数据壁垒=历年评审决策库+资格条款结构化库+催缴行为响应库，竞争对手无法短期复制。

**移除测试结果（设计自判）**：抽掉AI后退化为：固定表单列表+学生自己从几百个奖助项目里找资格+人工逐份看材料判资格+群发催缴短信+简单事务也走完整审批链+跨系统人肉搬运。意图驱动办理、资格智能匹配、个性化助推、欠费辍学风险预判、动态审批路由——这五项构成模块差异化价值的全部能力消失，系统只剩CRUD表壳。诚实地，资金划拨/记账/票据/最终授权决策保留为确定性服务与人工，不依赖AI；但模块的核心智能编排层完全由AI构成且不可降级替换。设计自判 native。

**应用的标准**：架构标准（Intelligence-First学生一句话进入即过LLM意图分类+资格匹配非外挂聊天框、harness governs、概率性逻辑为核心资格判定给置信度而非布尔通过）；数据与知识标准（飞轮评审决策反馈→资格规则库+模型重训、上下文工程学籍/缴费/奖助快照最小高信号token集、检索按复杂度规则/RAG/ML三档、专有数据壁垒历年评审决策库+资格条款结构化库）；Agent 系统标准（agentic loop感知→规划→工具→反思四步显式、简洁性优先简单事务不进agent直批避免过度代理、ACI工具接口工程化、规划模式按任务选低风险autopilot/高风险copilot）；产品设计标准（意图驱动生成式 UI按意图动态渲染而非固定表单列表、copilot vs autopilot奖助名额copilot必经人工/在读证明autopilot、可纠错可撤销、结果导向交付"已办理/已匹配资格/已生成缴费方案"而非给工具）；业务流程标准（AI在价值交付关键路径资格判定/缴费方案/审批路由是核心价值而非边角自动化、人机协同in-loop高风险复核/on-loop低风险监督/out-of-loop确定性支付执行、headcount decoupling事务量增长不再线性增辅导员）；评估与可观测性标准；成本与经济模型标准；安全与对齐标准（LLM06高风险动作人工复核门槛+服务端二次权限校验禁止LLM输出直接落库授权、LLM03供应链第三方支付/票据供应商准入审计、prompt injection学生材料/说明里的注入隔离、输出Moderation guardrails）；多模态标准（上传的病假条/困难证明/票据等二进制外置为artifact按需加载、稳定前缀缓存资格条款系统提示复用）；边缘与端侧标准（云-边-端三层端侧轻量意图识别/催缴消息本地生成云端重推理/ML评分、路由层最高杠杆按风险/复杂度路由autopilot/copilot/人工三档）；合规标准（财务数据敏感信息PIPL单独同意+PIA、自动化决策影响重大权益GDPR Art22强制人工干预+申诉权PIPL Art24提供拒绝自动化决策选项奖助名额/退费绝不纯自动落库、未成年人数据最小化PIPL Art31家长同意+专门规则休学/退学触发家长确认通道、可解释性AI审批建议附资格依据呼应AI Act Annex III高风险教育系统数据治理/人工监督/日志义务）；组织与团队标准。

**对抗核验结论：truly_native。** 诚实呈现残留风险与修正。

逐子模块重做"今晚移除所有AI调用"推演：【学费缴纳】确定性后端——缴费项配置/按学年生成账单/对接支付网关/缴费记录查询/退费手工录单/财政票据开具/欠费名单报表——全部照常运行（设计明确保留为确定性服务，用于合规，非AI依赖证据）。但智能编排层消失：AI主动缴费管家不再主动感知缴费状态+学生画像、不再生成个性化缴费方案、Mainstay式行为助推对话退回群发催缴短信、经济困难信号识别与路由人工消失、欠费辍学风险预判消失。关键：学生侧缴费引导主入口从"AI主动触达"退回"学生登录查账单"——前端入口坍塌。【奖助学金】确定性后端——项目配置/学生填表上传/人工逐份看材料/评审打分/名额分配/发放登记/公示——全部照常。但智能层消失且不可降级替换：资格匹配消失（红头文件条款活在非结构化文本中，无LLM无法规模化结构化为机器可校验规则，学生须在几百个项目里自己读文件找资格）；IDP+LLM材料抽取+交叉核验退回人海战术；ML经济困难度评分消失。此项是模块最强native证据——无LLM则该能力在规模化场景下根本不存在，非"降级"而是"能力消失"。【事务申请与审批】确定性后端——申请类型表单/固定多级审批流/催办/归档——全部照常。但智能层消失：意图驱动办理消失、智能预填消失、动态审批路由退回固定链、LLM-as-Orchestrator跨财务/学籍/宿舍系统编排退回人肉搬运、生成式UI退回固定表单。

关键判定：三个子模块的"执行后端"（支付/记账/票据/授权/发放）移除AI后照常——这是设计主动保留的确定性层（GDPR Art22/PIPL Art24禁止纯自动化重大权益决策），不构成bolt-on证据。但"智能编排前端"移除AI后完全消失，且奖助资格匹配在非结构化条款场景下无LLM无法规模化（不可降级替换）。学生侧主入口从"自然语言意图驱动生成式UI"退回"固定表单列表"——属前端入口坍塌而非单纯体验降级。设计自承"系统只剩CRUD表壳"——CRUD表壳是确定性后端照常运行，但模块差异化价值（5项智能能力）全部消失。符合native：AI在请求路径最前端（Intelligence-First），移除则核心价值交付路径坍塌。

**反模式风险**：(1) 支付子模块赋能伪装残留风险——支付执行本身确定性（合规正确），但设计未明确缴费管家是否为缴费引导的唯一主入口。若学生可不经过AI缴费管家直接走传统账单页缴费并获取所有选项，则AI缴费管家退化为可选增强而非承重路径，构成bolt_on。(2) Strangler Fig回退歧义风险——设计写"AI失败流量回退legacy表单"，未界定legacy为迁移期失败-only降级还是永久对等选项。若legacy表单作为用户可选对等路径永久保留，则AI非承重，套壳风险成立。(3) AI当if-else风险——意图分类→自动选申请类型若仅做intent→form_id映射后渲染固定表单，则退化为glorified if-else。(4) mega-prompt风险。(5) 高风险决策evals不足风险——奖助名额/退费属影响学生重大权益的自动化决策，50-100条golden样例对高风险场景偏少，且缺对抗性测试。(6) 未成年人数据飞轮合规风险——飞轮用学生交互数据训练模型，未成年人交互数据进入训练若未单独同意/脱敏，违反数据最小化。(7) LLM-as-authorization橡皮图章风险——评审委员会对AI资格判定建议若仅确认不独立核验事实，则de facto自动化决策。(8) 成本优先而非风险优先的模型路由风险——60%流量用低端模型，但合规关键的资格判定若用低端模型，准确率不足直接影响重大权益。(9) 上传材料prompt injection风险——学生上传的材料含注入向量，若原始文本直接入LLM prompt而非经IDP抽取为结构化数据隔离，构成OWASP LLM01注入面。(10) 套壳聊天框风险（已部分缓解）。

**修正后的设计要点**：(1) 缴费管家承重化——将AI缴费管家设为缴费引导的唯一主入口（个性化方案/缓缴/助学贷款对接/困难补助路由/经济困难信号识别必须经AI），传统账单页仅作AI失败降级，使学生无法绕过AI获取个性化缴费方案。否则支付子模块停留在赋能伪装。(2) Strangler Fig sunset——legacy表单明确为迁移期失败-only降级，设sunset日期（迁移完成后下线），禁止作为用户可选对等路径永久保留。否则AI非承重。(3) 字段级生成式UI强制规范——表单字段按意图+学籍上下文动态选择（只问缺的字段，从档案拉已知字段），而非intent→form_id→固定表单渲染。须有字段级动态生成验证用例。(4) 风险感知模型路由——合规关键资格判定（奖助资格/退费资格/困难认定）强制用高端模型（Claude Sonnet/GPT-4），低端模型仅用于意图分类/字段抽取等低风险任务。成本服从风险，非反之。(5) 高风险evals强化——奖助资格/退费evals扩至200-500条golden样例+对抗性测试（材料矛盾/疑似造假/边缘案例/资格条款冲突）+人工专家复核eval结果，作CI/CD部署门禁。(6) 意义性人工复核护栏——评审委员会须独立核验至少一个事实（非仅确认AI建议），追踪AI协议率，过高（橡皮图章>90%）或过低（模型退化<50%）触发告警与调查。(7) 未成年人数据飞轮护栏——未成年人交互数据默认排除出模型训练，或须单独家长同意+脱敏后入飞轮，满足PIPL Art31数据最小化。飞轮须有未成年人数据隔离机制。(8) agentic loop真实性验证——LangGraph状态机为多节点（感知/规划/工具调用/反思）多工具调用，反思节点做置信度评估+幻觉检测（文本蕴含）+服务端二次权限校验，非单mega-prompt包装。(9) 高风险可解释性——AI资格判定附人类可读依据（匹配的资格条款原文+所用证据+置信度+风险标记），非仅置信度数值，满足AI Act Annex III高风险教育系统数据治理/人工监督/日志义务。(10) 材料 prompt injection防护——上传材料经IDP（PaddleOCR/MinerU）抽取为结构化数据后入核验流程，原始文本不直接入LLM prompt，隔离注入面（OWASP LLM01）。

**技术实现**：模型选型（多模型路由）意图分类/字段抽取用Qwen2.5-7B-Instruct或GPT-4o-mini（占60%流量）；材料理解/RAG答案生成用Claude Haiku 4.5/Qwen-Plus（30%）；复杂评审多步推理用Claude Sonnet 4.6/GPT-4（10%）。结构化风险模型XGBoost/随机森林做经济困难度评分与欠费辍学风险（特征=缴费行为+学业参与度+校园卡消费规律性，权重按cohort学习校准）。检索资格条款+历史评审决策用bge-m3向量化入Milvus（>1M向量）或起步pgvector，DPR式top-k检索；文本蕴含模型做幻觉检测（Jill Watson模式）。agent框架LangGraph做agentic loop状态机；工具层含资格规则检索器、IDP材料抽取器（PaddleOCR+MinerU解析扫描件/PDF）、交叉核验器（对接SIS/成绩/经济认定库REST API）、审批路由器、催缴消息生成器、票据生成器；置信度<阈值或高风险标记自动触发人工复核。数据层统一数据层+SIS/财务/宿舍数据经embedding流水线入向量库；per-tenant（多学院）命名空间隔离。基础设施语义缓存（相似度>0.95命中）、后台队列（Inngest/Trigger.dev）做批量材料embedding与奖助评审、provider故障转移、流式响应（time-to-first-token优先）。确定性执行层支付网关、账务记账、财政票据开具保留传统确定性服务，AI编排后调用。改造路径Strangler Fig增量替换（先接缴费催缴+事务预填低危场景，再接奖助评审），AI失败流量回退legacy表单（设sunset日期非永久对等）。evals 50-100条golden资格判定样例作CI/CD部署门禁（高风险场景扩至200-500+对抗样本）；LLM-as-judge评审建议质量；trace每笔事务成本/延迟/置信度/人工干预率。合规基础设施境内存储（PIPL Art38/40本地化）、未成年人单独同意流程（PIPL Art31）、自动化决策人工复核与申诉通道（GDPR Art22）、离校数据留存期与自动删除（PIPL Art47）。

## 七、数据飞轮与隐私合规工程化（未成年人场景硬约束）

教育场景的硬约束：未成年人数据最小化、本地化、同意管理、可解释、删除权。这是飞轮设计的红线，飞轮须在合规边界内运行而非反之。

### 7.1 未成年人数据最小化（PIPL Art31 + 未成年人保护法）
- **<14 岁数据定为敏感信息**：家长单独同意+专门规则+PIA+单独同意流程。
- **按需展开原则**（调和细粒度建模与最小化冲突）：仅诊断/干预所需知识点节点才实例化，非全量 10000+ 节点常驻；掌握度轨迹按需展开。
- **禁用人口统计变量**：风险模型只用行为数据（学生在做什么）+受限文本信号，禁用种族/收入等（Brainbridge 原则）。
- **知识状态向量作为敏感心理画像**：单独同意+PIA；定义掌握度轨迹留存/删除/模型遗忘（被遗忘权）。
- **未成年人飞轮数据默认排除训练**，或脱敏后入飞轮；飞轮须有未成年人数据隔离机制（修正缴费/学籍/排课模块反模式）。

### 7.2 本地化（PIPL Art38/40）
- **境内存储默认**：阿里云/华为云/腾讯云境内区域。
- **敏感数据不出校**：自杀意念检测全自建 BERT 边缘部署（解决学业预警模块合规矛盾一，不调云端 Moderation）；边缘 CV 原始视频本地丢弃只传 embedding。
- **跨境模型调用**：走网关脱敏+数据出境安全评估+标准合同；非必要不跨境。

### 7.3 同意管理（分层同意）
- **拆分三套同意**（修正沟通模块反模式）：(1) 同意 AI 处理并响应（基础同意）；(2) 同意数据用于模型训练（飞轮同意，可单独拒绝）；(3) 敏感数据（心理/危机/投诉）默认排除训练，需单独显式同意+DPIA。
- **未成年人重大事务家长确认通道**：休学/退学/大额缴费触发家长单独同意+增强告知+影响说明（PIPL Art31）。
- 同意状态可查询、可撤回、可导出。

### 7.4 可解释性（AI Act 附件 III 透明度义务）
- 每条风险评分附 SHAP/LIME top-3 归因特征。
- **归因单一真源**（修正学业预警模块）：SHAP 数值为锚→LLM 因果叙事，冲突时以 SHAP 证据为准，LLM 不得编造未被证据支持的归因。
- AI 答疑/诊断/资格判定引用文档页码/证据来源（Jill Watson 约束）。
- 高风险判定附人类可读依据（条款原文+证据+置信度+风险标记）。

### 7.5 删除权与留存期（PIPL Art47 / GDPR Art17）
- **离校自动删除/归档**：定时任务按留存期管控，学生离校触发数据清理流程。
- **模型遗忘**：被遗忘权延伸到模型层面，定义知识状态向量/画像的删除与模型遗忘机制。
- **文本原文不留存**：NLP 文本分析仅保留情绪标记与风险信号 embedding，原文不入训练不留存（修正学业预警模块合规矛盾二）。
- **课堂转写定期销毁**：作为敏感个人信息最小化留存。

### 7.6 自动化决策人工复核（GDPR Art22 / PIPL Art24 / AI Act 附件 III）
- **重大权益决策绝不纯自动化**：学位终审、奖助名额、退费、退学、心理转介、家长危机沟通须人工复核+申诉通道+审计日志。
- **意义性复核护栏**（修正缴费模块）：评审委员会须独立核验至少一个事实（非仅确认 AI 建议）；追踪 AI 协议率，过高（橡皮图章>90%）或过低（模型退化<50%）触发告警调查。
- **AI Act 附件 III 高风险义务**：数据治理+人工监督+日志+FRIA+偏差缓解（非仅透明度）+高风险系统注册（欧盟部署）。

### 7.7 飞轮因果化与防污染
- 干预效果用反事实/倾向分匹配/ITS 归因，区分"干预致效"与"自然波动"。
- 知识库 curation gate：教学法知识库/干预策略库新增须经教学专家人工审核或 LLM-as-judge+人工抽检双闸门，禁止 LLM 无监督自动写入（修正评教模块反模式）。
- 飞轮采纳信号去噪：按教师审核编辑距离区分"实质审核"与"橡皮图章"，仅前者入训练（修正沟通模块反模式）。

## 八、从传统到 AI 原生的迁移路径

### 8.1 演进原则
- **Strangler Fig 增量替换**：上游代理拦截，AI 服务与遗留并行，AI 失败回退 legacy，设 sunset 日期非永久对等路径（否则 AI 非承重，滑入赋能伪装）。
- **先建内核后建模块**：统一 AI 内核（学生长期记忆+共享风险引擎+agent 运行时+合规层）必须先就位，否则模块各建孤岛重蹈覆辙。
- **诚实分层**：确定性部分（求解器/工作流引擎/支付/票据/学位终审）保留并标注为非 AI；AI 只接管真正概率性部分（政策模糊/状态冲突/材料核验/风险感知/归因推理）。
- **量化"AI 不可替代 case 占比"**：目标>40%，否则该子流程降级 bolt-on（修正学籍模块反模式）。

### 8.2 演进顺序与风险

**Phase 0：平台基建（必须先建，6-9 个月）**——学生长期记忆骨架+学业状态语义层；共享 agent 运行时+工具层（ACI）+guardrails+evals 基建；模型 API 网关+云-边-端三层+异步队列；合规层（同意管理/境内化/删除权/人工复核工作流引擎）。风险：基建投入大、短期无业务可见价值；须以"通知沟通"模块作为基建首个消费方快速见效证明价值。

**Phase 1：原生确定性高、依赖弱、价值独立（3-6 个月，与 Phase 0 部分并行）**——通知与家校沟通（truly_native）：独立性强，飞轮快速启动，7×24 答疑与规模化个性化不可由堆人力替代，是证明 AI 原生价值的首选，须按修正项设 AI 为默认主路径。报表与决策分析（truly_native）：消费已有数据，不改写数据，风险低，须按修正项兜底视图降权。缴费与事务办理（truly_native，但高合规敏感）：奖助资格匹配是无 LLM 无法规模化的最强 native 证据，须按修正项缴费管家承重化+legacy sunset+风险感知路由+高风险 evals 强化+意义性复核护栏。

**Phase 2：风险引擎核心+信号源（6-9 个月）**——考勤与行为管理（truly_native）：边缘 CV 采集+行为信号源，为风险引擎供数，须按修正项透明化分层（ML 推理为 load-bearing，LLM agent 为 co-pilot）+fairness eval+NLP 情绪治理。学业预警与干预（truly_native，条件式）：共享风险融合引擎宿主，这是内核的核心，须在此阶段把"范围借用"的 EWS 提升为共享能力，须按修正项 LLM 承重位前移+消除合规矛盾一/二+公平性 evals+生命安全级评测兜底+飞轮因果化。依赖：Phase 0 内核就位+考勤信号源就位。

**Phase 3：深度改造模块（9-15 个月，高风险）**——成绩与学业评价管理（赋能伪装→修正）：数据模型倒置（作答事件流为唯一持久化，分数变投影）是深度改造，影响所有下游（长期记忆的掌握度向量在此产生），建议提前到 Phase 0.5 与内核共建。教师与教学评价管理（truly_native，条件修正）：依赖成绩掌握度+学业 delta，须按修正项修正 removal_test 自相矛盾（锁定分数非独立交付物）+未成年人合规+改进效果归因置信度+知识库 curation gate。依赖：成绩掌握度引擎就位。

**Phase 4：赋能伪装模块深度修正（12-18 个月，最高风险）**——学生档案与学籍管理（赋能伪装→修正）：重定位为"流失预测+留存编排"，剥离 EWS 到共享引擎（消除范围借用）；异动办理分层诚实归因（确定性同步归工作流引擎，AI 接管概率性子任务），须按修正项把主体价值从"档案 CRUD"转为"流失预测+挽留编排"。排课与课表管理（赋能伪装→修正）：LLM 真正承重（隐含偏好推断/课表质量评判/冲突权衡推理），deprecate 手动配置逃生舱，agent loop 为默认路径，须按修正项证明 AI 推断权重优于手动配置（A/B）。选课与培养方案管理（赋能伪装→修正）：培养方案知识图谱为唯一数据模型，退役并行 checklist，选课唯一入口为意图驱动规划器，须按修正项消除目录搜索 fallback。依赖：共享风险引擎+培养方案图谱+求解器工具层就位。

### 8.3 保留确定性的部分（诚实标注非 AI）
- **支付/记账/票据/最终授权决策**（缴费模块）：GDPR Art22/PIPL Art24 禁止纯自动化重大权益决策。
- **学位终审人工法定复核**（选课/学籍模块）：域级法律上限，AI 可承重计算但非终审，该域无法达 translation-app 级纯 AI 全自动 native。
- **排课 CP-SAT/GA 求解器**：经典运筹学确定性/随机搜索，受 LLM 编排调用但非 AI 价值心脏。
- **异动跨系统联动工作流引擎**（Temporal/Camunda+API）：数字化收益非 AI 收益，"跑 5 部门"归工作流引擎。
- **成绩单 PDF 模板/证明开具**：稳定前缀缓存优化，降级回退（非核心）。

### 8.4 关键风险与缓解
- **实践退化为赋能伪装**（最大风险）：若 legacy 路径不被 sunset、AI 路径不被采纳与纠错，飞轮停转，移除测试在实践层面失效。缓解：AI 为唯一默认入口，legacy 显式降级模式+sunset 日期+在线监控飞轮转速。
- **合规矛盾未消除即上线**：自杀意念检测数据出境、文本内容处理无 lawful basis、未成年人数据入飞轮无单独同意。缓解：Phase 0 合规层必须先就位，高风险模块上线前合规审计 gate。
- **飞轮学噪声相关性**：无因果归因的飞轮污染知识库形成负飞轮。缓解：反事实/倾向分归因+curation gate。
- **范围借用持续存在**：若各模块仍各自宣称 EWS 为心脏，重复建设与归因混乱。缓解：共享风险融合引擎作为平台能力，架构治理强制各模块订阅而非重建。
- **数据模型倒置的连锁影响**：成绩模块倒置影响所有下游（长期记忆掌握度向量）。缓解：提前到 Phase 0.5 与内核共建，双写过渡+影子模式验证。

## 九、自检清单（如何判断改造是否真达成 AI 原生）

每个模块/子流程须通过以下自检，否则即使宣称 native 也是赋能伪装：

- [ ] **移除测试**：今晚移除所有 AI 调用（含经典 ML 与 LLM），模块核心价值交付路径是否归零、无可行确定性回退？若退化为"仍能独立售卖部署的功能完整传统系统"，判赋能伪装。
- [ ] **概率性逻辑为核心**：核心交付物是模型推理产生的概率性输出（掌握度概率/风险概率/置信度判定），还是确定性规则分支？若决策层是"风险分阈值→分层→工具"的查表，ML 退化为特征抽取，判 if-else 伪 AI。
- [ ] **模型在请求路径最前端**：学生一句话/每次作答/每条家长消息是否过 LLM 或 ML 推理？还是 CRUD 照常跑、AI 仅作可选"智能"按钮（bolt-on 信号）？
- [ ] **承重 vs 便利层归因诚实**：跨系统联动归工作流引擎（非 AI）；求解器归运筹学（非 AI）；学位终审归人工法定（非 AI）。AI 只接管真正概率性部分。量化"AI 不可替代 case 占比>40%"。
- [ ] **legacy 强制 sunset**：AI 失败回退 legacy 表单是否设 sunset 日期、非永久对等路径？若 legacy 作为用户可选对等路径永久保留，AI 非承重，套壳风险成立。
- [ ] **范围借用消除**：是否多个模块各自宣称 EWS 为心脏？风险融合引擎是否作为平台能力被各模块订阅而非重建？
- [ ] **数据模型倒置**（成绩模块特有）：唯一持久化核心是否为作答事件流+知识状态向量？分数/排名/等级是否成为按需派生投影、无持久化 CRUD 副本？
- [ ] **飞轮因果化**：干预效果归因是否用反事实/倾向分/ITS？是否仅沉淀通过因果检验的策略-学生类型映射？知识库是否设 curation gate 防 LLM 无监督自动写入？
- [ ] **合规自洽**：(a) 自杀意念检测是否全自建 BERT 边缘部署（数据不出校），而非并列宣称可调云端 Moderation？(b) NLP 文本分析是否取得单独同意+PIA+最小化（仅 embedding/标记不留原文）？(c) 未成年人飞轮数据是否默认排除训练或脱敏后入飞轮？
- [ ] **公平性 evals**：是否新增人口学公平性/差异影响审计作为 eval 门禁，与 golden 样本、LLM-as-judge 并列？偏见超标是否阻断部署？
- [ ] **生命安全级评测兜底**：自杀意念检测是否设定召回率硬下限（≥0.95）与假阴率上限？高风险案例是否强制人工复核+接入心理危机应急通道？
- [ ] **意义性人工复核**：评审委员会是否独立核验至少一个事实（非仅确认 AI 建议）？AI 协议率过高（>90%橡皮图章）或过低（<50%模型退化）是否触发告警？
- [ ] **evals 为部署阻断门禁**：evals 是否作为 CI/CD 部署硬门禁（不达标不上线），而非事后报告型？是否覆盖核心 ML 指标（AUC/RMSE/召回）+LLM-as-judge+公平性审计三轨？
- [ ] **风险感知路由**：合规关键决策（奖助资格/退费/学位）是否强制走高端模型，成本服从风险？低端模型是否仅用于低风险任务（意图分类/字段抽取）？
- [ ] **可纠错可撤销**：AI 建议是否可编辑/拒绝？干预工单是否可撤回？调课是否可回退？危机沟通是否可 opt-out？
- [ ] **生成式 UI 字段级**：表单字段是否按意图+学籍上下文动态选择（只问缺的字段，从档案拉已知字段），而非 intent→form_id→固定表单渲染（套壳聊天框信号）？
- [ ] **归因可复现存证**：每份生成报告是否挂载 provenance（SQL/检索片段/模型版本/SHAP/trace id）支持同问题重放？
- [ ] **六行成本分解**：embedding/LLM 推理/检索/eval/observability/guardrails 是否均单列可观测？cost-per-query 是否按意图类型对外可辩护？Opus 战略路由是否设频次上限与预算告警？
- [ ] **域级法律上限诚实标注**：学位终审、奖助名额、退费、退学、心理转介是否保留人工法定复核+申诉通道（GDPR Art22/AI Act Annex III 域约束不可解除）？该域是否诚实标注"AI 承重计算但非终审"上限？
