## 一、双库选择决策树

**遇到一个需求时，按这个顺序判断用哪个库：**

```
你需要做什么？
│
├─ 新增/修改/删除一条客户记录？
│  └─ → MySQL（写入操作一律走 MySQL）
│
├─ 查询"某条具体记录"的详情？（按ID/编号查）
│  └─ → MySQL（精确查找，关系简单）
│
├─ 需要跨多张表 JOIN 才能查到的数据？
│  └─ → Neo4j（图遍历替代 JOIN）
│
├─ 需要发现"隐性关系"？（担保圈、关联交易、传导链路）
│  └─ → Neo4j（图算法 + 关系遍历）
│
├─ 需要用自然语言提问？
│  └─ → Neo4j（Text-to-Cypher + 本体辅助）
│
├─ 需要统计聚合？（COUNT/SUM/AVG + GROUP BY）
│  └─ → MySQL（关系型聚合更高效）
│
├─ 需要 AI 生成分析报告？
│  └─ → Neo4j（GraphRAG 提取上下文 → LLM 生成）
│
├─ 需要校验数据是否符合规则？
│  └─ → Neo4j（本体约束 + SHACL 校验）
│
└─ 需要审批流/工作流？
   └─ → MySQL（Flowable 引擎）
```

**一句话规则**：
- **写操作** → MySQL
- **读的简单查询** → MySQL
- **读的关系查询/分析/AI** → Neo4j

## 三、各场景详细交互设计

### ═══ MySQL 场景（传统操作） ═══

> 以下场景沿用传统表单界面，无需 AI 介入。

---

### 场景 1：客户登记/新增

**数据库**：MySQL
**交互方式**：传统表单

```
┌─────────────────────────────────────────────────┐
│  客户登记                                        │
│                                                  │
│  客户名称:  [____________]                       │
│  客户类型:  [企业客户 ▼]                         │
│  证件类型:  [统一社会信用代码 ▼]                  │
│  证件号码:  [____________]                       │
│  手机号:    [____________]                       │
│  注册地址:  [____________]                       │
│  注册资本:  [____________] 万元                  │
│  ...                                             │
│                                                  │
│          [  保存  ]  [  取消  ]                   │
└─────────────────────────────────────────────────┘
```

**数据流向**：用户填写 → 前端校验 → API → MySQL INSERT
**为什么不用 AI**：录入操作需要精确字段，表单最高效。AI 无法替代精确输入。

### ═══ Neo4j 场景（AI 原生交互） ═══

> 以下场景以对话为核心交互方式。用户用自然语言提问，AI 理解意图后查询图谱并返回结果。

### 场景 4：自然语言查客户 ⭐

**数据库**：Neo4j
**AI 原生程度**：★★★

**传统方式**：打开列表 → 输入筛选条件 → 翻页找 → 点进去看详情（3-5 步）
**AI 原生方式**：直接说一句话（1 步）

#### 交互流程

```
用户: "帮我找一下注册资本超过 5000 万的制造业企业客户"

  ┌─────────────────────────────────────────────┐
  │  AI 处理过程（用户不可见）                    │
  │                                              │
  │  1. LLM 读取本体定义:                        │
  │     - Customer 有 registeredCapital 属性     │
  │     - EnterpriseCustomer 有 mainBusiness     │
  │     - EnterpriseCustomer 有 companyType      │
  │                                              │
  │  2. LLM 生成 Cypher:                         │
  │     MATCH (c)                                │
  │     WHERE 'ns1__EnterpriseCustomer'          │
  │           IN labels(c)                       │
  │       AND c.cusName IS NOT NULL              │
  │     RETURN c.uri, c.cusName,                │
  │            c.registeredCapital               │
  │     ORDER BY c.registeredCapital DESC        │
  │     LIMIT 20                                 │
  │                                              │
  │  3. Neo4j 执行，返回结果                      │
  └─────────────────────────────────────────────┘

AI 回答:
"找到以下注册资本超过 5000 万的制造业企业（前 10 条）：

 1. 福田重工股份有限公司 — 注册资本 8.5 亿
 2. 北汽雷蒂机械科技有限公司 — 注册资本 5.2 亿
 3. 中集车辆集团股份公司 — 注册资本 5.0 亿
 ...

需要查看某家企业的详细信息吗？"

用户: "看一下第一家"

AI:（自动执行图遍历，进入场景 5 客户全景画像）
```

**选择 Neo4j 的原因**：需要跨多个属性条件查询 + 本体知道"制造业"对应什么字段。MySQL 也能做，但 AI 让用户不用记字段名。

### 场景 5：客户全景画像 ⭐

**数据库**：Neo4j
**AI 原生程度**：★★★

**传统方式**：分别打开客户表、联系人表、账户表、报表...（5+ 张表分别查看）
**AI 原生方式**：一句话获得全景摘要

#### 交互流程

```
用户: "帮我分析一下张三公司的全面情况"

  ┌─────────────────────────────────────────────┐
  │  AI 处理过程                                  │
  │                                              │
  │  Step 1: 图遍历获取全部关联数据               │
  │  MATCH (c)-[r]->(related)                    │
  │  WHERE c.cusName CONTAINS '张三'             │
  │  RETURN c, r, related                        │
  │                                              │
  │  Step 2: 提取关键数据点                       │
  │  - 基本信息: 名称、类型、评级                 │
  │  - 企业信息: 注册资本、主营业务               │
  │  - 联系人: 3 位                              │
  │  - 银行账户: 2 个                            │
  │  - 关联企业: 1 家                            │
  │  - 行业位置: 制造业-专用设备                  │
  │                                              │
  │  Step 3: 结构化数据 + 图特征 → LLM 生成报告  │
  └─────────────────────────────────────────────┘

AI 回答:
"## 张三公司全景画像

**基本信息**
- 企业客户，2015年3月成立
- 注册资本 5000 万元，实缴 3000 万
- 统一社会信用代码：91110000...
- 信用评级：AA
- 主营业务：机械设备租赁

**关联网络**
- 联系人 3 位（第一联系人：王五）
- 银行账户 2 个（工商银行、建设银行）
- 关联企业 1 家（北京张三物流有限公司）

**行业位置**
- 行业：制造业 > 专用设备制造业
- 同行业客户 156 家，该公司评级排名前 10%

**风险提示**
- 资料完整度 85%，缺少近三年审计报告
- 存在 1 条涉诉记录（已结案）
- 实际控制人近期变更（2024年6月）

**建议**
- 建议补充近三年财务报表
- 关注实际控制人变更后的经营状况"
```

**选择 Neo4j 的原因**：一次图遍历获取全部关联数据，MySQL 需要 5+ 次 JOIN 查询。AI 将结构化数据转化为可读报告。

### 场景 9：关系发现/关联查询 ⭐

**数据库**：Neo4j
**AI 原生程度**：★★★

**传统方式**：❌ MySQL 几乎无法实现（需要递归 JOIN 多张表）
**AI 原生方式**：自然语言提问 → 图遍历 → 可视化展示

#### 交互流程

```
用户: "张三公司和哪些客户有关联关系？"

AI:
"张三公司有以下直接关联：

 📊 关联关系图：
 
 张三公司 ──[关联客户]──→ 北京张三物流
 张三公司 ──[关联客户]──→ 李四贸易
 张三公司 ──[联系人-同一人]──→ 王五所在的其他企业

需要进一步查看这些关联企业的详情吗？"

用户: "再深一层，张三物流还关联了谁？"

AI:（二度遍历图谱）
"张三物流的关联：
 - 与 赵六建材 共享同一法定代表人
 - 与 孙七供应链 存在担保关系
 - 与 周八科技 在同一办公地址

⚠️ 注意：张三公司 → 张三物流 → 赵六建材 形成了
   一条 2 度关联链路，建议关注潜在的利益冲突。"
```

**选择 Neo4j 的原因**：这是 Neo4j 最擅长的场景——多层关系遍历。MySQL 做 2 度以上关联查询极其复杂且性能差。

```

  │                                              │

  │  3. 结果渲染为可视化图谱                       │
  └─────────────────────────────────────────────┘

"张三公司的担保圈分析：

 📊 [可视化图谱展示]
 
 张三公司 →担保→ 李四贸易 →担保→ 赵六建材
     ↑                              │
     └──────────担保────────────────┘
 
 ⚠️ 发现担保环路！涉及 3 家企业。
 如果其中一家出现债务问题，风险会在圈内传导。

 担保总额估算：约 1.2 亿元

 📊 传导链：

**数据库**：Neo4j（图数据）+ LLM（分析推理）
**AI 原生程度**：★★★

```
用户: "帮我准备一份关于张三公司的尽调摘要，下周要上风控会"

AI:
"## 张三公司尽职调查摘要

### 一、企业概况
- 全称：北京张三机械设备有限公司

- 注册资本：5000万元（实缴3000万）
- 主营：机械设备租赁及销售

### 二、股权结构
- 实际控制人：张三（持股 65%）

1. 实缴与注册差额较大（2000万）

## 五、AI 原生交互的技术实现

### 5.1 Text-to-Cypher 管道

```
用户自然语言
    ↓
┌───────────────────────────────────────┐
│ 1. 意图识别                           │
│    LLM: "用户想查询客户信息"           │
│                                       │
│ 2. 本体辅助                           │
│    读取 ontology.ttl → 获取类/属性名   │
│    知道: Customer.cusName,            │
│          EnterpriseCustomer.registeredCapital │
│                                       │
│ 3. Cypher 生成                        │
│    LLM + 本体 → 生成准确的 Cypher     │
│                                       │
│ 4. 执行查询                           │
│    Neo4j 执行 → 返回结果              │
│                                       │
│ 5. 结果格式化                         │
│    LLM 将结构化结果转化为自然语言回答   │
└───────────────────────────────────────┘
    ↓
AI 回答用户
```

### 5.2 GraphRAG 管道

```
用户提问（需要深度分析）
    ↓
┌───────────────────────────────────────┐
│ 1. 图遍历                             │
│    从目标节点出发，遍历 1-3 度关联      │
│    收集: 基本信息 + 关系 + 属性        │
│                                       │
│ 2. 上下文构建                         │
│    图数据 → 结构化文本                 │
│    "张三公司: 注册资本5000万,          │
│     联系人3位, 关联企业2家..."         │
│                                       │
│ 3. LLM 分析                           │
│    上下文 + 用户问题 → LLM 生成分析    │
│                                       │
│ 4. 输出                               │
│    结构化报告 / 自然语言回答            │
└───────────────────────────────────────┘
    ↓
AI 分析报告
```

### 5.3 多轮对话上下文管理

```
用户: "找一下制造业的客户"
AI:   "找到 156 家..."         ← 记住查询结果 ID 列表

用户: "评级最高的是哪家？"
AI:   "福田重工（AAA级）"       ← 在上一轮结果上过滤

用户: "帮我分析一下"
AI:   "## 福田重工全景画像..."  ← 对上一轮结果做 GraphRAG

用户: "它和哪些客户有关联？"
AI:   "福田重工的关联..."       ← 继续在同一实体上操作
```

**关键**：AI 维护对话上下文，知道"它"="福田重工"，不需要用户重复指定。

## 六、总结

### 数据库选择本质

| 你在做什么 | 用哪个库 | 为什么 |
|-----------|---------|--------|
| **改变数据**（增删改） | MySQL | 需要事务保证、审计日志 |
| **查一条数据** | MySQL | 简单、快 |
| **查关系** | Neo4j | 图遍历比 JOIN 快 100 倍 |
| **查深层关系** | Neo4j | MySQL 根本做不了 |
| **用嘴问** | Neo4j | AI 理解自然语言 → Cypher |
| **做分析** | Neo4j | 图算法 + LLM 推理 |
| **算统计** | MySQL | COUNT/SUM 更高效 |
| **走审批** | MySQL | Flowable 引擎 |

### AI 原生的本质变化

```
传统系统:  人适应系统（学菜单、记字段、翻页找）
AI 原生:   系统适应人（说一句话，系统理解并执行）
```

用户不再需要知道：
- 数据存在哪张表
- 字段叫什么名字
- 怎么 JOIN 多张表
- 怎么用筛选条件

用户只需要知道：
- **我想要什么信息**
- **我想分析什么问题**

剩下的交给 AI。
