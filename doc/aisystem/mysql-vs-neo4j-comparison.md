# MySQL vs Neo4j：从本次迁移看两个数据库的异同

> 本文以 mftcc-cus-server 的实际迁移为例，面向零基础读者

---

## 一、先理解一个比喻

想象你在管理一本**客户通讯录**：

**MySQL 的做法**：就像 Excel 表格。你建一张"客户表"、一张"联系人表"、一张"银行账户表"。表和表之间通过"客户编号"互相引用。要查一个客户的所有联系人，你需要先在"客户表"找到这个客户，拿着客户编号去"联系人表"里搜。

**Neo4j 的做法**：就像一张关系网/思维导图。每个客户是一个**点**，每个联系人也是一个**点**，点和点之间直接画一条**线**连着。要查一个客户的联系人？顺着线走就行了。

```
MySQL 视角（表格）:                Neo4j 视角（关系网）:

┌─────────────┐                    ┌──────────┐
│ 客户表       │                     │ 企业客户  │
│ id | name   │                     └────┬─────┘
├─────────────┤                          │ 有联系人
│ 001 | 张三  │  ──→  查表关联  ──→       ▼
└─────────────┘                    ┌──────────┐
                                   │ 联系人    │
┌─────────────┐                    │ 王五      │
│ 联系人表     │                    └──────────┘
│ id | cus_id │
├─────────────┤
│ 1  | 001    │  ← 通过 cus_id 关联
└─────────────┘
```

## 二、共同点（两个数据库都有的东西）

### 1. 都能存数据

这是最基本的。MySQL 把数据存在**表**里，Neo4j 把数据存在**节点**里。

| 概念 | MySQL 里的样子 | Neo4j 里的样子 |
|------|---------------|----------------|
| 一条客户记录 | 表里的一**行** | 图里的一个**节点** |
| 客户的名字 | 行的一个**字段值** | 节点的一个**属性** |
| 客户编号（唯一标识） | **主键** (id) | **URI** (唯一地址) |

**本次迁移实例**：
- MySQL 里 `cus_customer` 表有 22,966 行 → Neo4j 里变成了 22,966 个节点
- 每个节点都有唯一的 URI，比如 `http://mftcc.com/data/cus#customer/zhz-yls_kf100000003r3s`

### 2. 都支持数据类型

两边都区分"文本"、"数字"、"日期"等类型。

| 数据类型 | MySQL 写法 | Neo4j (RDF) 写法 |
|---------|-----------|-----------------|
| 文本 | `VARCHAR(100)` | `xsd:string` |
| 整数 | `INT` | `xsd:integer` |
| 小数 | `DECIMAL(16,2)` | `xsd:decimal` |
| 日期 | `DATE` | `xsd:date` |
| 日期时间 | `TIMESTAMP` | `xsd:dateTime` |
| 布尔 | `CHAR(1)` (0/1) | `xsd:boolean` |

**本次迁移实例**：
- MySQL 的 `cus_name VARCHAR(100)` → Neo4j 的 `cusName` 属性，类型 `xsd:string`
- MySQL 的 `registered_capital_amt DECIMAL(16,2)` → Neo4j 的 `registeredCapital`，类型 `xsd:decimal`

### 3. 都支持表/节点之间的关系

这是最关键的部分，也是两者最大的差异点（后面详讲）。

| 概念 | MySQL | Neo4j |
|------|-------|-------|
| 关系怎么表示 | 外键字段（一个列） | 边（直接连线） |
| 关系的名字 | 没有名字，靠列名理解 | 有明确的类型名 |
| 关系的方向 | 隐含的（子表指向父表） | 明确的（有箭头） |

**本次迁移实例**：

MySQL 里，联系人表有一个 `cus_id` 字段指向客户表：
```
cus_contact_info 表:
┌────┬──────────┬────────┐
│ id │ cus_id   │ name   │
├────┼──────────┼────────┤
│ 1  │ 001      │ 王五   │  ← cus_id=001 表示"属于客户001"
└────┴──────────┴────────┘
```

Neo4j 里，联系人节点和客户节点之间有一条"线"：
```
(客户张三) ──[有联系人]──→ (联系人王五)
```

### 4. 都有查询语言

| | MySQL | Neo4j |
|--|-------|-------|
| 语言名 | **SQL** | **Cypher** |
| 查询数据 | `SELECT * FROM ...` | `MATCH (n) RETURN n` |
| 条件过滤 | `WHERE name = '张三'` | `WHERE n.cusName = '张三'` |
| 关联查询 | `JOIN ... ON ...` | `MATCH (a)-[r]->(b)` |

### 5. 都支持索引

两边都能给常用字段建索引，加速查询。MySQL 有 B-tree 索引，Neo4j 有节点属性索引。

## 三、差异点（两个数据库本质的不同）

### 差异 1：数据模型 —— 表格 vs 关系网

**这是最核心的区别。**

**MySQL = 表格模型**
```
数据被强制分成一张张独立的表
表与表之间通过"编号"间接关联
要理解全貌，需要在脑子里把多张表"拼"起来

    ┌─────────┐     ┌─────────┐     ┌─────────┐
    │ 客户表   │     │ 联系人表 │     │ 账户表   │
    │         │     │         │     │         │
    │ 001 张三│ ←── │ 001 王五 │     │ 001 工行│
    │ 002 李四│ ←── │ 002 赵六 │ ←── │ 002 建行│
    └─────────┘     └─────────┘     └─────────┘
```

**Neo4j = 图模型**
```
数据就是一个大网络
每个事物是一个点，事物之间的关系是一条线
所有数据天然连在一起，不需要"拼"

         ┌──[有联系人]──→ (王五)
         │
(张三) ──┼──[有联系人]──→ (赵六)
         │
         ├──[有银行账户]──→ (工行账户) ──[属于银行]──→ (工商银行)
         │
         └──[有发票]──→ (发票信息)
```

**实际影响**：

| 场景 | MySQL | Neo4j |
|------|-------|-------|
| 查一个客户的所有联系人 | 需要 JOIN 两张表 | 顺着线走一步 |
| 查"客户的联系人的银行账户" | 需要 JOIN 三张表 | 顺着线走两步 |
| 查"客户→联系人→关联企业→该企业的联系人" | 需要 JOIN 四张表，SQL 很复杂 | 顺着线走四步，写法几乎一样 |

### 差异 2：关系是"一等公民" vs "二等公民"

**MySQL 里，关系是隐含的**：
- 你建了一个 `cus_id` 字段，但数据库不知道这叫"属于某个客户"
- 关系没有自己的属性（比如"什么时候建立的关联"没法直接存在关系上）
- 每次查询都要通过 JOIN 重新建立关系

**Neo4j 里，关系是和节点同等重要的存在**：
- 关系有明确的类型名：`hasContact`、`hasBankAccount`
- 关系有方向：`客户 → 联系人`（不是反过来）
- 关系可以有自己的属性（未来可以加"关联时间"等）

**本次迁移实例**：
```
MySQL 的隐含关系:
cus_contact_info.cus_id → cus_customer.id
（数据库只知道两个编号相等，不知道这代表什么含义）

Neo4j 的显式关系:
(客户) -[:hasContact {第一联系人: true}]-> (联系人)
（明确知道这是"拥有联系人"关系，还能附带额外信息）
```

### 差异 3：模式刚性 vs 灵活

**MySQL =  rigid schema（刚性模式）**
- 建表时必须定义好所有列和数据类型
- 想加一个新字段？需要 ALTER TABLE，大表可能要锁几分钟
- 不同类型的数据必须放在同一张表的同一列里

**Neo4j = flexible schema（灵活模式）**
- 节点可以随时加新属性，不需要"建表"
- 不同类型的节点可以有不同的属性
- 同一种关系可以连接不同类型的节点

**本次迁移实例**：
```
MySQL 的做法:
cus_customer 表有 72 个字段，不管你是企业客户还是个人客户，所有字段都在
企业客户用不到的字段（如 birthday）就是 NULL
个人客户用不到的字段（如 registeredCapital）也是 NULL
→ 大量 NULL 值浪费空间

Neo4j 的做法:
EnterpriseCustomer 节点只有企业相关的属性
PersonalCustomer 节点只有个人相关的属性
没有多余的空字段
→ 更干净、更省空间
```

### 差异 4：查询模式的本质差异

**MySQL 擅长：结构化查询、聚合统计**
```sql
-- 这个在 MySQL 里很自然
SELECT cus_type, COUNT(*) AS cnt, AVG(rating_score) AS avg_score
FROM cus_customer
WHERE register_date > '2024-01-01'
GROUP BY cus_type
ORDER BY cnt DESC
```

**Neo4j 擅长：关系遍历、路径发现**
```cypher
-- 这个在 Neo4j 里很自然，MySQL 很难做到
-- 查找与"张三"有 3 度以内关联关系的所有客户
MATCH path = (zhang:Resource)-[:ns1__relatedTo*1..3]-(other:Resource)
WHERE zhang.uri CONTAINS 'zhangsan'
RETURN other.uri, length(path) AS distance
ORDER BY distance
```

**对比总结**：

| 查询类型 | MySQL 表现 | Neo4j 表现 |
|---------|-----------|------------|
| 聚合统计（COUNT/AVG/SUM） | ★★★★★ | ★★★ |
| 精确查找（按ID/名称） | ★★★★★ | ★★★★★ |
| 关联查询（2-3 层） | ★★★ | ★★★★★ |
| 深层关系遍历（5+ 层） | ★（极慢或无法实现） | ★★★★★ |
| 路径发现（最短路径） | ☆（不支持） | ★★★★★ |
| 全文搜索 | ★★★ | ★★★ |

### 差异 5：语义层（本体）—— Neo4j 独有

**MySQL 没有语义理解能力**。它只知道"这个字段叫 cus_name"，不知道"这是一个客户的名字"。

**Neo4j + neosemantics 有本体层**。它知道：
- `EnterpriseCustomer` 是一种 `Customer`（继承关系）
- `hasContact` 连接的是 `Customer` 和 `ContactPerson`（定义域和值域）
- 可以用 SHACL 规则校验数据是否符合本体约束

**本次迁移实例**：
```
MySQL:
cus_customer.cus_base_type = '1' → 这只是个字符串"1"，数据库不知道这是"企业"

Neo4j:
节点标记为 EnterpriseCustomer → 系统知道：
  1. 它是 Customer 的子类
  2. 它有注册资本、经营范围等企业特有的属性
  3. 它可以被企业相关的评级模型评估
  4. 如果数据不符合规则（比如个人客户有了注册资本），可以被校验出来
```

## 四、用本次迁移数据做一个直观对比

### 同一条数据在两个数据库里的"长相"

**一个企业客户"张三公司"，有 2 个联系人，1 个银行账户**

**在 MySQL 里，数据分散在 4 张表中**：

```sql
-- 你需要查 4 张表才能了解全貌
SELECT * FROM cus_customer WHERE id = '001';           -- 基础信息
SELECT * FROM cus_corp_base_info WHERE cus_id = '001'; -- 企业信息
SELECT * FROM cus_contact_info WHERE cus_id = '001';   -- 联系人
SELECT * FROM cus_bank_acc_manage WHERE cus_id = '001'; -- 银行账户
```

**在 Neo4j 里，一个节点 + 它连出的线就是全貌**：

```cypher
// 一条查询看到所有
MATCH (c)-[r]->(related)
WHERE c.uri CONTAINS '001'
RETURN c, r, related
```

你会看到：
```
(张三公司) ──[hasContact]──→ (王五)
           ──[hasContact]──→ (赵六)
           ──[hasBankAccount]──→ (工行账户) ──[belongsToBank]──→ (工商银行)
           属性: cusName=张三公司, registeredCapital=5000000, mainBusiness=融资租赁
```

### 数据量对比

| 维度 | MySQL | Neo4j（本次迁移后） |
|------|-------|-------------------|
| 存储单位 | 行 (Row) | 节点 (Node) + 关系 (Relationship) |
| 客户数据 | 22,966 行 | 22,966 个节点 |
| 联系人 | 12,940 行 | 12,940 个节点 + 12,940 条关系 |
| 银行账户 | 10,359 行 | 10,359 个节点 + 10,359 条关系 |
| 关系总量 | 隐含在外键字段中 | **49,606 条显式关系** |
| 数据总量 | ~6 万行 | **221,588 节点 + 49,606 关系** |

注意：MySQL 里 6 万行数据，在 Neo4j 里变成了 27 万个图元素（节点+关系）。
这是因为 Neo4j 把"关系"也变成了独立的数据，而 MySQL 里关系只是外键字段里的一个数字。

## 五、各自适合什么场景

| 场景 | 选 MySQL | 选 Neo4j |
|------|----------|----------|
| 财务对账、精确统计 | ✅ | |
| 订单管理、库存管理 | ✅ | |
| 客户 360 度视图 | | ✅ |
| 关联关系分析 | | ✅ |
| 风控链路追踪 | | ✅ |
| 供应链/担保圈分析 | | ✅ |
| 推荐系统 | | ✅ |
| 传统报表 | ✅ | |
| 知识图谱 | | ✅ |

**一句话总结**：
- **MySQL** 适合"数据是表格"的场景（财务、订单、库存）
- **Neo4j** 适合"数据是关系网"的场景（社交网络、知识图谱、风控链路）

**本次迁移的意义**：把客户数据从"表格"变成了"关系网"，解锁了关系分析和图谱查询的能力，这是 MySQL 做不到或很难做到的。
