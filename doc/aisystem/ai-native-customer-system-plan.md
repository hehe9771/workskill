## 一、核心结论

### 1.1 只用本体数据库（Neo4j）够吗？

**不够。必须双库协同。**

| 能力 | MySQL | Neo4j | 结论 |
|------|-------|-------|------|
| 高频 CRUD（增删改查） | ★★★★★ | ★★ | MySQL 负责 |
| 事务一致性（ACID） | ★★★★★ | ★★★★ | MySQL 负责 |
| 关系遍历与分析 | ★★ | ★★★★★ | Neo4j 负责 |
| 语义推理（本体） | ☆ | ★★★★★ | Neo4j 独有 |
| AI 查询（自然语言） | ★★ | ★★★★★ | Neo4j 负责 |
| 图谱可视化 | ☆ | ★★★★★ | Neo4j 负责 |
| 报表/统计 | ★★★★★ | ★★★ | MySQL 负责 |
| 知识图谱 + RAG | ☆ | ★★★★★ | Neo4j 独有 |

**结论**：MySQL 做"数据写入层"（业务系统照常运行），Neo4j 做"AI 智能层"（查询、分析、推理）。两者通过 ETL 同步。

### 1.2 什么是"AI 原生"客户系统？

**传统 CRM**：人看菜单 → 点按钮 → 看表格 → 做判断
**AI 原生 CRM**：人提问（自然语言）→ AI 理解 → AI 查询图谱 → AI 分析推理 → 给出结论

```
传统:  用户 → 界面 → SQL查询 → 表格展示 → 人脑分析
AI原生: 用户 → 自然语言 → AI引擎 → 图谱遍历+推理 → 智能回答
```

核心区别：
1. **交互方式变了**：不再靠菜单和表单，靠对话
2. **分析能力变了**：不再靠人脑想关系，AI 自动发现关联
3. **决策支持变了**：不再靠经验，靠图谱推理 + 本体约束

## 三、两个数据库的职责分工

### 3.1 MySQL：业务写入层（System of Record）

**角色**：数据源、事务中心、业务操作入口

| 职责 | 说明 |
|------|------|
| **客户 CRUD** | 新增/修改/删除客户信息，所有业务操作在此完成 |
| **业务流水** | 合同、放款、还款、逾期等交易数据 |
| **权限管理** | 用户、角色、数据权限控制 |
| **审批流程** | Flowable 工作流引擎 |
| **报表统计** | 传统 BI 报表、监管报送 |
| **数据同步源** | 作为 ETL 的源库，向 Neo4j 推送变更 |

**关键原则**：现有业务系统不动，MySQL 照常用。

### 3.2 Neo4j：AI 智能层（AI Brain）

**角色**：知识图谱中心、AI 查询引擎、推理分析引擎

| 职责 | 说明 |
|------|------|
| **知识图谱** | 存储客户本体 + 实例数据，形成语义化知识网络 |
| **关系发现** | 自动发现客户间的隐性关联（担保圈、关联交易等） |
| **自然语言查询** | 用户用中文提问，AI 转化为图谱查询并返回结果 |
| **GraphRAG** | 基于图谱的知识检索增强，为 LLM 提供精准上下文 |
| **风险推理** | 基于本体约束 + 图结构推理风险传导路径 |
| **客户 360** | 自动生成客户全景画像（图遍历 + LLM 总结） |

## 四、AI 原生功能设计

### 4.1 功能矩阵

| 功能 | 描述 | 依赖 | 优先级 |
|------|------|------|--------|
| **自然语言查询** | "张三公司在哪个行业？注册资本多少？" | Text-to-Cypher | P0 |
| **客户全景画像** | "给我看一下这家企业的全部信息" | 图遍历 + LLM | P0 |
| **关系发现** | "这家公司和哪些客户有关联？" | 图遍历 | P0 |
| **担保圈分析** | "画出这家企业的担保圈" | 图算法 | P1 |
| **风险传导** | "如果这家企业出问题，会影响谁？" | 图推理 + 本体 | P1 |
| **智能推荐** | "这个行业的优质客户有哪些？" | 图查询 + 过滤 | P1 |
| **合规校验** | "检查这个客户的资料是否完整" | SHACL + 本体 | P2 |
| **变更影响** | "修改这个行业分类，会影响哪些客户？" | 本体推理 | P2 |

### 4.2 核心功能详解

#### 功能 1：自然语言查询（Text-to-Cypher）

```
用户提问: "张三公司的联系人有哪些？他们在哪个银行开户？"
    ↓
LLM 理解意图 + 本体辅助
    ↓
生成 Cypher:
MATCH (c)-[:ns1__hasContact]->(p)-[:ns1__hasBankAccount]->(ba)-[:ns1__belongsToBank]->(b)
WHERE c.uri CONTAINS '张三'
RETURN c.uri, p.uri, ba.uri, b.uri
    ↓
Neo4j 执行查询
    ↓
LLM 组织回答:
"张三公司有 3 位联系人：王五（工商银行）、赵六（建设银行）、李七（农业银行）"
```

**本体的作用**：LLM 通过读取本体定义，知道 Customer 有 hasContact 关系，BankAccount 有 belongsToBank 关系，从而生成正确的 Cypher。

#### 功能 2：客户全景画像（GraphRAG）

```
用户提问: "帮我分析一下张三公司"
    ↓
系统执行:
1. 图遍历获取客户全部关联数据
   - 基本信息、企业信息、联系人、银行账户、发票、报表...
2. 提取图结构特征
   - 关联客户数、行业位置、评级等级...
3. 将图数据 + 结构特征 作为上下文送给 LLM
    ↓
LLM 生成分析报告:
"张三公司成立于 2015 年，注册资本 5000 万，主营融资租赁。
 企业信用评级 AA，有 5 位联系人、3 个银行账户。
 关联企业中包含 2 家担保方，需关注担保圈风险..."
```

#### 功能 3：担保圈分析

```
用户提问: "画出张三公司的担保圈"
    ↓
Cypher:
MATCH path = (c)-[:ns1__relatedTo*1..5]-(other)
WHERE c.uri CONTAINS '张三'
RETURN path
    ↓
Neo4j 返回图结构
    ↓
前端渲染为可视化图谱（节点+边+高亮路径）
```

## 六、数据同步策略

### 6.1 同步架构

```
MySQL (业务写入)
    ↓  触发方式：
    │  1. Hive 日批 → 数仓更新 → 触发同步
    │  2. 定时任务（每小时/每天）
    │  3. MySQL binlog CDC（准实时，可选）
    ↓
Python ETL
    ↓  处理逻辑：
    │  1. 增量检测（updated_at > last_sync）
    │  2. 变更行 → 生成 DELETE + INSERT RDF
    │  3. 新关系 → 生成新边
    ↓
Neo4j (n10s 导入)
```

### 6.2 同步粒度

| 数据类型 | 同步频率 | 同步方式 | 预估延迟 |
|----------|----------|----------|----------|
| 客户基础信息 | 每天 | 增量同步（updated_at） | T+1 |
| 联系人/账户 | 每天 | 增量同步 | T+1 |
| 评级/模型 | 变更时 | 事件触发 | 准实时 |
| 字典/行业 | 极少变更 | 手动同步 | - |

### 6.3 增量检测机制

```python
# 每张表维护 last_sync_time
# 每次同步只处理 updated_at > last_sync_time 的行
# 对于删除操作：MySQL 软删除字段 is_deleted → 生成 DELETE RDF
```

## 八、本体在 AI 系统中的价值

### 8.1 为什么需要本体？

| 能力 | 没有本体 | 有本体 |
|------|----------|--------|
| **LLM 理解数据** | LLM 不知道"cus_name"是客户名称 | LLM 知道 Customer.cusName 是客户名称 |
| **查询准确性** | LLM 猜测字段名，经常写错 Cypher | LLM 读取本体定义，生成准确查询 |
| **推理能力** | 只知道数据，不知道规则 | 知道"企业客户必须有注册资本"等约束 |
| **关系语义** | 只知道两个 ID 关联 | 知道"hasContact"是客户到联系人的关系 |
| **数据校验** | 无法自动检查数据合规性 | SHACL 校验自动发现问题 |

### 8.2 本体如何辅助 LLM

```
场景: 用户问 "哪些企业的评级是 AA？"

无本体时 LLM 的猜测:
  SELECT * FROM customers WHERE rating = 'AA'  ← 可能猜错表名/字段名

有本体时 LLM 的推理:
  1. 读取本体 → 知道 EnterpriseCustomer 类有 ratingGrade 属性
  2. 读取本体 → 知道 EnterpriseCustomer 是 Customer 的子类
  3. 生成准确 Cypher:
     MATCH (c) WHERE 'ns1__EnterpriseCustomer' IN labels(c)
     AND c.`http://mftcc.com/ontology/cus#ratingGrade` = 'AA'
     RETURN c.uri, c.`http://mftcc.com/ontology/cus#cusName`
```

**本体 = LLM 的"数据字典"，让 AI 真正理解数据结构。**

## 十、总结

### 两个数据库如何协作？

```
MySQL（业务写入层）        Neo4j（AI 智能层）
┌──────────────────┐      ┌──────────────────┐
│                  │ ETL  │                  │
│  客户 CRUD       │ ───→ │  知识图谱         │
│  业务流水        │ 同步  │  本体推理         │
│  审批流程        │      │  自然语言查询      │
│  报表统计        │      │  关系发现         │
│                  │      │  风险传导         │
└──────────────────┘      └──────────────────┘
        ↑                         ↑
        │                         │
   业务人员操作              AI 对话/分析
   (传统界面)               (自然语言交互)
```

### 回答核心问题

**Q: 只使用本体数据库 OK 吗？**
**A: 不行。** MySQL 负责业务写入和事务，Neo4j 负责 AI 查询和分析。两者缺一不可。MySQL 是"数据的家"，Neo4j 是"数据的大脑"。

**Q: 本体数据库的核心价值是什么？**
**A: 让 AI 理解数据的含义。** 没有本体，LLM 只是猜测字段名；有本体，LLM 知道数据的语义和约束，能生成准确的查询和推理。

**Q: AI 原生和传统系统的区别？**
**A: 交互范式变了。** 传统系统靠菜单和表单，AI 原生靠对话和推理。用户不再需要"知道怎么操作系统"，只需要"知道自己想要什么"。
