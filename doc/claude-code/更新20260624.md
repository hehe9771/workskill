## 2. 五个新特性解释

### 2.1 嵌套 Skills 自动加载（v2.1.178）

**含义**：Claude Code 自动扫描 `.claude/skills/` 目录下所有子目录（含多层嵌套），无需手动注册。

```
.claude/skills/
├── brainstorming/SKILL.md           ← 1层，直接加载
├── superpowers/
│   ├── brainstorming/SKILL.md       ← 2层嵌套，以前不加载，现在自动加载
│   ├── writing-plans/SKILL.md
│   └── tdd/SKILL.md
└── my-team/
    └── custom-review/
        └── deep/SKILL.md            ← 3层嵌套，也能自动加载
```

**好处**：支持技能的模块化组织和复用。从"扁平列表"变成"文件夹管理"，可以按团队/项目/领域分组，不同子目录可有同名技能。

### 2.2 Skills 自动加载 — 无需市场安装（v2.1.157）

**含义**：只要把技能文件夹（含 `SKILL.md`）放到 `.claude/skills/` 下，Claude Code 启动时自动发现并注册，不需要通过 Plugin Marketplace 安装。

**好处**：降低技能的分发和使用门槛，从"安装-配置-使用"变成"放下-就用"。团队共享、个人技能、实验性技能、离线环境都更方便。

### 2.3 claude plugin init — 脚手架新插件（v2.1.157）

**含义**：提供 CLI 命令快速生成插件的模板/脚手架，不用手动创建目录结构和配置文件。

```bash
claude plugin init my-awesome-plugin
# 交互式创建，自动生成 skills/、agents/、CLAUDE.md 等目录结构
```

生成的脚手架：
```
my-awesome-plugin/
├── CLAUDE.md          # 插件描述和元信息
├── skills/
│   └── my-skill/
│       └── SKILL.md   # 技能定义
├── agents/
│   └── my-agent.md    # Agent 定义
└── package.json       # 插件元数据（用于发布到 marketplace）
```

### 2.4 嵌套子 Agent — 最深 5 层（v2.1.172）

**含义**：主 Claude 可以启动子 Agent，子 Agent 内部还可以再启动自己的子 Agent，形成层级结构，最多 5 层深。

```
主 Claude (Layer 0)
  └── Agent A: "分析代码架构" (Layer 1)
        └── Agent A-1: "查找所有 API 端点" (Layer 2)
              └── Agent A-1-a: "检查 auth 模块" (Layer 3)
                    └── Agent A-1-a-i: "查看具体函数实现" (Layer 4)
                          └── Agent A-1-a-i-α: "读取某函数源码" (Layer 5) ← 最深
```

**好处**：
- 更智能的分派：深读的 Agent 比主 Claude 更了解局部上下文，拆分更合理
- 减少主循环负担：主 Claude 不用管细节拆分，节省 token
- 真实工作流模拟：模拟"领导→组长→员工→实习生"的层级委托
- 复杂任务可解：单层做不到的深度递归审计

### 2.5 1-4 综合场景举例

团队开发"自动化代码审计插件"：
1. 用 `claude plugin init` 创建插件骨架
2. 把插件 skills 放到 `.claude/skills/` 下（自动加载 + 嵌套加载）
3. 使用技能触发工作流，嵌套 Agent 递归分派审计任务

## 3. 插件（Plugin）与技能（Skill）的区别

| | 技能 (Skill) | 插件 (Plugin) |
|---|---|---|
| **是什么** | 一个指令文件（`SKILL.md`） | 一个**可分发的包** |
| **包含什么** | 只有一段 prompt 指令 | 可以包含：多个 Skill + Agent + MCP Server + Hook + 配置文件 |
| **类比** | 一个函数 | 一个 npm 包 / Python 包 |
| **最小单位** | `SKILL.md` | `CLAUDE.md`（插件入口） |
| **分发** | 放在 `.claude/skills/` 下 | `claude plugin install` 安装或 git 克隆 |

结构对比：
```
# 一个 Skill（最小）
.claude/skills/
└── my-skill/
    └── SKILL.md          ← 就这一个文件

# 一个 Plugin（可以包含很多东西）
my-plugin/
├── CLAUDE.md             ← 插件入口，定义元信息
├── skills/               ← 可含多个 skill
│   ├── skill-a/
│   │   └── SKILL.md
│   └── skill-b/
│       └── SKILL.md
├── agents/               ← 可含自定义 agent
│   └── my-agent.md
├── mcp/                  ← 可含 MCP server 配置
│   └── server.json
└── hooks/                ← 可含 hook 脚本
    └── post-tool-use.sh
```

**核心区别**：
- **Skill = 内容**（告诉 Claude 怎么做某件事的指令）
- **Plugin = 容器**（把 Skill、Agent、MCP、Hook 打包在一起，方便分发和安装）

实际例子：`superpowers` 这一个插件包含了 14+ 个 skill（brainstorming、writing-plans、tdd 等），它们被安装后出现在 `.claude/skills/superpowers/` 下。

## 4. 主 Agent 嵌套子 Agent 示例

### 场景：代码生成 Agent（协调前端审查 + 后端审查）

### 目录结构
```
.claude/agents/
├── code-generator.md          ← 主 Agent（协调者）
├── frontend-reviewer.md       ← 子 Agent 1
└── backend-reviewer.md        ← 子 Agent 2
```

### 主 Agent：`code-generator.md`

```markdown
---
name: code-generator
description: 代码生成协调者。接收需求后生成代码，并自动委托前端和后端审查
tools: ["Read", "Write", "Edit", "Bash", "Grep", "Glob"]
model: sonnet
---

你是一个代码生成协调者。职责：根据需求生成代码，并确保代码经过前后端双重审查。

## 工作流程

### 1. 分析需求
- 明确功能范围，识别涉及的前端组件和后端接口

### 2. 生成代码
- 按项目规范编写，单文件不超 500 行
- 禁止硬编码，异常必须记录

### 3. 自动委托审查（关键步骤）

代码生成完成后，必须立即派生两个子 Agent：

- 启动 `frontend-reviewer` 审查前端文件
- 启动 `backend-reviewer` 审查后端文件

### 4. 汇总结果
- 收集审查报告，CRITICAL 必须修复，HIGH 询问用户

## 约束

- 每次生成代码后必须启动审查子 Agent，不能跳过
- CRITICAL 问题必须修复后再交付
```

### 子 Agent：`frontend-reviewer.md`

```markdown
---
name: frontend-reviewer
description: 前端代码审查专家，专注组件质量、状态管理、XSS 防护
tools: ["Read", "Grep", "Glob", "Bash"]
model: sonnet
---

## 审查维度

### 组件设计
- 单一职责、Props 传递层级（>3 层建议用 Context）

### 状态管理
- useState/useReducer 合理性、不可变模式

### 安全
- JSX 转义、dangerouslySetInnerHTML sanitization

### 性能
- 不必要重渲染、稳定 key、大数据量处理
```

### 子 Agent：`backend-reviewer.md`

```markdown
---
name: backend-reviewer
description: 后端代码审查专家，专注 API 安全、数据库查询、错误处理、认证授权
tools: ["Read", "Grep", "Glob", "Bash"]
model: sonnet
---

## 审查维度

### 安全
- SQL 注入（参数化查询）、认证授权、敏感信息泄露

### API 设计
- 响应格式一致性、分页、错误状态码

### 数据库
- N+1 查询、索引合理性、事务

### 错误处理
- try/catch 覆盖、日志不含敏感数据
```

### 关键要点

| 问题 | 答案 |
|------|------|
| 主 Agent 怎么调用子 Agent？ | 在 prompt 里指示 Claude 使用 Agent 工具启动子 Agent |
| 子 Agent 是嵌套声明还是独立文件？ | **独立文件**。每个 Agent 是独立的 `.md` 文件 |
| 能不能在主 Agent 里定义子 Agent？ | 不能。Agent 文件只能定义自己 |
| 实际运行时谁决定调用子 Agent？ | 主 Agent 的 prompt 内容。prompt 里写了"必须启动审查"，Claude 就会执行 |

**本质**：主 Agent 的 prompt 就是工作流程脚本，告诉 Claude 做完代码后要干什么。子 Agent 的派生靠 **prompt 指令**，不是文件嵌套关系。
